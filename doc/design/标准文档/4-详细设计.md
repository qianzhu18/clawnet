# 4-详细设计

## 1. 当前状态

当前还没有进入正式的前后端详细设计实现阶段。

但从这一步开始，必须先冻结一件事：

- 后续所有前端页面，都要有完整的界面设计文档

也就是说，先把“设计文档怎么写”定清楚，再进入真正的页面详细设计。

### 1.1 当前阶段补充判断

当前已经可以进入“第一轮设计交付准备态”。

这不等于：

- 所有页面都已经能直接开发
- 所有命名都已经完全冻结
- 所有角色权限和异常态都已经讲完

它只等于：

- 当前原型已经足够支撑设计团队先出第一版稿
- 设计和产品可以并行推进
- 新问题不再散落聊天里，而是必须回写标准文档和 `QA`

## 2. 这份文件当前回答什么

这份文件当前不写具体页面内容，只回答：

- 详细设计阶段要交什么文档
- 界面设计稿应该按什么维度拆
- 什么叫“可交设计 / 可交前端”的页面文档
- 什么叫“可单独发给外部设计师”的交付包
- 当前还没有进入哪些更深的技术设计

## 3. 后续界面设计文档的固定拆法

后续每一批页面设计文档，都必须同时按 3 个维度拆：

### 3.1 按链路拆

- 打开前
- 打开后
- 登录前
- 登录后
- 未加入基站
- 已加入基站
- 未接入 skill
- 已接入 skill

### 3.2 按用户角色拆

- 访客
- 普通用户
- 已接入用户
- 基站成员
- 站长 / 管理者

### 3.3 按页面拆

每一页都必须单独成稿，至少回答：

- 页面目标
- 首屏信息
- 模块顺序
- 按钮与状态
- 字体与色彩
- 布局与尺寸
- 空态 / 异常态 / 成功态

## 4. 后续详细设计文档的最小交付物

进入这一阶段后，至少要有 3 份文件：

- `界面设计文档规则.md`
- `页面设计稿模板.md`
- `界面设计审稿清单.md`

之后每个真实页面再单独落设计稿。

当前阶段如果要发给外部设计师，不再额外新建一套平行交付目录。

当前统一直接交这 `8` 份现有文档：

1. `1-需求分析.md`
2. `2-概要设计.md`
3. `3-产品原型设计.md`
4. `../原型细节/逐页详细设计稿.md`
5. `../原型细节/二级页与展开态设计稿.md`
6. `../原型细节/三级展开态设计稿.md`
7. `../原型细节/四级字段与反馈态设计稿.md`
8. `../概要细节/ClawNet项目问答集.md`

规则是：

- 设计师只靠这 `8` 份就能开始画稿
- 其余 `history/`、零散旧稿和聊天记录都不作为前置依赖

补充说明：

- `逐页详细设计稿.md` 解决一级页与核心页面骨架
- `二级页与展开态设计稿.md` 解决点击之后的承接页
- `三级展开态设计稿.md` 解决深层编辑态、确认态、异常态
- `四级字段与反馈态设计稿.md` 解决字段文案、空态、成功失败反馈和微状态

## 5. 页面设计稿必须包含的内容

以后任何页面设计稿，都不能只写“这页有几个卡片”。

必须至少写清：

- 页面服务对象
- 页面主任务
- 进入前提
- 首屏主信息
- 模块树
- 按钮树
- 状态树
- 色板
- 字体层级
- 栅格与间距
- 组件规格
- 页面切换关系

补充硬规则：

- 如果这份文档会被单独发给外部设计师，它就必须在自身内部把上下文讲清楚
- 不允许出现“请再去看仓库里的另一份文档才能理解”的前提
- 不允许把关键页面目标、用户角色、状态差异藏在其他文件里

## 6. 当前不在这一步展开什么

- 真实 API 文档
- 数据库字段设计
- 服务拆分
- 安全与性能设计

这些都要等界面设计规则和页面设计稿稳定后，再进入下一层。

## 7. 当前进入条件

只有下面条件同时满足，才进入“按页写详细设计稿”：

1. `1-需求分析.md` 已可公开阅读
2. `2-概要设计.md` 已讲清系统边界
3. `3-产品原型设计.md` 已讲清页面包和审稿顺序
4. `localhost:3000/prototype` 已能支撑人工审稿
5. 当前首期页面范围已冻结

### 7.1 当前进入条件的现实解释

当前已经满足“先把第一轮设计稿跑起来”的条件。

但还没有满足“把所有页面直接交开发”的条件。

所以当前正确顺序是：

1. 继续补齐原型和逐页细稿
2. 交设计团队出第一轮页面稿
3. 按设计稿和原型一起做迭代修正
4. 再进入真正的按页详细设计与开发冻结
