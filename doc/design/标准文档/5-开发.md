# 5-开发

## 1. 当前状态

当前还没有进入正式开发阶段。

现在的网页改动仍然服务于：

- 概要设计验证
- 产品原型验证

但从这一轮开始，原型阶段也必须按最小工程纪律推进：

- 每次继续改动前先读 `AGENTS.md`
- 每次继续改动前先看 `git status`
- 每次重要改动前先做一次 checkpoint commit，保证当前状态可回退、可迁移

## 2. 开发阶段要回答什么

进入这一阶段后，才正式回答：

- 哪些页面进入实现
- 哪些 mock 替换为真实数据
- 代码模块如何划分
- 任务如何分批推进
- 每一轮开发如何验收

## 3. 当前不应该做什么

- 不在原型未冻结时大规模写代码
- 不跳过详细设计直接铺后端
- 不把“看起来能跑”当成“产品已成立”
- 不长期堆积未提交修改，再继续往上叠新功能
- 不在同一个上下文已经明显失控时继续硬改

## 4. 进入条件

1. 标准文档 `1-4` 已稳定
2. 关键页面原型已冻结

4. 当前任务清单和验收口径已明确

## 5. 当前阶段也要执行的工程规则

### 5.1 改动前固定动作

每次准备继续改代码、原型或文档时，先做这 `3` 步：

1. 读 `AGENTS.md`
2. 读当前主线文档和对应专题文档
3. 检查 `git status`

### 5.2 checkpoint 规则

- 只要当前工作区里已经有值得保留的进度，就先做 checkpoint commit，再开始下一轮改动
- checkpoint commit 的目标不是“发布完成”，而是“把当前状态安全落盘”
- 提交信息优先使用：
  - `chore(checkpoint): backup current ui worktree`
  - `chore(checkpoint): backup current docs state`

### 5.3 新开对话迁移规则

如果已经出现以下任一情况，优先新开对话窗口处理，而不是继续在旧对话里堆：

- 代码耦合明显升高，改一处动多处
- 上下文过长，当前对话已经难以保持边界
- 同一轮里同时夹杂产品调整、UI 重构、状态修补和 bug 排查
- 继续修改前已经无法快速说清“下一步只改哪一件事”

新开对话前，至少要带上这份最小迁移包：

- 当前分支名
- 最新 checkpoint commit hash
- 当前预览命令
- 本机预览地址
- 局域网预览地址
- 当前卡点
- 下一轮只准备解决的一件事
