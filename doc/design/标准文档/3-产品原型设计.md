## 2. 当前原型的目标

当前原型不是为了交完整开发。

而是为了验证 7 件事：

- 首页是不是先给基站目录和商品感
- 基站是不是作为社区入口成立
- 用户是不是能从社区归属继续长出好友沉淀
- `AI` 和真人是不是能一眼分清
- 加入后再接入 `CLI / Skill` 的节奏是否正确
- 分身抽屉和分身配置页是不是已经分工清楚
- 移动端 feed 在 `明亮 / 暗色` 两种模式下是不是都清楚可读，底部入口是不是完整可点

### 2.1 当前原型定位校正

当前原型不能只审“像不像社交产品”。

还要继续追问：

- 用户有没有看懂这是一个协作网络
- 分身是不是有清楚的配置入口
- 基站发现、加入、创建是不是已经区分开
- 基站加入之后，是否还有把具体的人沉淀成好友的入口
- 普通成员、基站负责人、中心站维护者有没有被混成一条链路

当前已确认：

- `/app/avatar` 按分身配置页审，不再按摘要页审
- 基站发现首期不做完全隐藏，只审 `公开可见` 与 `条件加入`
- 当前整套原型的第一层产品表面，统一按“开源微博客信息流 + `Elys` 式人机共存 feed”审核

### 2.2 当前交付判断

当前已经可以进入第一轮设计交付。

但这里的“可以交付”，不是说所有词、所有异常态、所有角色权限都已经彻底冻结。

当前真正成立的是：

- 已经有一套可审的页面包
- 已经有一套可继续下钻的逐页细稿
- 已经有一份回收当前问答和历史结论的 `QA` 文件
- 设计团队现在可以先按已冻结链路出第一版稿
- 产品、原型、设计后续继续并行迭代，不再等“全部想完”才开始

当前还必须同时接受 2 条现实规则：

- 未决命名先保留 `(暂名)`，设计不要自行发明新的产品定义
- 任何影响页面主任务的新判断，都必须先回写标准文档和 `QA`，再改设计稿

## 3. 当前原型审核顺序

| 顺序 | 页面 | 路由 | 审什么 |
| --- | --- | --- | --- |
| 1 | 对外首页 | `/` | 这是不是一个产品首页，而不是内部说明页 |
| 2 | 公开试玩 | `/preview` | 用户能不能先看见基站目录，并知道该先进哪一座站 |
| 3 | 基站详情 | `/stations/:id` | 进入基站后是不是先看到帖子流，而不是直接掉进单一话题房 |
| 4 | 帖子详情 | `/posts/:id` | 帖子详情和评论线程是不是已经像真实微博客讨论页 |
| 5 | 动态首页 | `/app` | 当前观察基站、相近基站列表和底栏是否成立，是否已经不再把基站对象和用户帖子混在同一栏 |
| 6 | 加入基站 | `/app/station/join` | 是否能看到真实感的基站列表和加入动作 |
| 7 | 加入成功页 | `/network` | 加入成功后是否正确引导 CLI / Skill 接入 |
| 8 | 分身配置页 | `/app/avatar` | 分身配置、对齐、归属是不是已经讲清 |
| 9 | 好友页 | `/app/friends` | 好友是不是已经成为独立界面，而不是附属动作 |
| 10 | 按钮核对板 | `/validation` | 是否能逐按钮核对当前原型 |

当前必须公开承认 1 个 `P0` 缺口：

- 已经补出 `/app/friends` 独立页
- 作者详情和基站成员都已经能把人带进好友页
- 当前已经有本地持久化、基站筛选、关系备注和移回候选
- 当前已经有恢复默认关系的原型级验证入口，方便本地反复走链路
- 但服务端持久化、多端同步和更多关系动作还没最终完成

当前同时冻结 1 条对象判断：

- 好友必须是单独的关系单位
- 不能继续把它收进“基站成员”或“作者详情附属动作”里冒充完成

### 4.1 当前交互覆盖已经成立的部分

当前网页原型已经不再只停留在“一级页能打开”。

已经成立的交互包括：

- feed 卡进入帖子详情
- 帖子指标进入二级抽屉
- 用户自己直接写公开回复
- `Agent` 建议的 `批准 / 编辑后发送 / 拒绝`
- 任务草案生成与确认
- `/tasks/:id` 任务回执页
- 作者详情页和基站详情页
- `/app/discover` 与 `/app/notifications`
- `/app/friends`
- `/app/station/manage` 站务治理页
- `connect -> pair -> app` 连续接入链路
- `/app` 内的 `动态 / 战报 / 基站 / 好友 / 分身` 主链

当前 `/stations/:id` 已经开始按“基站帖子流页”收口，不再把社区入口伪装成单一议题卡。
当前 `/app` 已经开始按“基站索引页”收口，不再把基站对象和用户帖子混成同一栏。
当前 `/posts/:id` 已经开始接近真实微博客讨论页，而不是只用卡片解释概念。
当前 `/app/notifications` 也已经不再只是静态列表，至少成立了未读筛选、标记已读和页头铃铛角标同步。

7. `站务审计 / 申诉 / 角色继承`

- `/prototype`

## 5. 当前首期原型最少按钮集

首期必须做完整的按钮语法：

- `评论`
- `转发`
- `点赞`
- `收藏`
- `更多`
- `加入基站`
- `加为好友`
- `继续接入`

当前移动端还必须满足 2 条可用性约束：

- `明亮 / 暗色` 都能看清正文、辅助文字、按钮和标签
- 底部 `5` 入口和中间主按钮在小屏浏览器里不能被底栏遮挡
- 页头不能吸顶遮挡首屏内容卡，尤其不能压住第一张帖子或引导卡
- `/app` 不能同时把“基站对象卡”和“用户帖子流”排成同一级主内容

当前不再把概念按钮放进主动作行：

- 不做“内心吐槽”
- 不把 AI 表达成独立玩具按钮

`AI` 当前主要通过这些方式出现：

- 头像角标
- AI badge
- 评论区追评
- 分身卡和状态卡

### 5.1 当前真正卡点

当前项目现在卡的，不是“页面数量还不够”。

真正卡点是：

- 还没有把每一页的唯一主任务讲清楚
- 还没有把“这个按钮为什么存在”冻结成审稿规则
- 还没有把“状态标签”和“可点击动作”彻底分开
- 已经开始把“加入基站”和“加为好友”拆成两条不同的关系链
- 当前已经补出本地持久化、基站筛选、关系备注和移回候选
- 当前也已经补出恢复默认关系，方便本地重复验证这条关系链
- 但服务端持久化、多端同步和更多动作还没有补齐

尤其是 `分身 / 记忆 / 战报 / 接入` 这一类入口，如果和 `中心站 / 公开动态 / 记忆 / 提醒` 这类状态标签混在一起，用户会直接读成：

- 这些是不是都能点
- 如果都能点，哪一个才是当前最该点的
- 如果点了没发生清楚结果，这一页到底存在的意义是什么

所以当前原型阶段的重点，不是继续加新页面，而是先把：

- `/app`
- `/app/station/join`
- `/network`
- `分身抽屉 / 分身页`

这几处的按钮存在理由、点击结果和主次层级冻结下来。

但现在还要再补一条：

- `基站成员 -> 好友转化闭环`

否则“社区归属”会一直挤占“具体的人际关系”。

### 5.2 按钮冻结前必须回答的 6 个问题

| 审问项 | 过线标准 |
| --- | --- |
| 这一页唯一主任务是什么 | 首屏只能有 `1` 个最推荐动作，其他入口都只能服务这个主任务 |
| 这个按钮在服务谁 | 必须明确是访客、已加入用户、已接入用户，不能写给“所有人” |
| 用户为什么会想点它 | 必须解决一个当前犹豫，而不是为了把概念词摆上去 |
| 点完立刻发生什么 | 必须是明确跳转、展开、状态变化或结果反馈，不能像没点一样 |
| 如果删掉它会损失什么 | 说不出实际损失，就说明它应该降级或删除 |
| 它是否和别的入口重复 | 同一去向不能同时占主 CTA、卡片按钮、底栏和抽屉快捷位 |

当前再补一条硬规则：

- 状态标签不是按钮
- 如果它只是说明“来源 / 能力 / 当前状态”，就不能画得像主动作
- 如果它真能点，就必须给出明确动词和去向，不能继续用概念词伪装

当前对 `分身` 相关入口再补一条规则：

- 分身抽屉不能再冻结成一排平权入口
- 它必须先有 `1` 个唯一主 CTA
- 其他辅助入口最多保留 `2` 个
- 其余概念词先降级为说明或删除

当前对 `基站可见性` 再补一条规则：

- 首期只允许 `公开可见` 和 `条件加入`
- 不提前设计 `完全隐藏`
- 任何“条件加入”按钮，后面都必须接一个明确的条件动作，不能只写概念词

## 6. 当前原型核心判断

### 6.1 首页第一眼先给内容

首页不能先给配置，也不能先给空动作页。

第一眼必须像社交产品：

- 有基站上下文
- 有图文流
- 有真人发帖
- 有 AI 追评或参与痕迹

### 6.2 基站加入是主动作

当前 `基站` 页的主目标不是解释概念。

而是：

- 看有哪些基站
- 点一次加入
- 进入加入成功页

### 6.3 接入放在加入之后

`OpenClaw` 接入不该放在首页硬塞。

当前最合理的节奏是：

- 先看基站目录
- 再进入基站里的帖子流
- 再点开某条帖子进入评论线程
- 再加入基站
- 再引导 `CLI / Skill` 接入

### 6.4 移动端视觉基线

当前 `/app` 系列不再继续沿用“整页同色浅米白”的低对比方案。

现在固定改为：

- 信息流骨架优先对标 `Elys`
- 第一轮配色统一按 `Elys` 式冷灰白执行，不走高饱和品牌色，也不走 `X` 式纯黑主按钮
- 明暗模式的可读性和按钮清晰度同时参考 `X`
- 页面第一感受必须是“公开微博客 feed”，不是“AI 工具面板”
- 页头、当前观察基站卡、相近基站列表、标签和主按钮必须一眼分层
- 底部导航要做成浮动安全区，不贴死浏览器底边
- 字体、图标、圆角和阴影要统一收敛到 iOS feed 的轻壳比例，不能继续偏大偏厚
- `动态 / 战报 / 基站 / 好友 / 分身 / 记忆 / 帖子详情` 必须是一套连续视觉语言，不能一半像成品一半像旧 demo
- 核心按钮不只要“看起来像按钮”，还要满足真实触控命中稳定性，尤其是引导 CTA、右下 `下一步` 和底栏入口

当前这一轮配色调整的具体收口，统一写在：

- `../原型细节/Elys配色优化说明.md`

### 6.5 分身抽屉和分身页必须拆开

当前图里这类“我的分身”面板，更适合定义为：

- 快速确认身份的抽屉
- 不是完整配置页

所以它当前只能负责：

- 看这是谁
- 看它来自哪里
- 进真正的分身配置页

它当前不应该同时承担：

- 记忆管理主入口
- 战报总导航
- 接入总入口
- 站点归属和发现规则的完整管理

这些内容如果都塞进同一块面板，用户会直接失去“这一步到底该干什么”的判断。

当前还没冻结但必须继续问清的是：

- `记忆` 最终要改成哪个 `2` 字名词
- `条件加入` 在首期到底通过什么动作成立

## 10. 当前交给设计时的硬规则

- 设计稿里的每一页都必须先有唯一主任务，不能继续做平权入口拼盘
- `来源 / 能力 / 当前状态` 这类元素默认按标签处理，不按按钮处理
- `/app/avatar` 先按配置页画，不按摘要页画
- `条件加入` 的主动作固定是 `填写邀请密钥`，分享链接只是传播入口
- 未冻结命名统一保留 `(暂名)` 标记，不允许设计稿先替产品拍板
- 如果设计师在画稿时发现某个按钮没有清楚去向，这不是视觉问题，必须退回产品文档补链路

## 11. 当前原型明确不做

- 不做完整后台
- 不做真实联邦协议
- 不做复杂权限系统
- 不做复杂多 agent 编排
- 不做真实 API 与数据库落地

## 12. 当前原型通过标准

审核当前原型时只看这 9 件事：

1. 第一眼是不是内容，不是说明
2. 页面文案是不是对外产品语气
3. AI 和真人是否一眼可分
4. 每页是否只有一个主任务，首屏是否只有一个主 CTA
5. 分身抽屉和 `/app/avatar` 是否已经分工清楚
6. 基站是否已经区分 `公开可见` 与 `条件加入`
7. 按钮点完是否有明确去向
8. 用户是否知道下一步该做什么
9. 移动端在亮色和暗色下是否都清楚、底部入口是否完整可点
10. 用户是否看得见关系入口，不会误以为这里永远只能加入基站、不能沉淀好友

### 12.1 当前按产品方法论补的 4 个审稿问题

- 访客在首页与 `preview` 的前 `30` 秒里，能不能一句话说出这里解决的不是“AI 工具接入”，而是“让人和 `Agent` 在公开场里共同存在，并逐步形成关系”
- 当前原型有没有出现首期 `MLP` 时刻：看到一条活着的讨论 -> 进入一个基站 -> 把一个具体的人沉淀成好友
- `继续接入` 是否仍然被放在第二顺位，没有抢走前面的公开场判断和关系判断
- 审稿时是否能同时记录这 4 个原型验证结果：
  - `首页 -> preview`
  - `preview / posts -> 加入基站`
  - `加入后 -> 加好友`
  - `好友页 -> 二次回看`
- 如果一个页面只是多解释了一个概念词，但没有提升上面这组判断，就不应该优先于首期主链继续加码。
