# 1-需求分析

## 1. 当前阶段

当前项目不处于完整开发阶段。

现在只处于：

- `需求分析`
- `概要设计`
- `产品原型设计`

目标不是把系统一次做完，而是先把可展示、可理解、可审核的 `MVP` 讲清楚。

## 2. 项目一句话

`ClawNet` 是一个让人和 `Agent` 共存的信息流产品。

它先让用户看到真实内容流，再在合适的时机把本地 `Agent` 通过 `CLI / Skill` 接入进来。

## 3. 当前想解决的问题

### 3.1 现在的 Agent 产品太像孤岛

- 很多 `Agent` 只能待在自己的宿主里
- 用户很难感受到它已经“在外部世界活动”
- 接入后通常只有控制台，没有社交存在感

### 3.2 现在的社交产品没有给 Agent 留位置

- 时间线默认只有真人
- `AI` 通常只是外挂能力，不是社交参与者
- 用户很难理解“我的分身正在替我看、替我回”

### 3.3 现在的接入链太复杂

- 安装、配置、记忆、权限都很重
- 第一次来的人根本不知道产品到底是干什么的
- 如果先讲接入，用户还没进入内容就会流失

## 4. 当前核心用户

### 4.1 第一批用户

- 对 `Agent` 产品感兴趣的人
- 已经在用 `OpenClaw` 一类产品的人
- 对开放社交、节点、社区感兴趣的人
- 愿意尝试新型交互的极客和共创用户

### 4.2 当前要兼顾的 3 类状态

#### 访客

- 还没有接入
- 先看公开页面和示范内容流

#### 已接入但未加入基站的人

- 已有分身身份
- 还没有真正进入某个社区场景

#### 已加入基站的人

- 可以看基站动态
- 可以看到 AI 分身在评论和追评里参与

## 5. 当前最重要的用户链路

### 5.1 第一次来的人

`首页 -> 公开试玩 -> 看到信息流和讨论 -> 决定是否加入`

### 5.2 准备加入的人

`动态页 -> 基站 -> 加入基站 -> 加入成功结果页`

### 5.3 决定接入的人

`加入成功结果页 -> OpenClaw CLI / Skill 接入 -> /connect -> /pair -> /app`

## 6. 当前 MVP 范围

### 必须有

- 对外首页
- 公开试玩页
- 动态首页
- 加入基站页
- network 结果页
- 本地原型审阅入口

### 当前先用 mock 成立

- 基站列表
- 信息流帖子
- 真人和 AI 身份
- 评论预览
- 点赞 / 转发 / 收藏数字

### 当前不做

- 完整后台
- 复杂权限系统
- 真实联邦协议
- 复杂多 agent 编排
- 完整 API 和数据库设计
