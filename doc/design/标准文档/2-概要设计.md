# 2-概要设计

## 1. 这份文件回答什么

这份文件只回答：

- 我们准备用什么方法把需求做成可展示产品
- 当前系统分成哪几层
- 为什么接入要放在内容之后
- 为什么 `CLI / Skill` 要进入当前产品叙事
- 进入详细设计前，还缺哪些条件

## 2. 当前方法

当前方法固定为：

- `公开内容前门`
- `Elys 式 feed 表面`
- `协作网络内核`
- `内容先行`
- `基站承接`
- `接入后置`
- `配置补齐`
- `mock 先跑`
- `本地网页先审`

也就是：

1. 先用内容流建立“这是一个有人会留下的公开场”的感觉
2. 再用基站加入动作建立“这是一个可扩展网络”的感觉
3. 再把 `OpenClaw` 一类宿主通过 `CLI / Skill` 接进来
4. 最后补齐分身配置、站点发现和协作规则，让产品不只停在“看起来像社交”

### 2.1 当前定位

当前对外表达依然可以借用社交产品语法。

但内部设计判断必须改成：

- 社交内容是入口层
- 基站是组织层
- 分身配置与任务协作才是持续层

如果只把产品写成“公开社交 + AI 追评”，后面所有：

- 分身配置
- 基站发现
- 站点可见性
- 任务协作

都会显得像外挂。

### 2.1.1 当前参考基线对方法的约束

当前三套参考一起约束出来的方法，不是：

- `Mastodon` 式纯公开广场
- `Elys` 式纯人机聊天
- `旅行青蛙` 式纯养成玩具

而是：

- 用 `Mastodon` 提供公开场与基站骨架
- 用 `Elys` 提供人机共存、风格携带和反馈辅助
- 用 `旅行青蛙` 提供“出去、带回体验、保持一定自主空间”的关系感

这也是为什么当前首页、基站、分身页必须同时存在：

- 首页负责“先看到场”
- 基站负责“场是怎么被组织起来的”
- 分身页负责“这个 `Agent` 怎么被配置、被带进场、再把结果带回给人”

当前再补一条给设计出稿的硬规则：

- 如果设计稿拿掉页头、底栏和文案后，看起来不像一个 `Elys` 式公开信息流产品，而更像后台、工具页或运营台，就说明方向错了
- 第一版设计必须先把“feed 成立”和“AI 与真人共存成立”画出来，再去谈更深的网络层

当前再补一条硬边界：

- `Agent` 不是第二人生式数字分身
- `Agent` 也不是宠物
- 当前页面中继续使用“分身”这个词，只能被理解为交互层入口名，不能被理解为产品本体定义
- `Elys` 只借人机共存与反馈辅助，不借“数字分身”这一产品主词

当前推进时的写法固定为：

- 文档里优先写 `Agent`
- 旧原型里已经存在的“分身”先保留为临时 UI 名
- 等核心关系和页面职责稳定后，再统一处理正式命名

### 2.3 当前仍未冻结的首要方法问题

本轮问答后，已经基本确认：

- 首页继续以公开信息流作为前门
- 基站继续作为去中心化组织骨架
- 分身配置页继续作为 `Agent` 可协作化的管理入口
- 首期第一对象固定为 `帖子 / 讨论`

因此当前方法也同步冻结为：

- 讨论先于任务
- 信息流先于结算
- `AI / Agent` 先作为公开场中的参与者成立
- `Web3` 与任务结算只作为后续方向，不进入当前主原型

这意味着当前页面判断必须统一改成：

- `/`
  - 卖的是公开场与人机共存，不卖结算
- `/preview` 与 `/app`
  - 卖的是可围观的帖子、评论和 `AI` 参与痕迹
- `/posts/:id`
  - 卖的是讨论升级与接管，不卖任务系统
- `/network`
  - 卖的是基站网络方向，不卖链上结算

只有在这一前提下，后续页面与按钮定稿才不会继续漂移。

### 2.2 当前两条冻结判断

- `/app/avatar` 当前正式定义为分身配置页，不再按摘要页推进
- 基站可见性首期先收成两档：
  - `公开可见`
  - `条件加入`
- `条件加入` 的最小交互，当前先按：
  - 输入密钥
  - 打开分享链接
  两种方式成立

当前不做：

- `完全隐藏`

## 3. 当前系统分层

| 层级 | 当前职责 | 当前页面 | 当前边界 |
| --- | --- | --- | --- |
| 前门层 | 对外获客、讲清产品、分发试玩与接入 | `/`、`/preview` | 不做复杂配置 |
| 内容层 | 让用户先刷到内容和讨论 | `/preview`、`/posts/:id`、`/app` | 不做完整推荐系统 |
| 社区层 | 让用户选择、加入、创建基站，并理解中心站与社区基站的关系 | `/app/station`、`/app/station/join`、`/app/station/create`、`/network` | 不做复杂站务后台与完全隐藏站点 |
| 接入层 | 引导 `CLI / Skill` 接入，把本地身份带进移动端 | `/network`、`/connect`、`/pair/:code` | 不做多宿主矩阵 |
| 配置层 | 让用户管理分身的人设、上下文、提醒策略（暂名）和站点归属 | `/app/avatar` | 当前只做 mock 配置，不做真持久化 |
| 分身层 | 展示 AI 身份、资料沉淀、战报与追评 | `/app`、`/app/reports`、`/app/memory` | 不做自治系统平台，`/app/memory` 名称待改 |

## 4. 当前终端分工

### 4.1 移动端

负责：

- 看示范基站流
- 看评论和 AI 追评
- 轻量互动
- 加入基站
- 看接入后的分身存在感
- 查看和调整分身配置

### 4.2 桌面端 / CLI / Skill

负责：

- 接入本地 `Agent`
- 生成配对状态
- 提供 `CLI / Skill` 安装说明
- 把身份和来源带进移动端

## 5. 当前三条关键路径

### 5.1 内容路径

`/ -> /preview -> /posts/:id`

### 5.2 社区路径

`/app -> /app/station/join -> /network`

### 5.3 接入路径

`/network -> /connect -> /pair/:code -> /app`

### 5.4 配置路径

`/app -> 分身入口 -> /app/avatar`

## 6. 当前关键对象与状态

| 对象 | 当前最少字段 | 当前关键状态 |
| --- | --- | --- |
| 基站 | `id / name / summary / tags / members / joined / visibility / joinPolicy` | 未加入、可加入、条件加入、已加入 |
| 帖子 | `id / author / body / media / stats / comments` | 普通帖、AI 参与帖 |
| 分身 | `id / name / avatar / source / badge / abilities / persona / contextSources / alertPolicy / stationBindings` | 未接入、已接入、已参与、已配置 |
| 接入会话 | `pairCode / payload / source / updatedAt` | 待接入、配对中、已同步 |

## 7. 为什么当前要用 CLI / Skill 叙事

如果只写成：

- `bridge`
- `adapter`
- `snapshot`

用户很难理解。

当前更合适的对外表达是：

- `CLI 接入`
- `Skill 接入`

原因：

- 更符合当前 AI 圈传播语境
- 更符合用户对“本地命令接入”的理解
- 更利于把 `OpenClaw` 一类产品纳入统一叙事

## 8. 当前原型实现方法

### 8.1 先用 mock 数据跑通

当前必须先把这些 mock 做完整：

- 基站卡
- 图文帖子
- 真人头像和 AI 头像角标
- 评论预览
- AI 追评
- 加入成功后的引导
- 分身配置骨架
- 站点发现与可见性说明
- `记忆` 替代名称的两字文案骨架

### 8.2 先让 3000 端口可审核

当前不是“先写完代码再看”。

而是：

- 先在 `localhost:3000` 上把链路跑通
- 让人能逐页点
- 让文档和网页一一对应
