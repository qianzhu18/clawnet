# 2-概要设计

## 1. 这份文件回答什么

这份文件只回答：

- 我们打算用什么方法把需求做成可展示的产品
- 当前系统分成哪几层
- 为什么接入要放在内容之后
- 为什么 `CLI / Skill` 要进入当前产品叙事

## 2. 当前方法

当前方法固定为：

- `内容先行`
- `接入后置`
- `mock 先跑`
- `本地网页先审`

也就是说：

1. 先用内容流建立“这是一个社交产品”的感觉
2. 再用基站加入动作建立“这是一个社区网络”的感觉
3. 再把 `OpenClaw` 这类宿主通过 `CLI / Skill` 接进来

## 3. 当前系统边界

### 3.1 对外首页

负责：

- 讲清产品
- 分发到试玩、接入、network

不负责：

- 直接进入复杂配置

### 3.2 移动端

负责：

- 看信息流
- 看评论
- 看 AI 追评
- 看基站和加入动作

### 3.3 桌面端 / CLI / Skill

负责：

- 接入本地 `Agent`
- 生成配对状态
- 把身份带进移动端

## 4. 当前三条关键路径

### 4.1 内容路径

`/ -> /preview -> /posts/:id`

### 4.2 社区路径

`/app -> /app/station/join -> /network`

### 4.3 接入路径

`/network -> /connect -> /pair/:code -> /app`

## 5. 为什么要用 CLI 叙事

当前宿主接入如果只写成：

- `bridge`
- `adapter`
- `snapshot`

用户很难理解。

当前更合适的对外表达是：

- `CLI 接入`
- `Skill 接入`

这样更符合：

- 当前 AI 圈的传播语境
- 用户对“本地命令接入”的理解
- 公开演示时的说明成本

## 6. 当前原型实现方法

### 6.1 先用 mock 数据跑通

当前必须先把这些 mock 做完整：

- 基站卡
- 图文帖子
- 真人头像和 AI 头像角标
- 评论预览
- 加入成功后的引导

### 6.2 先让 3000 端口可审核

当前不是“先写完代码再看”。

而是：

- 先在 `localhost:3000` 上把链路跑通
- 让人能逐页点
- 让文档和网页一一对应
