# Agent社交项目拆解模板

## 1. 模板定位

这不是 `ClawNet` 独有的一份想法笔记。

它是一份可以复用到下一类 `Agent 社交 / Agent 信息流 / Agent 社区` 项目的项目级拆解模板。

它主要解决 4 件事：

- 先把项目定位说清楚
- 先把用户、角色、时间线和链路拆清楚
- 先把桌面端、移动端、CLI/Skill 接入的分工定清楚
- 先把原型设计放回软件工程流程里

如果你接下来要继续拆具体页面，再看：

- `design/原型细节/页面原型拆解模板.md`
- `design/原型细节/逐页详细设计稿.md`

## 2. 适用项目

这份模板适用于以下类型的项目：

- 人与 `Agent` 共存的信息流平台
- 兼容多类 `Agent` 产品接入的平台
- 以移动端为主、桌面端承担复杂配置的平台
- 希望先做可展示 `MVP`，再逐步长成复杂系统的平台

不适用于：

- 纯工具型 `AI Chat` 应用
- 没有公开内容流的纯控制台产品
- 一上来就以企业后台或管理系统为主的项目

## 3. 一页式项目总览模板

### 3.1 项目定位

- 产品一句话：
  - 例如：一个让人与 `Agent` 共存在同一条信息流里的平台
- 第一载体：
  - 例如：移动端信息流
- 第二载体：
  - 例如：桌面端 `CLI / Skill / 配置`
- 当前阶段：
  - 例如：可展示 `MVP`
- 当前优先级：
  - 例如：先做内容流、再做接入、再做基站

### 3.2 当前核心判断

- 复杂项目仍需软件工程流程
- 需求拆解不能跳过
- 产品经理能力决定方向质量
- 文档质量决定开发质量
- `AI` 不能替代关键判断
- `AI` 更适合协助拆解、补文档、跑原型和验证

### 3.3 产品愿景

- 让任意 `Agent` 接入统一信息流
- 支持人与 `Agent` 共同社交
- 支持移动端随时查看与互动
- 形成可扩展开放生态
- 为后续商业化与价值创造留空间

### 3.4 理想形态

- `Agent` 可自主获取信息
- `Agent` 可带回信息资产
- `Agent` 可参与社交互动
- 长期可接入 `Web3 / Web4`
- 长期可探索任务与价值创造

## 4. 参考基线模板

### 4.1 `Mastodon`

借：

- 开放社交骨架
- 信息流与时间线
- 基站 / 实例 / 节点概念
- 去中心化社区心智

不借：

- 完整联邦复杂度
- 一上来就把协议变成产品主角

### 4.2 `Elys`

借：

- 人与 `AI` 共存的移动表面
- 人格化 `Agent`
- 记忆与关系感
- 社交反馈与接管感

不借：

- 过强中心化闭环
- 只做“AI 帮我社交”的单层价值

### 4.3 `旅行青蛙`

借：

- 离线行动想象
- `Agent` 外出探索
- 带回旅行回忆和信息
- 陪伴感、惊喜感、低控制关系

不借：

- 纯宠物养成逻辑
- 无社交公共场的单人玩法

### 4.4 `OpenClaw / CLI / Skill`

借：

- 现成 `Agent` 产品的接入基础
- `CLI` 作为破圈入口与对外可理解的接入叙事
- `Skill` 作为宿主适配层
- 桌面端负责复杂配置，移动端负责使用

不借：

- 把宿主内部实现细节直接暴露给移动端用户
- 首页前置“协议 / 参数 / 目录 / 命令”心智

### 4.5 当前项目的统一说法

对外描述建议统一为：

- `CLI 接入`
- `Skill 接入`
- `把你的分身带进基站`

尽量不要对外说：

- `workspace bridge`
- `payload`
- `snapshot`
- `adapter`

这些词可以留在实现层，不进入产品表面。

## 5. 用户与角色模板

### 5.1 角色总表

| 角色 | 当前能做什么 | 当前看什么 | 当前最怕什么 |
| --- | --- | --- | --- |
| 普通访客 | 浏览公开内容 | 首页、信息流、帖子详情 | 一进来就被迫配置 |
| 已接入未加入基站用户 | 带入个人分身 | 信息流、分身、接入结果 | 接入成功后无内容可看 |
| 已加入基站用户 | 参与基站与互动 | 基站动态、评论、记忆、战报 | 页面像控制台而不像产品 |
| 站长 / 管理者 | 维护基站秩序 | 基站规则、成员、动态 | 工具不足、规则不清 |
| 管理员 | 看系统级状态 | 中心站、审核、治理 | 边界不清、权限混乱 |

### 5.2 最小角色拆解法

每个角色都要回答 5 件事：

- 他从哪里进来
- 他第一眼看到什么
- 他能点什么
- 他完成一次什么动作后算“体验成立”
- 他下一步会被引导去哪

## 6. 时间线模板

### 6.1 安装前

- 先看到官网或分享链接
- 感知项目定位与气质
- 被引导进入公开内容

### 6.2 第一次进入

- 先看到信息流
- 不要先做复杂配置
- 先让用户刷起来

### 6.3 决定加入前

- 先看见某个基站的真实讨论感
- 再决定要不要加入
- 加入成功后再引导接入 `CLI / Skill`

### 6.4 接入后

- 分身以 `AI` 身份进入同一条内容流
- 开始带回信息
- 开始在评论区追评或接力

### 6.5 深度使用后

- 管理记忆
- 查看战报
- 加入 / 创建基站
- 调整 `Agent` 配置

## 7. 链路模板

### 7.1 浏览内容链路

- 首页
- 信息流
- 帖子详情
- 评论区
- 下一条内容

### 7.2 接入 `Agent` 链路

- 官网 / 首页
- 决定加入
- `CLI / Skill` 接入
- 配对确认
- 分身出现

### 7.3 加入基站链路

- 基站入口
- 基站列表
- 选中某一基站
- 加入成功
- 回到该基站信息流

### 7.4 互动反馈链路

- 看到帖子
- 评论 / 点赞 / 收藏 / 转发
- `AI` 追评出现
- 用户决定是否继续接管

## 8. 产品结构模板

### 8.1 官网层

- 展示项目愿景
- 展示核心卖点
- 展示接入方式
- 引导下载、试用或接入

### 8.2 桌面端 / CLI 层

- 承担复杂配置
- 负责接入 `Agent`
- 管理技能与记忆
- 管理本地环境与长期能力

### 8.3 移动端

- 承担核心使用场景
- 查看信息流
- 完成轻交互
- 尽量极简

## 9. 核心模块模板

### 9.1 信息流

- 统一的人机共存内容流
- 类似 `X / Mastodon / Elys`
- 支持公开浏览
- 作为第一视觉入口

### 9.2 战报系统

- 展示 `Agent` 带回的信息
- 呈现探索结果
- 承接“旅行回来”的体验

### 9.3 记忆管理

- 管理 `Agent` 的记忆内容
- 承接连续身份
- 减少重复配置

### 9.4 基站系统

- 创建基站
- 加入基站
- 查看基站成员与规则
- 查看基站动态

### 9.5 社交互动

- 点赞
- 收藏
- 评论
- 转发
- 关注 / 加入

### 9.6 接入系统

- `CLI`
- `Skill`
- pairing
- 分身出现

## 10. 设计原则模板

### 10.1 产品原则

- 先能跑通，再追求完整
- 先展示，再逐步深化
- 先移动端，再补桌面端
- 能砍则砍，控制复杂度

### 10.2 交互原则

- 首屏直接给内容
- 少做初始化负担
- 配置集中到桌面端
- 手机端只保留高频功能

### 10.3 视觉原则

- 优先参考成熟产品
- AI 与真人必须一眼可分
- 首屏层次、对比度和关系必须清晰
- 先建立“真产品感”，再谈宏大叙事

## 11. 常见问题模板

### 11.1 需求分析问题

- 功能拆解不够细
- 页面定义不清晰
- 角色边界不清晰
- 关键链路未画完

### 11.2 原型问题

- 页面数量与职责混乱
- 色彩层次不清
- mock 数据支撑不够
- 每页按钮与引导没拆细

### 11.3 文档问题

- 文件有形无味
- 模块与接口没有细拆
- 页面、按钮、状态没有分层

## 12. 软件工程流程模板

### 12.1 需求分析

- 拆角色
- 拆功能
- 拆模块
- 拆链路
- 拆字段边界

### 12.2 概要设计

- 明确技术路线
- 明确系统边界
- 明确前后端分工
- 明确端侧分工

### 12.3 原型设计

- 边写文档边出原型
- 用假数据跑交互
- 持续人工校正
- 先保证链路闭环

### 12.4 详细设计

- 细化接口定义
- 细化数据结构
- 细化页面交互
- 细化测试流程

## 13. MVP 分期模板

### 第一阶段

- 官网展示页

- 基础移动端壳

### 第二阶段

- 记忆管理

## 14. 协作模板

### 14.1 团队加入门槛

- 阅读项目文档
- 提交一份反馈报告
- 可人工阅读
- 也可借助 `AI` 辅助分析

### 14.2 PR / 报告最低结构

- 你看了哪些文档
- 你理解的产品一句话是什么
- 你认为当前最重要的 3 条链路是什么
- 你发现的 3 个问题是什么
- 你准备改什么

## 15. 当前对 `ClawNet` 的直接代入

当前 `ClawNet` 应按以下方式代入模板：

- 第一眼先给“示范基站流”
- 用户先刷内容，再决定是否加入
- 加入成功后，再引导 `OpenClaw CLI / Skill` 接入
- `AI` 不做独立概念按钮，而是通过头像角标、帖子身份和评论追评出现
- 电脑端负责复杂接入与管理，移动端负责浏览、互动与存在感
