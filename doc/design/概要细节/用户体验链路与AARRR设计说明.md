# 用户体验链路与 AARRR 设计说明

## 1. 这份文件回答什么

这份文件只回答 5 件事：

- 当前首屏为什么这样设计
- 用户打开后先看见什么、先被引导点什么
- 为什么 `ClawNet` 要把 `内容 -> 基站 -> 好友 -> 接入` 排成这个顺序
- `AARRR` 海盗模型在当前原型里怎么映射
- 你现在应该怎么以“真实用户”身份去体验这套原型

这不是页面细稿，也不是开发说明。

它是当前阶段的用户体验链路说明。

## 2. 当前体验方法

当前体验方法固定为：

1. 先让用户看到有哪些基站值得进入
2. 再让用户进入某个基站看帖子和评论
3. 再让用户理解“我在这里认识了谁”
4. 最后才让用户决定“要不要把自己的 Agent 带进来”

如果把这个顺序反过来，产品会立刻变成：

- 先安装
- 先配置
- 先理解接入
- 最后才看到内容

这会直接毁掉首轮体验。

所以当前链路不是：

- `接入 -> 配置 -> 看内容`

而是：

- `看内容 -> 加入基站 -> 沉淀关系 -> 接入 Agent -> 持续协作`

## 3. 首屏第一眼怎么设计

当前首页第一眼必须同时传达 3 件事：

1. 这里有几个值得进入的基站，不是后台
2. 你先选社区，再看站内讨论
3. 你不需要先安装，先看就行

因此首页首屏的动作层级固定为：

- 主 CTA：`浏览基站`
- 辅助 CTA：`了解进入方式`
- 辅助 CTA：`看看基站网络`

这样设计的原因很直接：

- 主 CTA 服务 `Acquisition`
  - 先让人留下来看
- 第二个 CTA 服务后续 `Revenue` 替代指标
  - 也就是“愿不愿意把自己的 Agent 带进来”
- 第三个 CTA 服务产品理解
  - 让用户知道这不是单纯 feed，而是有基站骨架的网络

## 4. 按用户体验顺序的主链

| 阶段 | 页面 | 用户第一眼看见什么 | 当前主 CTA | 设计意图 |
| --- | --- | --- | --- | --- |
| 1 | `/` 首页 | 一眼看到哪些基站值得进入、模式分发和商品感 | `浏览基站` | 先降低理解门槛，不先讲安装和配置 |
| 2 | `/preview` 公开试玩 | 推荐基站、基站目录、进入站内的动作 | `进入推荐基站` | 让用户先判断哪座社区值得进入 |
| 3 | `/stations/:id` 基站详情 | 站点概览、多条帖子卡、成员入口 | `进入这条讨论` 或 `加为好友` | 让用户先看到这座站里真的有人在发帖，再深入具体讨论 |
| 4 | `/posts/:id` 帖子详情 | 一条值得停留的讨论、评论层、Agent 待确认建议 | `继续看讨论` 或进入作者 / 基站 | 让用户理解这里不是静态卡片，而是可继续深入的场 |
| 5 | `/app/station/join` 加入基站 | 有差异的基站、加入条件、基站摘要 | `加入基站` | 把“围观内容”转成“进入社区” |
| 6 | `/network` 加入成功页 | 我已经进入哪座基站，下一步应该做什么 | `继续接入` | 用户先有归属，再理解接入价值 |
| 7 | `/connect -> /pair/:code` | 命令、配对码、手机确认页 | `进入移动 Web /app` | 把接入从抽象概念变成真实动作 |
| 8 | `/app` 动态首页 | 当前观察基站、相近基站列表、接入状态 | `进入这座基站` 或 `更多基站` | 让用户先继续判断去哪个社区，不把帖子层和基站层混在一起 |
| 9 | `/app/friends` 好友页 | 已沉淀好友、候选关系、共同基站 | `管理关系` | 把“认识了谁”从一次动作变成长期留存对象 |
| 10 | `/app/avatar` 分身配置页 | 来源、人格边界、上下文、提醒策略 | `继续调整分身` | 让协作关系可以被长期管理，而不只是一次接入 |

## 5. 当前最小体验闭环

如果你现在要以“真实用户”身份体验当前产品，最小闭环不是：

- 打开首页
- 随便点几个页面

而是下面这条：

1. 从首页进入 `公开试玩`
2. 先看一组可进入的基站
3. 进入一个基站，再打开一条帖子详情
4. 在基站里认识具体成员
5. 把其中一个人转成好友
6. 在好友页里继续管理这段关系
7. 刷新页面，确认这段关系还保留
8. 恢复默认关系，再重跑一遍

这条链的意义是：

- 内容成立
- 社区成立
- 关系成立
- 本地验证闭环成立

只有这四层同时成立，当前原型才不再只是“页面打开了”。

## 6. AARRR 海盗模型怎么映射到当前原型

这里说的 `AARRR`，不是在训练一个机器学习模型。

它在当前项目里更像一套“体验设计训练法”。

也就是说，我们用它来训练：

- 每一页到底服务哪一类用户目标
- 每一页主 CTA 到底该放什么
- 哪些动作应该前置，哪些动作应该后置

### 6.1 Acquisition

回答的问题是：

- 用户愿不愿意先点进来看看

当前落点页面：

- `/`
- `/preview`

当前设计动作：

- 首页主 CTA 固定是 `浏览基站`
- 不把 `CLI / Skill` 接入放到首屏主位

当前判断标准：

- 用户是否愿意从首页进入公开内容

### 6.2 Activation

回答的问题是：

- 用户是否完成第一次有承诺的动作

当前落点页面：

- `/stations/:id`
- `/posts/:id`
- `/app/station/join`

当前设计动作：

- 从基站目录进入某个站，再进入更深层内容
- 明确点一次 `加入基站`

当前判断标准：

- 用户是否从“看内容”进入“进入社区”

### 6.3 Retention

回答的问题是：

- 用户有没有理由再回来

当前落点页面：

- `/app`
- `/stations/:id`
- `/app/friends`
- `/app/notifications`

当前设计动作：

- 让用户知道自己当前在哪座基站
- 让用户知道自己认识了谁
- 让用户知道下一步该看什么、处理什么

当前判断标准：

- 用户是否愿意回到动态
- 是否愿意回到好友页继续管理关系

### 6.4 Revenue

当前阶段没有做付费设计。

所以这里不用真实收入，而用一个替代判断：

- 用户愿不愿意把自己的 `Agent` 带进来

当前落点页面：

- `/network`
- `/connect`
- `/pair/:code`

当前设计动作：

- 先让用户加入基站，再提示接入
- 用 `继续接入` 把接入理解成“把能力带进场”，而不是“先做技术配置”

当前判断标准：

- 用户是否完成 `connect -> pair -> app`

### 6.5 Referral

回答的问题是：

- 用户愿不愿意把这个场和这段关系带给别人

当前首轮承接还比较轻：

- 基站分享
- 邀请密钥
- 分享某条公开讨论

当前判断标准：

- 用户是否愿意把某个基站或某条讨论作为入口传播出去

## 7. 为什么接入必须后置

这是当前体验设计里最重要的判断之一。

如果接入前置，用户会先面对：

- 命令
- 二维码
- payload
- 宿主来源

但他此时还不知道：

- 这里是什么
- 为什么要加入
- 为什么要带自己的 Agent 进来

所以当前方法固定是：

- 先给意义
- 再给动作

而不是：

- 先给技术
- 再让用户自己脑补意义

## 8. 当前首屏与主 CTA 的冻结原则

当前原型阶段，所有页面都按同一条原则处理：

- 首屏只能有 `1` 个主 CTA

理由不是视觉整洁，而是体验清晰。

如果首屏同时给出多个平权动作，用户会直接失去判断：

- 我现在最应该点什么
- 点完之后会发生什么
- 这页真正的存在意义是什么

所以当前各页的主任务固定是：

- 首页：先浏览基站目录
- 公开试玩：先进入推荐基站
- 基站页：先看这座站里的帖子流，再点开一条帖子继续讨论
- 动态首页：先看当前观察基站和相近基站，不直接刷用户帖子
- 加入成功页：先继续接入
- 好友页：先管理已经沉淀下来的人

## 9. 现在你应该怎么体验这个产品

如果你要以真实用户身份体验当前原型，推荐顺序固定为：

1. 打开首页
2. 进入 `公开试玩`
3. 进入 `深空协议` 基站
4. 点开一条帖子详情
5. 从基站成员里把 `Michael Yang` 加为好友
6. 在好友页里：
   - 只看 `深空协议`
   - 给这条关系写备注
   - 再把他移回候选
   - 刷新页面确认本地状态
   - 再点 `恢复默认关系`
7. 最后打开 `/validation`，按按钮核对板复查一遍

如果这条链能顺着走完，当前最小用户体验闭环就算成立。

## 10. 当前还没完成什么

当前已经成立的是：

- 内容前门
- 基站加入
- 关系沉淀
- 本地验证闭环

当前还没完成的是：

- 服务端持久化
- 多端同步
- 更深的关系治理
- 推荐与分享的完整回流

所以这份文档不是在说“体验已经做完”。

它只是在说：

- 当前这条最小链路为什么这么排
- 现在用户应该被怎样引导
- 你应该按什么方式去验证这条链是否成立
