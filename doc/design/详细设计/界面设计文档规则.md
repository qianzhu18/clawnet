# 界面设计文档规则

## 1. 规则目标

这份规则只解决一个问题：

- 后续所有前端页面，应该按什么方法写成真正可交付的界面设计文档

它不是页面内容本身。

它是页面设计稿的统一规则。

## 2. 总原则

### 2.1 先文档，后出图

先把页面设计文档写完整，再交给设计师或设计 agent 出图。

### 2.2 先链路，后单页

先确定用户是怎么走进来的，再拆每一页。

### 2.3 先角色，后视觉

先明确这页服务谁，再谈字体、颜色和布局。

### 2.4 先状态，后样式

先明确默认态、空态、异常态、成功态，再定义色块和按钮。

## 3. 固定拆解顺序

后续任何一组页面，都必须按这个顺序拆：

1. 场景与链路
2. 用户角色
3. 页面清单
4. 页面结构
5. 按钮与状态
6. 视觉 token
7. 审稿与验证

## 4. 链路拆解规则

每一批页面都必须先回答：

### 4.1 打开前 / 打开后

- 用户打开产品前知道什么
- 用户第一次打开时看到什么
- 用户打开后第一步应该做什么

### 4.2 登录前 / 登录后

- 登录前能看什么
- 登录前不能做什么
- 登录后新增了什么能力

### 4.3 加入前 / 加入后

- 未加入基站时的页面样子
- 已加入基站后的页面变化

### 4.4 接入前 / 接入后

- 未接入 `CLI / Skill` 时的页面语气和 CTA
- 已接入后分身如何出现

## 5. 用户拆解规则

后续任何页面，都必须标明自己服务哪一类用户：

- 访客
- 普通用户
- 已接入用户
- 基站成员
- 站长 / 管理者

如果同一页服务多个角色，必须分角色写差异，不允许混写。

## 6. 页面拆解规则

每一页都必须单独成稿。

每份页面设计稿至少要有这些部分：

1. 页面定位
2. 用户与前提
3. 首屏信息
4. 模块树
5. 页面布局
6. 字体系统
7. 颜色系统
8. 按钮系统
9. 状态变化
10. 组件规格
11. 跳转关系
12. 审稿标准

## 7. 页面布局规则

布局文档必须写清：

- 页面是移动端、桌面端还是双端
- 采用几层结构
- 顶部区、内容区、底部区怎么分
- 每个主模块的宽度、高度、边距、间距
- 固定层和滚动层怎么区分

不允许只写“上面一个标题，下面几个卡片”。

## 8. 字体系统规则

每页必须写清：

- 字体家族
- 大标题字号 / 行高 / 字重
- 模块标题字号 / 行高 / 字重
- 正文字号 / 行高 / 字重
- 辅助文案字号 / 行高 / 字重
- 标签字号 / 行高 / 字重

如果是移动端，还必须说明：

- 最小可接受字号
- 长文换行规则
- 中英文混排规则

## 9. 颜色系统规则

每页必须写清：

- 页面背景色
- 卡片背景色
- 主文字颜色
- 次文字颜色
- 弱文字颜色
- 主按钮颜色
- 次按钮颜色
- 边框颜色
- 分割线颜色
- AI 身份色
- 真人身份色

如果有明暗模式，必须分别写两套 token。

## 10. 按钮与可点击元素规则

每个按钮都必须写清：

- 按钮文案
- 按钮类型
- 所在模块
- 交互状态
- 点击结果
- 跳转去向
- 成功反馈
- 失败反馈

按钮状态最少包括：

- 默认
- 悬停
- 按下
- 禁用
- 加载中

## 11. 组件规格规则

每页要列出自己使用的组件规格：

- 头像
- 角标
- 标签
- 卡片
- 输入框
- 评论框
- 底栏
- 抽屉
- 浮层
- CTA 按钮

每个组件至少写：

- 尺寸
- 圆角
- 边框
- 阴影
- 内边距
- 文案层级

## 12. 状态规则

每页必须至少覆盖：

- 默认态
- 首次用户态
- 已登录态
- 已加入态
- 已接入态
- 空态
- 异常态
- 成功态

如果某页不适用某一状态，也要明确写“不适用”。

## 13. 页面跳转规则

每页必须给出：

- 从哪里进入
- 点哪些按钮离开
- 离开后到哪一页
- 带哪些参数
- 页面回来后保留什么状态

## 14. 审稿规则

每页设计稿的审稿必须回答：

1. 页面服务的是谁
2. 页面唯一主任务是什么
3. 首屏第一眼先看到什么
4. AI 和真人怎么一眼区分
5. 用户下一步会点哪里
6. 页面有没有多余说明性废话
7. 这页能不能直接交给设计师出图

## 15. 当前特别规则

对 `ClawNet` 当前阶段，额外固定 4 条：

1. 首页先给内容，不先给配置
2. 加入基站先于接入 skill
3. AI 主要通过头像角标、badge 和评论参与出现
4. 手机端页面文档优先于桌面端页面文档
