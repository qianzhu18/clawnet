# 逐页详细设计稿

## 1. 文档定位

这份文件不是再讲一遍“产品方向”。

阅读顺序固定为：

1. `../标准文档/1-需求分析.md`
2. `../标准文档/2-概要设计.md`
3. `../标准文档/3-产品原型设计.md`
4. `原型审阅指南.md`
5. 最后再看这份

它只做 4 件事：

- 把当前真实原型的每一个页面拆到可继续出 UI 稿的粒度
- 明确每页首屏先出现什么、模块怎么排
- 明确每个按钮点完去哪里、带什么状态
- 明确当前用什么 mock 数据表达页面成立

使用规则：

- 这份文档只基于当前代码里真实存在的页面写
- 不把候选壳层写成现状
- 后续设计稿和前端细化，默认都从这份文档继续往下走

### 1.1 当前给设计团队的使用规则

当前这份文档已经不是内部草稿。

它现在要承担的角色是：

- 让设计团队知道当前到底有哪些真实页面
- 让设计团队知道先画哪一批
- 让设计团队知道哪些词已经冻结，哪些只能先保留暂名

所以当前补 4 条硬规则：

- 设计团队默认不需要回看 `history/` 和聊天记录
- 如果一个页面目标和按钮结果在这里没有写清，就视为产品链路未完成
- 设计稿可以先画视觉，但不能擅自改写页面主任务
- 未决命名必须显式标 `(暂名)`，不能被视觉稿偷偷定死

## 3. 全局设计规则

### 3.1 当前移动端固定壳

当前移动端固定为：

- 顶部页头
- 页面主内容
- 底部 5 入口
- 右下动作按钮

底部 5 入口固定为：

- `动态`
- `战报`
- `基站`
- `好友`
- `分身`

其中：

- `基站` 是底部中间主按钮
- `好友` 已经补进底部一级导航，作为独立关系入口存在
- `记忆` 仍然保留，但退到抽屉和动作层，不再占用底部一级关系位
- 底栏必须浮在安全区里，不能贴死浏览器底边
- 小屏下底栏与右下动作按钮不能互相遮挡
- 页头不能吸顶遮挡首屏内容卡，尤其不能压住第一张帖子或引导卡

### 3.2 当前连续状态规则

当前页面之间的连续感主要靠两类状态：

- `payload`
  - 带着已接入 agent 的身份、来源、能力、宿主快照
- `stationAction`
  - 带着这次是“joined”还是“created”，以及对应基站信息

设计时必须保证：

- 同一次接入的 agent 名称、来源、会话，不允许在 `/connect /pair /app` 三页漂移
- 同一次基站动作的基站名称、简介、标签，不允许在 `/app/station/* -> /network` 漂移

### 3.2.1 当前未决词处理规则

当前已经明确有几组词还没有完全冻结。

设计稿必须按下面规则处理：

| 词 | 当前规则 |
| --- | --- |
| `分身` | 允许继续作为旧原型 UI 名出现，但正文解释统一优先写成 `Agent` |
| `/app/memory` 页名 | 还没有最终两字名；设计稿里必须标注 `(暂名)` |
| `提醒 / 提醒策略` | 还没有最终正式名；设计稿里必须标注 `(暂名)` |
| `/app/avatar` 首屏主 CTA | 交互方向已确定，但最终动词未冻结，设计稿里不可自行发明产品定义 |

设计时的底线是：

- 可以保留暂名
- 不能假装这些名字已经最终定稿

### 3.3 当前按钮语法

社交主语法：

- `查看讨论`
- `评论`
- `转发`
- `点赞`
- `收藏`
- `更多`

agent 语法：

- `创建我的 agent`
- `接入已有 agent`
- `加入这座基站`
- `进入基站`
- `@agent`
- `编辑后发送`

接入语法：

- `复制命令`
- `打开手机确认页`
- `进入移动 Web /app`
- `回到接入页`
- `回到桌面`

### 3.3.1 当前按钮存在性审问

逐页写按钮表时，不能只写“这里有几个按钮”。

每一个可点击元素都必须先过这张表：

| 审问项 | 当前要求 |
| --- | --- |
| 当前用户是谁 | 必须能指出是访客、已加入、已接入中的哪一类 |
| 当前页唯一主任务是什么 | 这个按钮只能是主链动作、辅助动作或退路之一 |
| 用户点击前脑内问题是什么 | 要能回答“他为什么现在想点” |
| 点击后用户能感知到什么变化 | 必须有跳转、展开、回填、禁用或成功反馈 |
| 如果没有这个按钮，用户会卡在哪 | 说不出卡点，说明它不是必需入口 |
| 是否和现有入口重复 | 已经在底栏、抽屉和动作层里出现过的去向，不能再抢首屏主位 |

当前固定 4 条落地规则：

- 首屏只允许 `1` 个主 CTA
- 同一层不并排摆出一排概念词按钮
- 说明来源、能力、状态的元素，默认只做标签，不做跳转
- 如果某个标签未来真的可点，必须改成动词型文案并给出明确去向

### 3.4 当前页头

当前移动端页头不再承担功能折叠。

| 按钮 | 当前职责 | 打开结果 | 不承载什么 |
| --- | --- | --- | --- |
| 左侧品牌区 | 只显示 `ClawNet` 和当前页名 | 无点击 | 不承载导航折叠 |
| 明暗模式切换 | 只负责切换 `/app` 系列主题 | 在亮色 / 暗色之间切换，并保持高对比可读 | 不承载页面跳转 |
| 右上用户图标 | 分身入口 | 打开右侧分身抽屉，收拢分身卡片与高频动作 | 不承载全局页面导航 |

### 3.5 页头下方新手引导条（已移除）

| 模块 | 内容 |
| --- | --- |
| 当前状态 | 当前移动端已取消可见的新手引导条 |
| 移除原因 | 这块文字太像内部说明，会抢正文和卡片空间 |
| 替代承接 | 用当前观察基站卡、相近基站列表、右上身份抽屉和右下动作层承接第一步 |

当前硬规则：

- `/app` 系列正文上方不再长期保留一条解释性文案块
- 页头只负责识别、主题切换和分身入口
- 首次动作应通过当前观察基站卡或相近基站卡给出，而不是再加一条说明条

### 3.6 右侧身份抽屉

| 模块 | 内容 |
| --- | --- |
| 抽屉标题 | `My Proxy / 我的分身` |
| 分身卡片 | 名称、状态、来源、能力 |
| 标签区 | `来源 / 能力 / 当前状态`，只做展示，不承担跳转 |
| 主 CTA | `进入分身配置` 或 `接入我的 agent`，按是否已接入切换 |
| 辅助入口 | 最多 `2` 个，只能服务当前主链 |
| 规则卡 | 只承接身份确认与分发，不做复杂设置面板 |

作用：

- 让右上图标变成一个真实存在的分身入口
- 让“当前是谁”先被确认，再进入真实配置页
- 抽屉的唯一主任务是：先确认“这是谁”，再选一条明确下钻路径

当前补充硬规则：

- 抽屉内 `来源 / 能力` chip 只表示状态，不是按钮
- 真正可点击的只保留主 CTA 和少量辅助入口
- 如果后续要让 `中心站` 或某个能力真的可点，必须单独做成 `查看来源`、`查看能力` 这一类动作，不继续伪装成标签
- 设计稿里不要再把抽屉画成一个小号总导航页

### 3.6.1 分身抽屉的当前重构结论

这一层不是 `/app/avatar` 的缩略版。

这一层更适合只回答 3 个问题：

- 这是谁
- 它现在处于什么状态
- 你接下来最该去哪里

所以当前重构后优先冻结为：

- 未接入时：
  - 主 CTA：`接入我的 agent`
- 已接入时：
  - 主 CTA：`进入分身配置`

辅助入口当前只允许从下面三类里选，不同时全上：

- `查看当前基站`
- `回到动态`
- `查看最近战报`

当前不再保留：

- `记忆`
- `接入`

作为已接入态抽屉的平权快捷入口。

原因：

- 这会把抽屉重新做成导航拼盘
- 会冲掉“进入分身配置”的唯一主动作

### 3.7 右下动作按钮

| 模块 | 内容 |
| --- | --- |
| 组件位置 | 页面右下角，悬浮在内容层与底栏之间，并高于底栏安全区 |
| 打开结果 | 弹出当前页动作面板 |
| 当前动作 | 跟随所在页面变化，并补一组页面切换 |
| 关闭方式 | 遮罩、`关` 按钮、点任意动作 |

当前动作面板固定为：

- `/app`：`去基站 / 好友列表 / 查看战报 / 我的分身`
- `/app/reports`：`回到动态 / 好友列表 / 我的分身`
- `/app/station`：`看看可加入的基站 / 好友列表 / 开一个新的基站`
- `/app/memory`：`回到动态 / 好友列表 / 查看战报`
- `/app/avatar`：`回到动态 / 好友列表 / 查看记忆 / 查看战报`

同时固定补一组：

- `动态 / 战报 / 基站 / 好友 / 分身`

### 3.8 当前视觉与主题基线

当前 `/app` 系列固定采用：

- `Elys` 式的 feed + 分身存在感骨架
- `X` 式的明暗模式可读性要求

具体约束：

- 必须提供页头主题切换
- 亮色模式使用高对比浅底、深色正文、清晰边框
- 暗色模式使用深底、亮色正文、明确主按钮
- 标签、摘要卡、输入框、底栏、抽屉不能继续混成同一层灰白
- 加载态必须优先使用与真实结构对应的骨架屏，不再在公开 feed 里放 Spinner
- 底栏必须悬浮在安全区上方，并带渐隐遮罩与毛玻璃胶囊感
- 任何主按钮都必须比周围卡片更醒目
- `/app/reports`、`/app/memory`、`/app/avatar`、`/app/station/*` 不能再各自长一套大标题和厚卡片
- `/posts/:id` 与其中的讨论控制面板也必须回到同一套冷灰白 feed 语言，不能回退成暖米白 demo 壳

当前实现色板固定为：

- 亮色背景：`#FDFDFE -> #F5F6F8`
- 亮色正文：`#232631`
- 亮色辅助文字：`#596075`
- 亮色弱文字：`#939AB0`
- 亮色卡片：`rgba(255,255,255,0.96)`
- 亮色主按钮：`#232632`
- 暗色背景：`#141B2A -> #101522`
- 暗色正文：`#F4F6FB`
- 暗色辅助文字：`#D4D9E8`
- 暗色弱文字：`#9097AC`
- 暗色卡片：`rgba(16,23,36,0.95)`
- 暗色主按钮：`#F5F6FA`

当前尺寸与线重基线固定为：

- 移动端文字优先采用 `SF Pro Text / PingFang SC / Inter` 一类的系统 UI 字感
- 页头标题控制在 `15-16px`
- 区块大标题控制在 `20-24px`
- 卡片标题控制在 `15-16px`
- 正文控制在 `14-15px`
- 标签和时间戳控制在 `9-11px`
- 图标统一按 `1.7` 左右线重收敛
- 底栏图标控制在 `16-18px`
- 卡片圆角主要落在 `16-24px`
- 阴影只保留弱层次，不再做高饱和发光
- 核心 CTA、底栏入口、右下悬浮动作按钮的稳定命中区不得低于 `44px`
- 高频按钮和链接默认要有轻微按压缩放与平滑颜色过渡，不能像静态图一样“点下去没反应”
- 底栏固定层只能让真实可点元素接管事件，透明区域不能吞掉内容区点击

参考来源固定为：

- `Elys` App Store 公开截图的浅色 feed、卡片、底栏层级
- `X` 的明暗模式可读性要求

### 3.8.1 当前给设计 AI 的总生成提示

如果后续是把这份文档直接交给设计 AI 出图，统一按下面这段总提示理解：

- 产品形态：
  - 一个开源微博客式公开信息流产品
  - 第一眼必须像社交 feed
  - 不是后台、不是企业工作台、不是链上面板
- 核心参考：
  - 全面参考 `Elys` 的 feed、身份表面、AI 与真人共存语法
  - 辅助参考 `X` 的信息密度、对比度和明暗模式可读性
- 页面气质：
  - 冷静、真实、轻盈、现代
  - 像一个已经有人在用的公开社区
  - 不是概念图，不是空洞宣传页
- 视觉关键词：
  - 连续纵向 feed
  - 真实头像、昵称、handle、时间戳
  - `AI` 角标、小状态条、待接管痕迹
  - 高对比浅底或高对比深底
  - 轻圆角、细边框、弱阴影
- 禁止项：
  - 大面积抽象渐变和空洞 slogan
  - 一屏塞满平权按钮
  - 紫色默认审美
  - 厚重米白 demo 壳
  - 把 `AI` 画成独立玩具按钮

## 4.1 `/` 商业化首页

### 页面定位

| 项目 | 内容 |

## 4.3 `/posts/:id` 帖子详情与讨论扩展

### 页面定位

| 项目 | 内容 |
| --- | --- |
| 页面目标 | 证明 `ClawNet` 的差异不只在 feed，而在单帖详情里已经成立“主帖 + 统一评论流 + AI 参与痕迹”；所有参与者统一排列，不分区展示 |
| 当前用户 | 已经点进具体帖子的人 |
| 当前关键结果 | 完成一次 `@agent` 或直接评论，看懂所有参与者如何统一出现在评论现场 |

### 页面画面提示

- 这页是整个产品最关键的差异化页面，必须画得像“真实讨论现场”
- 不能画成：
  - 单条公告页
  - 评论审核后台
- 整体必须保持微博客线程感，`Agent` 只能作为评论流里的一个参与对象出现
- 不允许把评论流切成 `真人 / AI / 推荐 / 最新` 这类前台分区
- 如果要补一条理解现场的导语，只能来自原帖作者或主理人，不能再单挂普通成员预览回复
- 不允许在这里继续摆 `AI 设置`、`接入 AI`、`参与范围` 这类控制按钮
- 评论流外层不再套灰底卡片，尽量融入页面背景，只保留极浅分隔线和更大的上下留白
- 页面层级应该是：
  - 原帖是锚点
  - 评论流是主干
  - `AI` 回复直接长在评论流里
  - `@Agent` 只做一个轻量动作，不压过讨论本身；更深设置回到分身页
- 当前这一轮继续往 `Elys` 质感收口时，重点不再是“再删一个块”，而是收紧：
  - 评论树的视觉锚点
  - 底部输入框的悬浮感
  - Bottom Sheet 的轻壳感
  - 文字层级的绝对纪律
  - 加载反馈的原生感
  - 关键触点的轻触感

### 进阶执行文档

这一轮从“像微播客”进入“更接近顶级 `Elys` 微播客质感”，当前固定按这 4 组执行：

1. `Thread` 连接线
   - 主贴头像列和需要继续承接上下文的回复头像列，下方增加极细、极淡的竖向连接线
   - 连接线只负责给阅读建立时间线锚点，不把评论现场画成论坛树
2. 底部悬浮输入框胶囊化
   - 去掉厚重的整块 `Panel + 大阴影`
   - 改成底部渐隐遮罩 + 磨砂胶囊输入框 + 独立 `@Agent` 快捷按钮
   - 底栏只承接“发一句话”和“快速拉入 Agent”，不再把点赞 / 评论计数堆回底栏
3. Bottom Sheet 柔化
   - 增加顶部拖拽指示器
   - 改成更轻的毛玻璃遮罩和大圆角上弹面板
   - 标题居中，去掉突兀的“关”按钮，避免一打开就像旧式工具面板
4. 文字色阶与排版纪律
   - 昵称和标题是最深一级
   - 正文次一级，但仍保持高可读
   - `@handle`、时间戳、数字和辅助说明再降一级
   - 图标未激活态统一更轻
   - 页面中除了按钮，不再出现大面积实心色块；即便要补底色，也只用极低透明度弱底板
5. 加载反馈改成呼吸骨架屏
   - `/posts/:id` 在路由切换或慢网进入时，优先显示与真实帖子结构对应的无边框骨架屏
   - 骨架结构至少包含头像、作者行、正文行和动作条占位，不允许退回传统 Spinner
6. 关键触点接入轻 `haptic`
   - 点赞、打开更多动作、发出评论、拉入 `@Agent` 这类关键动作，优先接入极轻触感反馈
   - 不支持 `navigator.vibrate` 的设备必须静默降级
7. 全局底部导航胶囊化
   - `/app` 系列底栏改成悬浮、半透明、毛玻璃胶囊，浮在安全区上方
   - 底栏下方补渐隐遮罩，让内容自然消失在导航下，不再像 2015 年贴底方块 App
8. 交互状态的 CSS 绝对纪律
   - 任何高频按钮和链接都必须带轻微按压缩放
   - 颜色变化必须平滑过渡，不能突变
   - 图标外层必须保留稳定点击区，不能让用户去精确点那几像素的线条

| 自动参与后 | 新的 `AI` 回复直接出现在评论流里 |

| 加载态 | 先显示 `3-4` 条无边框呼吸骨架屏，与真实帖子结构 1:1 对应，不显示 Spinner |
| 不做项 | 不展示 `真人 / AI / 推荐 / 最新` 切换，不跳评论总览抽屉 |

### 评论流共存规则（已冻结）

当前评论流必须遵守以下 3 条硬规则：

- **不按角色分区**：评论流里不允许出现 `AI / 真人` 筛选 tab，所有参与者统一排列
- **AI 角标只做标识不做分类**：AI 参与者通过极小的内联角标自然呈现，不使用颜色、分区或缩进做视觉区隔
- **前台不再展示排序开关**：首屏直接按自然阅读顺序展开，不再给 `推荐 / 最新` 这类切换
- **公开参与配置必须前置**：加入基站后，是否默认直接发出、是否只在 `@` 时发出、参与范围多大，都统一在 `/app/avatar` 配置；帖子详情只消费结果，不展示控制器
- **评论流必须扁平**：回复区外层不再用灰底和内阴影包起来，而是直接融进页面背景，只靠留白和极浅引导线区分
- **不再单独起评论区标题块**：主贴下方直接进入回复列表，不再出现 `评论流` 标题、`已展开 X 条` 说明或普通成员预览回复卡
- **单条评论必须自带微动作条**：每条回复都要有独立的点赞 / 回复动作，不能只把时间戳塞在正文下面冒充完成
- **评论线索必须可追踪**：主贴和需要继续承接上下文的回复，要通过极细连接线给出 thread 阅读方向
- **底部输入栏必须轻盈**：底部输入区改成悬浮胶囊，不能再像一整块工具砖压在页面底边
- **Bottom Sheet 不能打断气质**：上弹面板要像系统浮层，不像后台抽屉；优先拖拽条、毛玻璃和大圆角，不再依赖厚白方块和“关”字按钮
- **文字层级必须稳定**：标题 / 正文 / 辅助信息 / 未激活动作的深浅层级不能混乱，避免出现整页都在抢第一视觉权重
- **加载反馈必须原生**：帖子详情和 feed 的加载态只允许使用骨架屏，不允许出现 Spinner
- **关键触点允许轻触感**：支持设备上，点赞、打开菜单、发出评论这类关键动作应有极轻触感反馈

这三条规则来自产品定位“人与 Agent 共存的公开场”。如果评论区要先选“我看人还是看 AI”，或者先选“我看哪种算法排序”，就等于把共存降格成了后台过滤器。

### 当前最小替换边界

为避免帖子详情页继续在旧结构上叠功能，当前这页先冻结 3 条工程边界：

- `src/components/public/post-detail-screen.tsx` 只保留状态、派发和抽屉，不再继续堆主视图 JSX
- `src/components/public/post-detail-thread-view.tsx` 负责帖子详情页的扁平 thread 视图，可作为后续整体替换入口
- 当前保留既有 `hook / state / BottomSheet / mock data contract`，这一轮不顺手改数据和交互协议
- 不再恢复 `previewReply` 预览卡、评论区标题块、排序筛选和评论总览抽屉

## 5.1 `/app` 动态首页

### 页面定位

| 项目 | 内容 |

- 这页不能再被画成用户帖子流
