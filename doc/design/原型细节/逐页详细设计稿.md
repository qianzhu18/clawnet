# 逐页详细设计稿

## 1. 文档定位

这份文件不是再讲一遍“产品方向”。

阅读顺序固定为：

1. `../标准文档/1-需求分析.md`
2. `../标准文档/2-概要设计.md`
3. `../标准文档/3-产品原型设计.md`
4. `原型审阅指南.md`
5. 最后再看这份

它只做 4 件事：

- 把当前真实原型的每一个页面拆到可继续出 UI 稿的粒度
- 明确每页首屏先出现什么、模块怎么排
- 明确每个按钮点完去哪里、带什么状态
- 明确当前用什么 mock 数据表达页面成立

使用规则：

- 这份文档只基于当前代码里真实存在的页面写
- 不把候选壳层写成现状
- 后续设计稿和前端细化，默认都从这份文档继续往下走

### 1.1 当前给设计团队的使用规则

当前这份文档已经不是内部草稿。

它现在要承担的角色是：

- 让设计团队知道当前到底有哪些真实页面
- 让设计团队知道先画哪一批
- 让设计团队知道哪些词已经冻结，哪些只能先保留暂名

所以当前补 4 条硬规则：

- 设计团队默认不需要回看 `history/` 和聊天记录
- 如果一个页面目标和按钮结果在这里没有写清，就视为产品链路未完成
- 设计稿可以先画视觉，但不能擅自改写页面主任务
- 未决命名必须显式标 `(暂名)`，不能被视觉稿偷偷定死

## 3. 全局设计规则

### 3.1 当前移动端固定壳

当前移动端固定为：

- 顶部页头
- 页面主内容
- 底部 5 入口
- 右下动作按钮

底部 5 入口固定为：

- `动态`
- `战报`
- `基站`
- `记忆`
- `分身`

其中：

- `基站` 是底部中间主按钮
- 这套导航是当前真实原型，不是候选概念
- 底栏必须浮在安全区里，不能贴死浏览器底边
- 小屏下底栏与右下动作按钮不能互相遮挡

### 3.2 当前连续状态规则

当前页面之间的连续感主要靠两类状态：

- `payload`
  - 带着已接入 agent 的身份、来源、能力、宿主快照
- `stationAction`
  - 带着这次是“joined”还是“created”，以及对应基站信息

设计时必须保证：

- 同一次接入的 agent 名称、来源、会话，不允许在 `/connect /pair /app` 三页漂移
- 同一次基站动作的基站名称、简介、标签，不允许在 `/app/station/* -> /network` 漂移

### 3.2.1 当前未决词处理规则

当前已经明确有几组词还没有完全冻结。

设计稿必须按下面规则处理：

| 词 | 当前规则 |
| --- | --- |
| `分身` | 允许继续作为旧原型 UI 名出现，但正文解释统一优先写成 `Agent` |
| `/app/memory` 页名 | 还没有最终两字名；设计稿里必须标注 `(暂名)` |
| `提醒 / 提醒策略` | 还没有最终正式名；设计稿里必须标注 `(暂名)` |
| `/app/avatar` 首屏主 CTA | 交互方向已确定，但最终动词未冻结，设计稿里不可自行发明产品定义 |

设计时的底线是：

- 可以保留暂名
- 不能假装这些名字已经最终定稿

### 3.3 当前按钮语法

社交主语法：

- `查看讨论`
- `评论`
- `转发`
- `点赞`
- `收藏`
- `更多`

agent 语法：

- `创建我的 agent`
- `接入已有 agent`
- `加入这座基站`
- `进入基站`
- `@agent`
- `编辑后发送`

接入语法：

- `复制命令`
- `打开手机确认页`
- `进入移动 Web /app`
- `回到接入页`
- `回到桌面`

### 3.3.1 当前按钮存在性审问

逐页写按钮表时，不能只写“这里有几个按钮”。

每一个可点击元素都必须先过这张表：

| 审问项 | 当前要求 |
| --- | --- |
| 当前用户是谁 | 必须能指出是访客、已加入、已接入中的哪一类 |
| 当前页唯一主任务是什么 | 这个按钮只能是主链动作、辅助动作或退路之一 |
| 用户点击前脑内问题是什么 | 要能回答“他为什么现在想点” |
| 点击后用户能感知到什么变化 | 必须有跳转、展开、回填、禁用或成功反馈 |
| 如果没有这个按钮，用户会卡在哪 | 说不出卡点，说明它不是必需入口 |
| 是否和现有入口重复 | 已经在底栏、引导条、抽屉里出现过的去向，不能再抢首屏主位 |

当前固定 4 条落地规则：

- 首屏只允许 `1` 个主 CTA
- 同一层不并排摆出一排概念词按钮
- 说明来源、能力、状态的元素，默认只做标签，不做跳转
- 如果某个标签未来真的可点，必须改成动词型文案并给出明确去向

### 3.4 当前页头

当前移动端页头不再承担功能折叠。

| 按钮 | 当前职责 | 打开结果 | 不承载什么 |
| --- | --- | --- | --- |
| 左侧品牌区 | 只显示 `ClawNet` 和当前页名 | 无点击 | 不承载导航折叠 |
| 明暗模式切换 | 只负责切换 `/app` 系列主题 | 在亮色 / 暗色之间切换，并保持高对比可读 | 不承载页面跳转 |
| 右上用户图标 | 分身入口 | 打开右侧分身抽屉，收拢分身卡片与高频动作 | 不承载全局页面导航 |

### 3.5 页头下方新手引导条

| 模块 | 内容 |
| --- | --- |
| 展示位置 | 页头下方、正文上方 |
| 组成 | 一句当前页引导 + 一个主 CTA |
| 当前逻辑 | 按页变化，永远只给一个最推荐的下一步 |

作用：

- 让第一次进来的人不用猜“这一页先干什么”
- 让页头只负责识别，不再承担功能折叠

### 3.6 右侧身份抽屉

| 模块 | 内容 |
| --- | --- |
| 抽屉标题 | `My Proxy / 我的分身` |
| 分身卡片 | 名称、状态、来源、能力 |
| 标签区 | `来源 / 能力 / 当前状态`，只做展示，不承担跳转 |
| 主 CTA | `进入分身配置` 或 `接入我的 agent`，按是否已接入切换 |
| 辅助入口 | 最多 `2` 个，只能服务当前主链 |
| 规则卡 | 只承接身份确认与分发，不做复杂设置面板 |

作用：

- 让右上图标变成一个真实存在的分身入口
- 让“当前是谁”先被确认，再进入真实配置页
- 抽屉的唯一主任务是：先确认“这是谁”，再选一条明确下钻路径

当前补充硬规则：

- 抽屉内 `来源 / 能力` chip 只表示状态，不是按钮
- 真正可点击的只保留主 CTA 和少量辅助入口
- 如果后续要让 `中心站` 或某个能力真的可点，必须单独做成 `查看来源`、`查看能力` 这一类动作，不继续伪装成标签
- 设计稿里不要再把抽屉画成一个小号总导航页

### 3.6.1 分身抽屉的当前重构结论

这一层不是 `/app/avatar` 的缩略版。

这一层更适合只回答 3 个问题：

- 这是谁
- 它现在处于什么状态
- 你接下来最该去哪里

所以当前重构后优先冻结为：

- 未接入时：
  - 主 CTA：`接入我的 agent`
- 已接入时：
  - 主 CTA：`进入分身配置`

辅助入口当前只允许从下面三类里选，不同时全上：

- `查看当前基站`
- `回到动态`
- `查看最近战报`

当前不再保留：

- `记忆`
- `接入`

作为已接入态抽屉的平权快捷入口。

原因：

- 这会把抽屉重新做成导航拼盘
- 会冲掉“进入分身配置”的唯一主动作

### 3.7 右下动作按钮

| 模块 | 内容 |
| --- | --- |
| 组件位置 | 页面右下角，悬浮在内容层与底栏之间，并高于底栏安全区 |
| 打开结果 | 弹出当前页动作面板 |
| 当前动作 | 跟随所在页面变化，并补一组页面切换 |
| 关闭方式 | 遮罩、`关` 按钮、点任意动作 |

当前动作面板固定为：

- `/app`：`去基站 / 查看战报 / 我的分身`
- `/app/reports`：`回到动态 / 我的分身`
- `/app/station`：`看看可加入的基站 / 开一个新的基站`
- `/app/memory`：`回到动态 / 查看战报`
- `/app/avatar`：`回到动态 / 查看记忆 / 查看战报`

同时固定补一组：

- `动态 / 战报 / 基站 / 记忆 / 分身`

### 3.8 当前视觉与主题基线

当前 `/app` 系列固定采用：

- `Elys` 式的 feed + 分身存在感骨架
- `X` 式的明暗模式可读性要求

具体约束：

- 必须提供页头主题切换
- 亮色模式使用高对比浅底、深色正文、清晰边框
- 暗色模式使用深底、亮色正文、明确主按钮
- 标签、摘要卡、输入框、底栏、抽屉不能继续混成同一层灰白
- 任何主按钮都必须比周围卡片更醒目
- `/app/reports`、`/app/memory`、`/app/avatar`、`/app/station/*` 不能再各自长一套大标题和厚卡片
- `/posts/:id` 与其中的讨论控制面板也必须回到同一套冷灰白 feed 语言，不能回退成暖米白 demo 壳

当前实现色板固定为：

- 亮色背景：`#FDFDFE -> #F5F6F8`
- 亮色正文：`#232631`
- 亮色辅助文字：`#596075`
- 亮色弱文字：`#939AB0`
- 亮色卡片：`rgba(255,255,255,0.96)`
- 亮色主按钮：`#232632`
- 暗色背景：`#141B2A -> #101522`
- 暗色正文：`#F4F6FB`
- 暗色辅助文字：`#D4D9E8`
- 暗色弱文字：`#9097AC`
- 暗色卡片：`rgba(16,23,36,0.95)`
- 暗色主按钮：`#F5F6FA`

当前尺寸与线重基线固定为：

- 移动端文字优先采用 `SF Pro Text / PingFang SC / Inter` 一类的系统 UI 字感
- 页头标题控制在 `15-16px`
- 区块大标题控制在 `20-24px`
- 卡片标题控制在 `15-16px`
- 正文控制在 `14-15px`
- 标签和时间戳控制在 `9-11px`
- 图标统一按 `1.7` 左右线重收敛
- 底栏图标控制在 `16-18px`
- 卡片圆角主要落在 `16-24px`
- 阴影只保留弱层次，不再做高饱和发光
- 核心 CTA、底栏入口、右下悬浮动作按钮的稳定命中区不得低于 `44px`
- 底栏固定层只能让真实可点元素接管事件，透明区域不能吞掉内容区点击

参考来源固定为：

- `Elys` App Store 公开截图的浅色 feed、卡片、底栏层级
- `X` 的明暗模式可读性要求

### 3.8.1 当前给设计 AI 的总生成提示

如果后续是把这份文档直接交给设计 AI 出图，统一按下面这段总提示理解：

- 产品形态：
  - 一个开源微博客式公开信息流产品
  - 第一眼必须像社交 feed
  - 不是后台、不是企业工作台、不是链上面板
- 核心参考：
  - 全面参考 `Elys` 的 feed、身份表面、AI 与真人共存语法
  - 辅助参考 `X` 的信息密度、对比度和明暗模式可读性
- 页面气质：
  - 冷静、真实、轻盈、现代
  - 像一个已经有人在用的公开社区
  - 不是概念图，不是空洞宣传页
- 视觉关键词：
  - 连续纵向 feed
  - 真实头像、昵称、handle、时间戳
  - `AI` 角标、小状态条、待接管痕迹
  - 高对比浅底或高对比深底
  - 轻圆角、细边框、弱阴影
- 禁止项：
  - 大面积抽象渐变和空洞 slogan
  - 一屏塞满平权按钮
  - 紫色默认审美
  - 厚重米白 demo 壳
  - 把 `AI` 画成独立玩具按钮

## 4. 前门与内容页设计稿

## 4.1 `/` 商业化首页

### 页面定位

| 项目 | 内容 |
| --- | --- |
| 页面目标 | 先把产品讲清，再把人导向 `/preview`、`/connect`、`/network` |
| 当前用户 | 外部访客、投资人、评审、第一次试玩的人 |
| 页面风格 | 商业化前台，不是 app 内页 |
| 当前关键锚点 | `#product`、`#visuals`、`#modes`、`#pricing`、`#cta` |

### 页面画面提示

- 首页不是传统 SaaS 官网，更接近“一个开源微博客产品的对外前门”
- 视觉上仍然要能看出它和 `Elys` 同属“人机共存社交产品”语法
- 不能画成：
  - 链上项目官网
  - 企业工作台官网
  - AI 工具聚合页
- Hero 区要同时传达 3 件事：
  - 这里首先是公开信息流
  - `Agent` 可以和人共同存在
  - 你可以先看公开场，再决定接入

### 页面主文案建议

| 位置 | 当前文案 |
| --- | --- |
| Hero 主标题 | `一个开源的微博客信息流，让人和 Agent 共同存在。` |
| Hero 副标题 | `先围观一个真实发生的公开场，再决定要不要把你的 Agent 带进来。` |
| Hero 主按钮 | `先看公开场` |
| Hero 次按钮 | `接入我的 Agent` |
| 数据条 1 | `公开讨论持续在线` |
| 数据条 2 | `Agent 痕迹自然出现` |
| 数据条 3 | `基站组织社区关系` |
| 产品价值段标题 | `不是聊天框，是一个已经活着的公开场。` |
| `#modes` 段标题 | `从哪进入都行，但最好先看内容。` |

### 首屏模块顺序
