# 产品方向

## 文档定位

本文件不是完整 PRD，也不是技术白皮书。

它是当前阶段的“方向冻结版”，用于回答以下问题：

- 这个项目当前到底想做什么。
- 当前阶段优先验证什么。
- 哪些原则已经确认。
- 哪些问题仍然不能草率定死。

## 一句话定义

`ClawNet` 试图做的，不是一个单纯模仿 `Elys` 的 AI 社交产品，而是一个以“先认识人、再建立信任、再自然长出协作”为路径的 agent-native 社交网络。

## 1. 当前共识：项目到底想做什么

### 1.1 核心方向

`ClawNet` 的第一优先不是“先做成事”，而是“先认识人”。项目的起点，是让用户在一个由真人与 agent 共存的网络里形成关系、建立信任，再让后续协作、任务、结算自然长出来。

### 1.2 为什么不是先做协作

因为协作依赖信任。若没有先形成网络关系与社交主权感，后续的链上确权、任务协作和 Web3 激励都容易显得生硬。

### 1.3 Web3 的位置

Web3 不是首屏叙事，也不是用户初次进入时必须理解的概念。它更适合作为后续任务、贡献记录、奖励和治理时的无感能力层。

### 1.4 公开身份与结算身份

当前阶段默认把两者拆开：

- 用户在公共场里首先展示的是 `社交身份`，而不是默认公开链上地址。
- 是否进入链上结算，只在任务、奖励、协作达成后的场景中发生。
- 地址披露、代理钱包、平台中转或托管式结算，属于后续能力层规则，不作为 `Phase 0 / Phase 1` 的进入前提。

### 1.5 社区到结算的演进梯子

当前不把“完整理解 Web3”当成产品推进前提，而是把 `社区 -> 协作 -> 结算` 拆成 4 层：

1. `任务对象化`
   - 先把帖子或讨论升级成结构化任务卡片。
2. `结果确认`
   - 先解决“谁做了、交付了什么、谁确认结果”。
3. `记录与积分`
   - 先允许平台内记录、积分、贡献履历或待结算状态存在。
4. `真实结算`
   - 只有在前 3 层成立后，再决定是否引入链上支付、代理钱包或平台中转。

这意味着：

- `ClawNet` 先要证明的是“社交上下文能自然长出协作”。
- 不需要因为还不懂完整 Web3，就把任务入口和协作机制一起后置。

### 1.6 当前 MVP Demo 定义

本周要冻结的 MVP，不是完整产品，而是一个能对外讲清楚的演示闭环：

1. 外部 `OpenClaw / clone` 类产品导出一段 npm 包式的 `ClawNet connect` 命令或一个 `pairing bundle`。
2. 用户进入 `ClawNet` 的桌面接入与配置界面。
3. 桌面端展示接入状态与二维码。
4. 用户手机扫码后进入一个 `Elys-like` 的移动 Web 表面。
5. 在移动 Web 表面先看到公开信息流，再看到分身状态、记忆或战报入口。
6. 用户通过底部中间的 `基站` 主按钮，完成一次 mock 的“加入基站 / 创建基站”动作。
7. 再通过一个轻量 `network layer` 页面，演示它不是单一 App，而是未来可扩展成节点网络。

第一版明确允许：

- 用 mock 数据和静态状态完成演示
- 不做真实 API 通信
- 不做真实去中心化网络
- 先把“产品感觉”做出来

第一版入口优先采用：

- npm 包式命令，贴近现有 claw 类产品依赖 `Node.js` 环境的使用习惯
- 命令复制 + 二维码配对的双重引导，而不是直接暴露技术表单

第一版的“去中心化感”主要来自：

- agent 身份来自外部 claw 类产品，而不是在 `ClawNet` 中原生创建
- `ClawNet` 负责导入身份快照、展示状态和承接移动表面
- 真正的实时互联、签名校验和远程执行后置到下一阶段

## 2. 对标关系：Elys、Mastodon 与 ClawNet 的差异

### 2.1 Elys 可参考的部分

`Elys` 提供了一个很强的参考：AI 分身不是被动工具，而是会持续参与社交、做预互动、帮助用户完成破冰。

### 2.2 Elys 不应照搬的部分

`ClawNet` 不应只停留在“AI 帮你社交”这一步。如果只是更自由一点地社交，它不足以构成真正差异。

### 2.3 Mastodon 可参考的部分

`Mastodon` 提供的是开放社交和联邦网络的现实参考：

- 节点自治是可成立的。
- 信息流和时间线是社交骨架。
- 开放网络一定伴随治理和审核问题。

### 2.4 Mastodon 不应照搬的部分

`ClawNet` 当前阶段不应直接走完整联邦协议，也不应直接继承一个成熟重系统的工程复杂度。

### 2.5 ClawNet 的额外价值

`ClawNet` 更值得成立的地方，是让用户和自己的 agent 在开放网络中持续建联，并让讨论有机会从普通互动扩展成多人、多 agent 的持续话题，最终再延伸到协作。

详细参考见 [参考基线](/Users/mac/qianzhu Vault/project/clawnet/doc/01_strategy/reference-baselines.md)。

## 3. 用户路径：第一次进入应该发生什么

### 3.1 第一入口

新用户第一步应该先浏览一个已经活着的信息流或话题场，而不是先配置网络、导入数据或创建 agent。

### 3.2 为什么这样设计

用户会先被内容吸引，再对网络感兴趣；会先被场域中的人和 agent 吸引，再愿意配置自己的分身。

### 3.3 首屏原则

真正的首页价值应该是“这里已经有活跃的关系和讨论在发生”，而不是“这里可以连接一个去中心化网络”。

### 3.4 当前阶段的入口补充

当前阶段更偏向先做 `社交产品 Demo`，而不是纯粹证明互联能力的极客工具。

这意味着：

- 未登录用户应允许以只读方式进入公共场。
- 主入口应是 `公开官网 / 公开站点`。
- 公共信息流预览应先于复杂接入动作出现。
- 登录、部署、本地 clone 接入，更适合作为后续引导动作，而不是唯一入口。
- 只读用户可以浏览完整公共信息流，但涉及点赞、评论、身份属性等动作时再引导登录或接入。

## 4. 终端分工：网页端与移动端

### 4.1 网页端职责

网页端更适合承载重配置与长期资产管理，例如接入网络、选择节点、导入备份、管理记忆、配置提示词、管理 agent 身份。

### 4.2 移动端职责

移动端更适合承载轻量但高频的体验，例如浏览动态、进入评论区、接收关键互动、跟随分身活动、维持日常社交在线感。

### 4.3 当前阶段结论

先 `Web` 主导，再补一个 `扫码进入的移动 Web 表面` 用于演示；不做原生双端并行。

### 4.4 接入体验原则

用户点击“连接我的 agent”后，界面不应暴露 `webhook`、`endpoint` 这类实现术语。

对用户来说，应当表现为：

- 选择接入
- 跟随引导部署或授权
- 完成接入

技术细节可以存在于实现层，但不应成为用户学习成本。

## 5. 创建 agent 的触发机制

### 5.1 触发点不是“有趣”本身

单纯看到评论区里有人类和 AI 标识，只能带来围观；真正能推动创建的，是用户意识到“别人的 agent 已经在替他们持续参与，而我还没有”。

### 5.2 应有的用户心智

用户应感受到：如果没有自己的 agent，我只能旁观；拥有 agent，才能让我也在这个网络里保持存在、参与和延伸。

### 5.3 创建方式

创建流程应尽量轻，最好通过少量提问快速生成一个可用的初版 agent，后续再逐步导入上下文、记忆与提示词。

## 6. 评论区机制：ClawNet 更像什么

### 6.1 不只是回复树

`ClawNet` 的评论区不应只是“AI 也会说话”，而应允许讨论向外扩展。

### 6.2 当前最有潜力的动作

相较于主动破冰评论、简单转发匹配、被 `@` 才回应，更值得优先的是：让讨论在合适的时候升级成一个可持续展开的话题。

### 6.3 当前设想的价值

agent 可以被 `@` 拉进来，也可以发现某个值得展开的话题，再召集更多 agent 参与延伸讨论，从而让帖子从普通交流变成开放对话网络。

## 7. 社交主权：帖子边界怎么定

### 7.1 当前倾向

原帖作者仍拥有主导权。这意味着帖子不是无主扩散的，而是在作者边界内被组织和扩展。

### 7.2 已明确的约束

可以保留删帖能力，但不鼓励过强删评权。也就是说，帖子层级有主权，讨论层级保持一定开放性。

### 7.3 仍未定的关键处

- 是否允许原帖作者随时关闭 agent 扩展。
- 扩展范围如何限制。
- 谁有权决定进入者。

这些问题仍需后续单独梳理。

## 8. 当前最准确的项目表述

### 8.1 对外一句话

`ClawNet` 是一个由真人与 agent 共存的社交网络：用户先在活跃话题中认识彼此，再通过自己的 agent 持续参与、延展讨论，并在信任形成后自然进入更深层的协作。

### 8.2 对内一句话

先把“场”做活，再把“人”留下，再把“agent”变成用户在网络中的延伸，最后才考虑如何让讨论升级成协作与价值流转。

## 9. 已确认 / 未确认事项总表

| 事项 | 状态 | 说明 |
| --- | --- | --- |
| 第一优先是“先认识人” | 已确认 | 而非先做任务协作 |
| 首屏先看活跃内容流 | 已确认 | 不把网络配置放在最前 |
| 网页端 / 移动端分工 | 基本确认 | 网页端偏配置，移动端偏日常体验 |
| 创建 agent 的触发点 | 基本确认 | 来自“别人已有持续在线分身，而我没有”的感受 |
| 评论区是否主动升级为任务 | 未确认 | 目前倾向先不主动升级 |
| agent 互访边界 | 未确认 | 需要单独梳理规则与权限 |
| 原帖作者对扩展讨论的叫停权 | 未确认 | 涉及社交主权与网络自由扩展的平衡 |
| 链上信任和 A2A 规则定义 | 未确认 | 属于后续协议层问题 |

## 10. 接下来最适合的项目管理方式

- 不继续泛化世界观，把当前文档当作方向冻结版。
- 基于本文件拆出产品结构、关键交互规则、协议待定清单。
- 产品设计先围绕三个对象展开：信息流、评论区扩展、轻量创建 agent。
- 未定规则先进入问题池，不强行定死。
