# commands

这里不是 Shell 命令手册，而是“你现在可以直接让 AI 做什么”的入口。

## 当前 AI 可执行的 6 类动作

- `文件操作.md`
- `文档操作.md`
- `Markdown工作台.md`
- `测试操作.md`
- `编码操作.md`
- `附加要求.md`

## 当前最常用的 command

### 目录整理

- `梳理当前文档目录并给出阅读顺序`
- `把重复文档合并成少量 active 主文档`
- `把旧文档归档到 history`
- `把 design 文档改成软件工程主线`

### 标准文档整理

- `把当前需求整理成一份对外可读的需求分析`
- `把概要设计补成软件工程 2 阶段主文档`
- `把产品原型设计收成唯一主入口`
- `补齐 4-7 阶段占位标准文档`

### Markdown 工作台整理

- `检查当前 doc 工作台缺什么`
- `把 markdown 治理规则收口到 commands`
- `给我一条文档任务的完成定义`
- `给我跑一次 doc:check`

### 原型整理

- `把当前原型拆成逐页详细设计稿`
- `把按钮和跳转收成一张主表`
- `把参考产品拆解写进原型参考`
- `把 3000 端口页面改得和文档一致`

### 本机验证

- `启动 3000 端口让我手工审核原型`
- `跑 build 和 lint`
- `给我一个本机逐页验证顺序`
- `把 /prototype 改成当前审稿台`

### 页面实现

- `把首页改得更像产品页`
- `补齐基站 mock 数据`
- `把 AI 身份显示成头像角标`
- `把某个页面改到能直接展示给设计师`

## 使用原则

1. 先整理 `design/标准文档/`
2. 先固定文档入口、验收和自检，再扩更多目录
3. 再下钻 `概要细节/` 和 `原型细节/`
4. 页面改动要同步到 `localhost:3000`
5. 重要结论不能只停留在聊天里
6. 每次有效代码改动都要同步到 git，保证分支内可追溯
7. 面向用户的页面文案不能直接暴露开发说明语气
