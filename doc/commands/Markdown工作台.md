# Markdown工作台

## 1. 这份文件回答什么

这份文件只回答 4 件事：

- 当前 `Markdown` 文档工作台应该从哪里开始整理
- 当前这套 `doc/` 目录离 `harness engineering` 还缺什么
- 什么叫“文档任务做完了”
- AI 每次整理完以后该怎么自检

它不负责：

- 产品需求本身
- 页面逐稿设计
- 代码实现细节
- 历史材料的完整搬运

## 2. 当前判断

当前项目已经有一套公开文档主线：

- `AGENTS.md`
- `doc/README.md`
- `doc/design/标准文档/`
- `doc/design/概要细节/`
- `doc/design/原型细节/`
- `doc/history/`

这说明你已经完成了“把文档从杂乱知识堆，整理成公开阅读顺序”的第一步。

但它还不是一个稳定可复用的 `Markdown harness`。

当前真正缺的，不是继续开更多目录，而是补齐下面 3 个能力：

1. `AI 知道去哪改`
2. `AI 知道什么叫改完`
3. `AI 改完以后能自己检查`

## 3. 当前最适合选的重复任务

不要先试图把整个项目都做成 `harness engineering`。

当前最适合被固定下来的每周重复任务是：

`标准文档与原型入口的一致性整理`

它固定包含 5 个动作：

1. 检查 `doc/design/标准文档/1-7` 是否仍然各司其职
2. 检查 `doc/README.md`、`doc/design/README.md`、`doc/commands/README.md` 的阅读顺序是否一致
3. 检查新增结论有没有同步回标准文档或专题补充
4. 检查旧稿有没有继续停留在 active 区而不是进入 `history/`
5. 检查文档说的页面入口，能不能在 `localhost:3000/prototype` 与 `/validation` 上对应起来

如果这个任务能稳定三周，后面再扩到：

- 参考文档治理
- 详细设计治理
- 开发和测试治理

## 4. 三周推进法

### 第一周：先整理结构和规范

这周只做“让 AI 找得到”。

固定动作：

- 保持 `AGENTS.md` 只负责项目协作边界和目录入口
- 具体文档治理规则收口到 `doc/commands/`
- 公开阅读入口固定为 `doc/README.md -> doc/design/README.md -> 标准文档 1-7`
- 同类规则不重复写在多个 active 主文档里
- 新结论优先写回现有标准文档，不新建平行主稿

这一周的完成标准不是“文档写得多”，而是：

- 第一次进项目的人知道先读哪几份
- AI 知道文档该落到哪个目录
- 同一个问题不会出现两个 active 主入口

### 第二周：再设验收标准

这周只做“让 AI 知道什么叫完成”。

每次文档任务都必须说清 4 件事：

1. 本次修改服务哪一阶段
2. 结论应该落到哪份文件
3. 哪些文件必须同时同步
4. 完成后用什么口径验收

当前 `Markdown` 任务的最小验收口径固定为：

- 目标文件放对目录
- 标题和职责没有跑层
- 阅读顺序没有断
- 旧内容已归档，不和 active 主线并列冲突
- 文档里的页面入口和当前原型入口一致

### 第三周：建立反馈机制

这周只做“让 AI 能自己发现问题”。

固定反馈分成两层：

#### 机器检查

- 跑 `npm run doc:check`
- 只检查 active 主线，不把 `history/` 旧稿当成阻塞项
- 检查核心目录、核心文档和本地相对链接是否有效
- 检查标准文档里关键入口是否还在

#### 人工抽查

- 读 `doc/README.md`
- 读 `doc/design/标准文档/1-需求分析.md`
- 读 `doc/design/标准文档/2-概要设计.md`
- 读 `doc/design/标准文档/3-产品原型设计.md`
- 打开 `http://127.0.0.1:3000/prototype`
- 再打开 `http://127.0.0.1:3000/validation`

如果机器检查过了，但人工仍然看不懂，说明问题不在“缺文件”，而在“主入口写得不清楚”。

## 5. main 当前缺什么

如果以 `harness engineering` 的角度看当前 `main`，它至少缺 5 件事：

1. 缺 `4-详细设计.md`、`5-开发.md`、`6-测试.md`、`7-迭代.md` 这四个阶段占位，软件工程主线不完整
2. 缺 `doc/design/概要细节/README.md` 和 `doc/design/原型细节/README.md` 这类子区入口，AI 不容易快速知道补充材料先读哪份
3. 缺一份专门面向 `Markdown` 治理的工作台文档，当前只有泛化的 command 说明
4. 缺一条可执行的文档自检命令，反馈机制还停留在口头要求
5. `AGENTS.md` 仍然承担了较多流程说明，但 `doc/` 里的“文档治理专用入口”还不够明确

所以当前不该继续扩目录，而该先补：

- 一个清楚的 `Markdown` 工作台入口
- 一个清楚的完成定义
- 一个清楚的自检命令

## 6. 当前起点怎么定

当前起点固定为：

`先把 doc 目录变成 AI 可稳定巡检的工作台，再谈更大的 harness`

具体顺序固定为：

1. 固定入口
2. 固定验收
3. 固定自检

不从这里开始的话，后面增加再多文档，也只是把知识库堆得更厚，不会让 AI 更稳定。

## 7. 现在可以直接让 AI 做什么

你现在可以直接这样下命令：

- `检查当前 doc 工作台缺什么`
- `把这轮结论同步到标准文档和 README 入口`
- `把重复 markdown 归档到 history`
- `给我跑一次 doc:check`
- `只整理 1-3 阶段 active 文档，不动历史稿`

## 8. 本轮建议

如果只做一件事，就先持续维护这条最小闭环：

`AGENTS.md -> doc/README.md -> design/标准文档/1-7 -> /prototype -> /validation -> doc:check`

这条链一旦稳定，你的 `harness engineering` 才开始真正成立。
