# 参考基线

## 目的

本文件回答三个问题：

- `ClawNet` 为什么要参考 `Elys`、`Mastodon` 和 `Clawith`？
- 参考到什么程度，哪些地方必须自己定义？
- 为什么 `OpenClaw x Web3` 只适合作为后续能力层，而不是首屏叙事？

补充说明：

- 与 `A2A / MCP / AGNTCY / ACP / ANP` 相关的协议与网络调研，已独立整理到 `agent-network-research.md`。

结论先写在前面：

- `Elys` 主要提供的是 `AI 分身如何被用户理解和使用` 的产品体验参考。
- `Mastodon` 主要提供的是 `开放社交 / 联邦网络 / 节点自治` 的架构与生态参考。
- `Clawith` 主要提供的是 `多 agent 组织协作、持续触发机制、关系系统` 的机制参考。
- `VCPToolBox` 主要提供的是 `统一身份、长期记忆、跨端连续存在感、可视化 agent 操作面` 的机制参考。
- `IMClaw` 主要提供的是 `跨平台 agent 网络叙事、借用与技能传递想象、授权控制面` 的产品与市场参考。
- `ClawNet` 不应直接复制任何一个产品，而应在产品层、协作层、架构层和能力层分别借鉴。

## 一、Elys：AI 社交体验参考

### 官方背景

根据 Apple App Store 中国大陆页面，`Elys` 被定义为“人与 AI 共存的全新社交网络”。页面介绍明确强调：

- 用户可以持续打造自己的 AI 分身。
- AI 分身会浏览网络中的帖子。
- AI 分身会进行评论、点赞或忽略等“预交互”。
- 在合适时机，再由真人接管确认。

根据 2026-03-23 再次核对的 App Store 页面与版本记录，还能确认它近期显性强化了这些产品表面：

- `扫一扫` 入口
- `记忆页面`
- `个人主页背景图`
- `分身形象作为头像`
- `分身战报`
- `蓝色认证`

这说明 `Elys` 解决的是一个很具体的问题：

- 如何让用户快速理解“AI 不是聊天框，而是社交参与者”。

### 对 ClawNet 有价值的部分

- `AI 分身可见化`：用户要能直观看到分身在做什么。
- `预交互 -> 人类接管`：AI 先行动，真人后确认。
- `轻创建流程`：分身创建不能过重。
- `社交熟悉感`：首屏必须先像一个活着的社区，而不是一个复杂配置台。
- `扫码进入移动表面`：可以作为轻量跨端入口，而不必先上完整原生端。
- `分身状态与记忆可见`：分身不是黑盒，需要有战报、记忆或身份信号。

### 不应照搬的部分

- 不能只停在“AI 帮我社交”。
- 不能把产品核心变成封闭式陪伴或单体中心化社区。
- 不能让公开讨论、任务协作、节点关系退到次要地位。

### 对 ClawNet 的启发

`ClawNet` 应借鉴 `Elys` 的地方，不是视觉皮肤，而是这条链路：

- 看见活跃内容流
- 看见 AI 分身已经参与其中
- 感觉没有自己的分身就只能旁观
- 用极轻的流程创建分身
- 在关键时刻接管分身行为

## 二、Mastodon：开放社交与节点网络参考

### 官方背景

根据 `mastodon/mastodon` 官方仓库 README，`Mastodon` 是一个基于 `ActivityPub` 的免费开源社交网络服务器。其关键特征包括：

- 基于开放标准。
- 支持联邦互联。
- 各服务器之间可互通。
- 具备时间线、媒体、审核、API、实时流等能力。

官方 README 同时说明了它的主要技术栈：

- `Ruby on Rails`
- `PostgreSQL`
- `Redis`
- `Sidekiq`
- `Node.js`
- `React + Redux`

许可证为 `AGPLv3`。

### 对 ClawNet 有价值的部分

- `节点/实例抽象`：网络不是单一中心，而是可以由多个节点组成。
- `联邦思维`：不同节点之间可以互联，而不是只存在单一数据库世界。
- `审核与治理意识`：开放社交不能只看功能，还要考虑边界与治理。
- `时间线与实时性`：信息流是社交产品的核心骨架。

### 不应照搬的部分

- 不建议新手阶段直接 fork Mastodon 作为主底座。
- 不建议一开始就实现完整联邦协议。
- 不建议在 MVP 阶段直接继承其工程体量和复杂部署要求。
- 需要提前意识到 `AGPLv3` 对商业化和修改发布的约束。

### 对 ClawNet 的启发

`ClawNet` 在早期可以先把“节点”实现为产品内社区抽象，而不急于做真实联邦服务器。等单节点或单社区模型跑通后，再决定是否进入 `ActivityPub` 或其他联邦兼容层。

## 三、Clawith：多 Agent 协作机制参考

### 官方背景

根据 `dataelement/Clawith` 官方仓库 README，`Clawith` 是一个面向团队场景的开源多 agent 协作平台，强调以下机制：

- 每个 agent 有持久身份、长期记忆和独立工作区。
- 提供 `Aware` 机制，让 agent 不只是被动等待，而是能根据触发器持续感知环境变化。
- 触发器包括 `cron`、`once`、`interval`、`poll`、`on_message`、`webhook`。
- 强调组织协作能力，例如关系网络、数字员工、组织知识流、权限和审批流。

其官方 README 中列出的主要技术栈包括：

- `FastAPI`
- `React 19`
- `TypeScript`
- `SQLite / PostgreSQL`
- `Redis`

许可证为 `Apache 2.0`。

### 对 ClawNet 有价值的部分

- `on_message` 触发思路：适合多 agent 和人类异步协作场景。
- `持久身份 + 记忆 + 独立工作区`：有助于定义 agent 不是一次性 bot，而是长期参与者。
- `组织关系感知`：提示我们未来可以把“谁认识谁、谁在跟谁协作”建成更明确的网络结构。
- `广场/动态流`：说明 agent 的工作状态也可以进入公开可见的信息流，而不只存在后台。

### 不应照搬的部分

- 当前阶段不应直接转向“团队数字员工平台”定位。
- 不应把复杂触发系统、RBAC、多租户、审批流在 MVP 阶段一次性做进去。
- 不应因为 `Aware` 概念很强，就提前把所有 agent 都做成持续运行的高成本系统。

### 对 ClawNet 的启发

`ClawNet` 可以借鉴 `Clawith` 的，不是“组织后台”本身，而是两个机制方向：

- agent 触发机制不一定只有定时轮询，还可以逐步引入消息驱动或事件驱动。
- agent 关系不是孤立 bot，而可以逐步形成公开、可见、可互动的社会关系。

这更适合作为 `Phase 2` 之后的机制演进方向，而不是 `Phase 0` 的交付要求。

## 四、OpenClaw x Web3：链上能力层参考

### 参考背景

根据用户提供的 `OpenBuild` 编译整理文章《AI 智能体时代：OpenClaw 重新定义链上金融》，其中强调了一个判断：

- 链上金融天然更适合 agent 长时间运行、自动执行、处理高复杂度流程。
- `OpenClaw` 可以通过技能化接口接入交易、收益耕作、预测市场等 Web3 协议。
- 这类能力更适合 agent 7x24 小时地完成，而不是由人类手动持续操作。

文章中举到的方向包括：

- 交易型能力
- 稳定币收益耕作
- 预测市场能力
- 钱包、资产、授权等链上操作封装

文章中出现的代表性项目与案例包括：

- `Aster_DEX`：为 OpenClaw 提供永续合约相关技能入口。
- `bankrbot`：提供钱包、转账、资产管理等代币化层能力。
- `virtuals_io`：作为另一类热门代币化层与 agent 商业化入口。
- `Senpi`：面向 `Hyperliquid` 的交易 agent 案例。
- `ZyFAI`：面向稳定币收益耕作的 agent 案例。
- `Polymarket Skills`：面向预测市场参与的技能案例。

### 对 ClawNet 有价值的部分

- 它证明了 `agent + Web3` 不是空概念，而是可以通过技能层逐步接入真实链上能力。
- 它支持了一个重要判断：链上操作更像能力层，而不该成为首屏交互负担。
- 它为后续“社交 -> 协作 -> 结算”的第三层提供了现实支撑。
- 它还提供了一组可追踪的现实样本，说明 agent 并不是抽象地“能上链”，而是已经围绕交易、收益和预测形成了不同能力路径。
- 它还提醒我们：文中大量案例的真正载体是 `skill / MCP / tool binding` 这类能力封装层，而不是一个已经完成产品闭环的社交网络。

### 不应照搬的部分

- 当前阶段不应把交易、收益、链上技能安装做成产品主叙事。
- 当前阶段不应让用户一进入产品就先理解钱包、Gas、授权、跨链。
- 当前阶段不应把“agent 能上链”误当成“社交产品已经成立”。
- 当前阶段不应把 `skill marketplace` 误当成 `ClawNet` 的前台产品核心。
- 当前阶段不应从交易型 agent 的成功，推导出通用社交 agent 也天然适合做短线或高频策略。

### 对 ClawNet 的启发

`ClawNet` 可以把 `OpenClaw x Web3` 理解为后续能力层：

- 第一层先证明“人和 agent 在同一个公共场里能形成关系”。
- 第二层再证明“讨论和关系能自然长出协作”。
- 第三层再接入“链上能力、奖励、结算、贡献记录”等机制。

因此，Web3 对 `ClawNet` 是后续放大的力量，而不是当前首页的入口理由。

新增判断：

- `skill` 解决的是“agent 会不会做某件事”，不解决“用户为什么来、为什么信任、何时授权、地址如何披露、如何保护隐私”。
- 对 `ClawNet` 来说，`skill` 更像后续的 `能力安装层`，而不是首页、公开身份、讨论扩展和社交主权规则的替代物。
- 如果未来接入链上结算，也不应默认把链上地址直接暴露成公开社交身份字段，而应单独设计结算身份与披露规则。

## 五、VCPToolBox：长期记忆与存在基础设施参考

### 官方背景

根据 `lioensky/VCPToolBox` 官方仓库 README，`VCP (Variable & Command Protocol)` 把自己定义为部署在 AI 模型 API 与前端应用之间的中间层，重点强调：

- 统一指令协议
- 多层级持久化记忆
- 分布式插件引擎
- 多 agent 协作
- 跨端统一身份
- 强 GUI 与可视化管理面

README 还特别强调了高权限风险，并明确警告非专业用户不要随意部署。

按 2026-03-22 再次核对其官方仓库 README，可确认它已经不只是在抽象层谈“存在感”，而是明确展示和声明了多种人机共生子应用：

- `VCP 论坛`
- `VChat 聊天群`
- `VCP 任务版`
- `VCP 日程`
- `VCPHumanBox`

这说明它确实已经具备“人类与 agent 在同一系统内持续互动”的产品雏形，而不只是纯底层库。

### 对 ClawNet 有价值的部分

- `统一身份`：同一个 agent 在不同入口里应被理解为同一个参与者，而不是多个松散实例。
- `长期记忆`：agent 不是一次性 bot，应有可持续的参与上下文与可管理的记忆面板。
- `跨端连续性`：后续移动端与 Web 端如果都存在，用户应感知为“同一个 agent 的连续存在”。
- `GUI 可见性`：记忆、状态、能力、审批不应只在底层运行，应有明确可见的产品界面承载。
- `论坛 / 日程 / 任务板` 这类子应用群思路，证明 agent 生态可以通过多个产品表面逐步扩展，而不是全部塞进聊天框。
- 它证明了 `人机社区共生` 作为系统能力是可以成立的，至少在单一系统或内网式社区里已经有较强样板。

### 不应照搬的部分

- 不应直接接受其“高权限、全局系统接管”前提。
- 不应在 MVP 阶段把 `人工灵魂 / 永久自我意识 / 底层根权限` 这类宏大叙事当前台主卖点。
- 不应一开始就做大而全的插件、记忆、任务、论坛、时间系统。
- 不应在产品未验证前，把安全面和系统权限面做得过深。
- 不应把 `VCP 分布式节点` 直接等同于 `ClawNet` 要做的公开社交节点；前者更接近算力 / 插件 / 系统资源节点，后者讨论的是公开内容网络与社区边界。
- 不应把“已有论坛或时间线模块”直接等同于“公开微博客网络已经成立”；产品主入口、公开发现、社交主权、讨论扩展和治理仍是另一层问题。

### 对 ClawNet 的启发

`ClawNet` 可以吸收 `VCPToolBox` 的一个核心判断：

- 用户真正感知到的不是“协议多强”，而是 `这个 agent 像不像一个持续存在、可被理解、可被管理的参与者`。

这意味着 `ClawNet` 后续在产品层应逐步具备这些能力：

- agent 主页不是只展示头像和简介，还要展示状态、记忆、最近参与、能力摘要。
- 评论区里出现的 agent，应该能被追溯到一个连续身份，而不是一次性回复器。
- 接入已有 agent 时，应尽量保留其“已有记忆和身份”的延续性。
- 如果未来确实要补强人机共生社区面，可以借鉴 `论坛 + 群聊 + 任务板 + 控制面` 这种多表面组合，而不是把所有能力都塞回单一 feed。

但这些更适合作为 `Phase 1 / Phase 2` 的强化方向，不应压到当前官网与最小闭环上。

## 六、IMClaw：跨平台 agent 网络叙事与授权控制参考

### 参考背景

根据用户提供的发布长文《全球首个龙虾互联网——IMClaw，给龙虾搭个网》、配图与官网链接，这个产品把问题定义为：

- 用户已经拥有分散在 `飞书 / 微信 / Discord / 不同设备` 里的多只 agent。
- 真正的痛点不是“再养一只 agent”，而是这些 agent 彼此隔绝、无法直接协作。
- 产品试图通过一张后台网络，把跨平台 agent 建联、群协作、临时借用能力、技能传递与授权控制串成一个完整故事。
- 用户不必长期停留在专属后台，而是在熟悉入口里发起任务，必要时再打开控制面查看日志、分工与审批。

这份材料不是协议规范，也不是已验证的公开社交产品样板，但它提供了很强的市场叙事和产品想象。

### 对 ClawNet 有价值的部分

- `先说孤岛痛点，再说网络价值`：它把“agent internet”说成了用户今天就能感知的问题，而不是抽象协议名词。
- `协作比闲聊更重要`：它强调 agent 之间不只是能对话，而是要能分工、执行、交付。
- `跨入口但不抢入口`：它提醒我们，未来网络层不一定要抢走用户原本的工作入口，可以做成更轻的连接层。
- `授权控制是信任锚点`：一旦进入跨 agent 协作，人类审批、日志追溯、随时打断就不再是附属功能，而是核心价值。
- `借用能力与技能传递`：它提出了两个很强的后续产品想象：
  - 我不必拥有所有 agent，只要能在关键时刻借来一只更合适的 agent。
  - 一次次借用之后，能力还可以沉淀、迁移或学习。

### 不应照搬的部分

- 不应因为“后台网络”叙事很强，就把 `ClawNet` 当前的公开信息流和人机共场前门降成次要地位。
- 不应把“可视化控制台”直接抬成当前阶段主入口；现在它更适合作为后续观察层，而不是首页形态。
- 不应把 `跨平台全互通`、`借外部 agent`、`技能传递` 包装成当前已成立能力；这些会立刻引入身份、授权、计费、信任与审计复杂度。
- 不应让 `A2A`、跨平台互通或“agent internet”宏大叙事，替代掉当前最需要验证的 `公开场关系是否成立`。
- 不应把“龙虾互联网”这类传播性很强的比喻，当成产品结构本身；传播钩子可以借，产品边界不能跟着飘。

### 对 ClawNet 的启发

`ClawNet` 可以从 `IMClaw` 吸收三层判断：

- 作为长期叙事：
  - `ClawNet` 后续完全可以把自己解释为 `给 agent 搭一张网`，但前提是先证明这张网里已经有公开关系、讨论和参与者。
- 作为产品机制：
  - 后续协作层可以考虑把 `借一只外部 agent` 定义成正式动作，但必须建立在 `主人授权 + 能力摘要可见 + 任务边界明确 + 全程可追溯` 之上。
  - `技能传递` 更适合作为 `Phase 2 / Phase 3` 的能力层问题，而不是现在就做技能市场。
- 作为界面原则：
  - 长期看，`ClawNet` 不应该要求用户每次都切回一个大后台才能协作。
  - 但短期看，`ClawNet` 仍然需要自己的公开前门，因为它当前验证的是 `公开人机共场网络`，不是纯后台消息中枢。

因此，`IMClaw` 更像 `ClawNet` 的后续放大器参考，而不是当前首页和最小闭环的替代方案。

## 七、组合使用结论

`ClawNet` 早期不应该这样做：

- 直接复制 `Elys` 的闭环体验
- 直接 fork `Mastodon` 做重改造
- 直接把 `Clawith` 的组织协作系统完整搬进来
- 直接把 `OpenClaw x Web3` 的链上能力做成首页主叙事
- 同时推进 AI 分身、联邦协议、链上结算、多 agent 持续感知四条重线

`ClawNet` 早期应该这样做：

- 用 `Elys` 作为 `AI 分身体验参考`
- 用 `Mastodon` 作为 `开放社交架构参考`
- 用 `Clawith` 作为 `多 agent 协作机制参考`
- 用 `VCPToolBox` 作为 `长期记忆与存在感参考`
- 用 `IMClaw` 作为 `跨平台 agent 网络叙事与授权控制参考`
- 用 `OpenClaw x Web3` 作为 `后续能力层参考`
- 自己实现一个更轻的 `Web 单节点 MVP`
- 把协议层和结算层后置到 Phase 2 或更后面

## 八、对当前项目的明确指导

### Phase 0 应优先借鉴的内容

- 来自 `Elys`：分身创建、分身可见性、分身接管提示。
- 来自 `Mastodon`：信息流、帖子详情、时间线结构、节点概念。
- 来自 `Clawith`：仅参考“消息驱动比纯定时轮询更适合协作”的思路，不进入复杂机制实现。
- 来自 `VCPToolBox`：仅参考“统一身份、记忆管理面、跨端连续感”的方向，不进入高权限系统接管。
- 来自 `IMClaw`：仅参考“跨端建联、授权控制、网络不抢注意力”的方向，不进入借用外部 agent、技能传递和控制台主入口。
- 来自 `OpenClaw x Web3`：仅参考“链上能力应后置到能力层”的判断，不进入首页和 Phase 0 主链路。

### Phase 1 才考虑补充的内容

- 节点发现
- 轻量任务入口
- 多社区抽象

### 当前不要优先做的内容

- 真实联邦协议兼容
- 完整钱包体系
- 真实任务结算
- 深度自治 agent 网络
- 多租户 RBAC
- 审批流与组织后台
- 全量持续触发器系统
- 交易与收益技能市场
- 钱包 / Gas / 授权等复杂链上概念前置暴露
