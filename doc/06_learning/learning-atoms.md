# Learning Atoms

## 用途

这份文件不是传统学习笔记。

它只记录通过当前项目顺手学到、以后还能复用的 `知识最小单元`。

## 学习原子格式

每个 `Learning Atom` 固定包含：

- `场景`
- `概念`
- `为什么对当前项目重要`
- `怎么判断自己学会了`
- `关联文件或下一步`

## 当前学习原子

### LA-001：公开内容网络不等于 agent 协议

- `场景`：讨论 `ClawNet` 是否该先做节点、A2A、MCP、联邦时
- `概念`：公开内容网络、agent 协作协议、工具协议、目录发现层是 4 个不同层级
- `为什么对当前项目重要`：如果不分层，MVP 会被协议复杂度拖死
- `怎么判断自己学会了`：能用自己的话解释为什么 `/network` 页面现在只做方向证明，而不做真实联邦
- `关联文件或下一步`：`agent-network-research.md`、`F005-network-layer-demo.md`

### LA-002：Mock 先于集成

- `场景`：准备做 `connect + QR + app` 演示链路时
- `概念`：先验证产品表面和演示闭环，再决定是否接真实 API
- `为什么对当前项目重要`：当前最缺的是可讲清的 MVP，不是底层联调
- `怎么判断自己学会了`：能清楚区分“产品化接入演示”和“真实协议接入”
- `关联文件或下一步`：`decision-log.md`、`F010-agent-connect.md`

### LA-003：扫码进入移动 Web 表面，不等于原生移动端优先

- `场景`：MVP 要参考 `Elys` 的 `扫一扫` 和移动表面
- `概念`：同一个产品可以先用二维码把用户带入移动 Web 表面，而不承诺做完整原生端
- `为什么对当前项目重要`：这能显著提升演示说服力，又不把范围拉爆
- `怎么判断自己学会了`：能解释为什么当前只做 `/app`，而不立刻做 iOS / Android
- `关联文件或下一步`：`product-direction.md`、`information-architecture.md`

### LA-004：规格先于开发

- `场景`：准备开始写页面前
- `概念`：`backlog` 定范围，`feature spec` 定验收，`todo-list` 定本周执行
- `为什么对当前项目重要`：否则会一边写代码一边改目标
- `怎么判断自己学会了`：能在动手前明确说出“我现在是按哪个 spec 在做”
- `关联文件或下一步`：`backlog.md`、`feature-specs/`

### LA-005：产品拆解先拆演示闭环，再拆系统实现

- `场景`：决定这周先从 `T001 /connect` 开始，而不是先补协议和 API
- `概念`：先拆“用户能走通的一条演示链路”，再拆“系统底层怎么真正跑起来”
- `为什么对当前项目重要`：当前最容易失控的地方，就是一上来把去中心化连接当成底层工程题
- `怎么判断自己学会了`：能把 `connect -> QR -> app -> network` 当作产品闭环讲清，而不先掉进协议细节
- `关联文件或下一步`：`todo-list.md`、`F010-agent-connect.md`

### LA-006：连接感来自连续身份，不来自真实通信先打通

- `场景`：把本周 MVP 提高到“命令行接入后，扫码进入网页 app，并完成加入基站”
- `概念`：对用户来说，强连接感首先来自身份连续、状态连续、动作连续，而不是先来自底层通信已经连通
- `为什么对当前项目重要`：这样可以在 mock 数据阶段先把 `npm 命令 -> 二维码 -> 移动信息流 -> 加入基站` 做得像真实产品
- `怎么判断自己学会了`：能解释为什么这周先要求“接入身份可见、加入状态可变”，而不是先做协议联调
- `关联文件或下一步`：`product-direction.md`、`F005-network-layer-demo.md`

### LA-007：先验证本地可执行命令，再做公网分发

- `场景`：讨论 `connect` 是否现在就要做公网 npm 包和后端接口
- `概念`：分发形态和演示闭环是两个问题，先验证“任意本地 Node.js 目录里能跑命令并弹出二维码”，再决定是否进入公网发布
- `为什么对当前项目重要`：这样能把本周焦点锁在 `CLI -> QR -> 手机 Web App`，避免过早陷入包发布和后端规划
- `怎么判断自己学会了`：能解释为什么本周先做 local CLI demo mode，而不是先做 npm 公网包
- `关联文件或下一步`：`F010-agent-connect.md`、`todo-list.md`

### LA-008：任务编号不是优先级，检测通过才算完成

- `场景`：发现本周 TO DO 仍然按 `T001 / T002 / T003` 排，容易误导执行顺序
- `概念`：任务 `ID` 只负责稳定引用；真正推进顺序要按 `P0 / P1 / P2` 管理，且完成必须有检测记录
- `为什么对当前项目重要`：这样才能把精力持续压在当前关键路径，而不是被历史流水号牵着走
- `怎么判断自己学会了`：能说清当前先做哪些 P0 项，以及为什么某任务虽然“看起来做了”但还不能算 `done`
- `关联文件或下一步`：`todo-list.md`、`todo-verification-log.md`

### LA-009：Demo 构建不能依赖外网资源

- `场景`：本机 `npm run build` 因为在线 Google 字体拉取失败而中断
- `概念`：演示版产品的构建链应优先选择本地可复跑的资源，不把外网请求当成默认前提
- `为什么对当前项目重要`：否则看起来只是“字体问题”，实际会直接阻断接入链路演示和交付节奏
- `怎么判断自己学会了`：能解释为什么 Phase 0 先用本地 fallback 字体栈，而不是继续追求在线字体一致性
- `关联文件或下一步`：`src/app/layout.tsx`、`src/app/globals.css`、`todo-verification-log.md`

### LA-010：MVP 验证先跑最短用户闭环，不先补整套系统

- `场景`：决定继续从 `connect demo` 往前推，直到用户能亲手试玩 MVP
- `概念`：真正可验证的 MVP，不是“页面数量变多”，而是用户能按固定顺序走完一条价值链
- `为什么对当前项目重要`：这样才能判断公开信息流、帖子详情、agent 创建/接入是否真的连成产品，而不是各自成立
- `怎么判断自己学会了`：能用一句话说清当前试玩顺序是 `/ -> /posts/:id -> /agents/new 或 /connect`
- `关联文件或下一步`：`core-user-flows.md`、`todo-list.md`、`decision-log.md`

### LA-011：真实手机扫码不等于 `localhost` 可用

- `场景`：准备把本地 CLI 输出的二维码交给真实手机扫码试玩
- `概念`：CLI 里写 `http://localhost:3000` 只适合当前电脑本机预览；真实手机需要 `0.0.0.0` 监听加局域网地址
- `为什么对当前项目重要`：如果不区分这两种运行方式，MVP 看起来已经完成，但用户一拿手机就会误判为“二维码不可用”
- `怎么判断自己学会了`：能解释为什么要用 `npm run start:lan`，并把 `CLAWNET_HOST` 改成 `http://<局域网IP>:3000`
- `关联文件或下一步`：`package.json`、`local-cli-demo-execution.md`、`mvp-validation-runbook.md`

## 使用规则

- 只有能复用的知识才记录。
- 只有能回链到当前项目的知识才记录。
- 一条原子尽量控制在 5 行内，不写成长文。
