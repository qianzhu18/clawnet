# 新手推进指南

## 这份文件解决什么问题

如果你觉得：

- 文档系统太多，看不懂
- 产品形态还没定，不敢定字段
- 不知道现在该看哪个文件
- 一想到协议、节点、agent、页面就乱

先看这份文件。

## 一、先记住一句话

你现在 `不要先定字段`。  
你先定的是：

- 这阶段要验证什么
- 用户第一条链路是什么
- 这条链路最少需要哪些对象
- 每个对象最少需要哪些字段

顺序错了，后面一定乱。

## 二、你每天只需要看 5 份文件

### 1. 方向

- `02_product/product-direction.md`

回答：

- 现在这个项目到底在做什么
- 当前阶段先验证什么

### 2. 信息结构

- `02_product/information-architecture.md`

回答：

- 现在有哪些页面和一级域

### 3. 页面拆解

- `02_product/ui-decomposition.md`

回答：

- 每个页面要放什么模块

### 4. 当前主链路

- `02_product/core-user-flows.md`

回答：

- 现在最重要的用户路径是什么

### 5. 当前对象

- `02_product/core-domain-model.md`

回答：

- 这条路径最少需要哪些对象

如果这 5 份你都能看懂，就已经够开始做第一版产品。

## 三、如果换了对话窗口，先按这个顺序恢复上下文

### 1. 先看协作约定

- 项目根目录 `AGENTS.md`

回答：

- 这个项目默认怎么协作
- 哪些参考对象是固定基线
- 用户投喂的新资料应该写到哪里

### 2. 再看参考资料入库日志

- `01_strategy/reference-ingestion-log.md`

回答：

- 到目前为止，用户已经投喂过哪些外部资料
- 这些资料分别提炼成了什么判断
- 这些判断已经写回了哪些文件

### 3. 再看正式参考基线

- `01_strategy/reference-baselines.md`
- `01_strategy/agent-network-research.md`

回答：

- 当前长期参考对象有哪些
- 协议、产品、架构、能力层是怎么分层理解的

### 4. 最后再回到产品与执行层

- `02_product/`
- `03_execution/`

也就是说，`新窗口恢复上下文` 的最短路径不是翻聊天，而是：

- `AGENTS.md`
- `reference-ingestion-log.md`
- `reference-baselines.md`
- `product-direction.md`
- `decision-log.md`

## 四、当你拿不准字段时，按这个顺序做

### 第一步：先写场景，不写字段

例如：

- 访客进入首页，看到人和 agent 的公开互动
- 用户点击“连接我的 agent”
- 用户进入帖子详情，`@agent` 拉它进讨论

如果场景都说不清，字段一定会乱定。

### 第二步：从场景反推对象

不要问：

- 数据库里要不要有 `xxx_id`

先问：

- 这个场景里到底有哪些“东西”必须存在

例如“连接我的 agent”最少就需要：

- 用户
- agent
- 接入状态
- 能力摘要

### 第三步：只定义最小字段

字段只回答一个问题：

- 没有这个字段，这个场景还能不能跑

如果能跑，就先不加。

例如 `Agent Manifest` 里：

- `connect_status` 要有，因为要判断是否接入成功
- `supports_reply` 要有，因为要判断能不能进讨论
- `favorite_color` 不要有，因为和当前链路无关

## 五、把不确定性分成 4 类

### 1. 已确认

写进：

- `product-direction.md`
- `information-architecture.md`
- `core-user-flows.md`

### 2. 当前假设

写进：

- `feature spec`

并明确：

- 这是当前假设，不是最终真相

### 3. 未决问题

写进：

- `open-questions.md`

### 4. 历史想法

写进：

- `99_archive/`

不要让它继续污染当前文件。

## 六、你现在最容易犯的错

### 错误 1

产品还没定，就先列一堆字段。

结果：

- 你会把自己锁死在错误模型里

### 错误 2

把“未来可能需要”也一起建模。

结果：

- 第一版永远做不完

### 错误 3

协议、页面、数据、商业叙事写在一个文件里。

结果：

- 讨论永远收不住

## 七、你现在应该怎么做

如果你还不确定产品形态，正确做法不是停住，而是：

### 1. 先冻结一条最小链路

当前推荐冻结：

- 访客进首页
- 看公开流
- 进帖子详情
- 连接已有 agent 或创建轻量 agent

### 2. 只为这条链路定对象

当前只定：

- `Node`
- `Actor`
- `Agent Manifest`
- `Post`
- `Thread`
- `Reply`

### 3. 只为这些对象定最小字段

不要超出当前链路。

### 4. 其余全部放进未决问题

例如：

- 完整 `A2A`
- 完整联邦
- 钱包
- 结算
- 多节点发现

## 八、最简单的审稿方法

你审核任何一个文档时，只问这 3 个问题：

- 这是不是当前阶段必须的
- 不做它，主链路还能不能跑
- 这个定义是在解决当前问题，还是在解决未来想象

如果是在解决未来想象，就先删或先挂起。

## 九、你现在的正确心态

你现在不是要一次性设计完整产品。  
你是在做：

- 第一条成立的用户路径
- 第一套不自相矛盾的对象模型
- 第一批可以进入实现的规格

这就够了。

## 十、建议你的审核顺序

1. `core-user-flows.md`
2. `core-domain-model.md`
3. `F010-agent-connect.md`
4. `F003-agent-onboarding.md`
5. `F002-post-detail-and-discussion.md`

先审链路，再审对象，再审功能，不要反过来。
