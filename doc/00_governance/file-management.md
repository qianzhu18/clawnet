# 文件管理规则

## 目标

任何时候，这个项目都应该能快速回答三个问题：

- 当前正式生效的文档是哪一份？
- 最近发生了什么变化？
- 还有哪些问题没有定？

如果一个目录回答不了这三个问题，说明结构还不够好。

## MVP 演示阶段心法

当目标是 `先做一个能对外讲清楚的 MVP Demo` 时，不要试图同时管理全部文件。

只管理两层：

### 第一层：当前真相文件

这是你每天真正会看的文件，默认只看这些：

- `02_product/product-direction.md`
- `02_product/ui-decomposition.md`
- `02_product/open-questions.md`
- `03_execution/backlog.md`
- 当前阶段对应的 `feature spec`

这 5 类文件决定：

- 你现在到底在演示什么
- 页面要长什么样
- 还有哪些问题没定
- 本周只做哪几个功能
- 每个功能具体做到什么程度

如果本周已经进入视觉稿或设计稿阶段，再额外看：

- `02_product/mobile-web-design-brief.md`

它只负责给设计工具或设计师提供稳定的页面需求，不替代产品定义和功能规格。
如果它被用作外部交付件，就必须单独可读，不能依赖其他仓库文件才能理解。

### 第二层：参考与研究文件

这些文件重要，但不参与日常推进：

- `01_strategy/reference-baselines.md`
- `01_strategy/reference-ingestion-log.md`
- `01_strategy/agent-network-research.md`
- `99_archive/`

只有在以下情况才回头看：

- 方向变了
- 外部有人质疑你的参考来源
- 需要补一个长期判断

### 演示期 5 条硬规则

- 不为“也许以后会用”新建文件。
- 每轮只允许改 `1-3` 个当前真相文件。
- 一个新想法只能先进入 `backlog`、`open questions` 或 `decision log` 三者之一。
- `99_archive/`、模板、深度研究默认不看，除非当前问题明确需要。
- 如果一个文件不会影响本周 Demo，就先别动它。

一句话心法：

> 演示期不是“把知识管理完整”，而是“把当前真相压缩到最少”。

## 企业版文档管理规则

当项目开始讨论 `企业版 / 私有化 / 安全增强版` 时，不要立刻复制一整套平行文档。

当前默认采用：

- `主线文档`：继续描述所有版本共享的产品骨架与核心价值。
- `企业版差异层`：只记录企业版新增的 `安全、权限、审计、部署、合规` 差异。
- `决策日志`：记录已经冻结的企业边界，不让口头判断漂移。
- `未决问题池`：记录尚未拍板的安全优先级、部署模式和权限边界。

### 什么内容写进企业版差异层

- 私有化部署与托管部署的边界
- 身份体系、组织结构、角色权限
- 审批流、接管流、审计留痕
- 数据隔离、模型调用边界、外发限制
- 企业管理员控制台需要看见什么

### 什么内容不要直接写进仓库

- 真实客户名称与敏感业务细节
- 真实密钥、令牌、证书、账号
- 可直接复用的生产拓扑细节
- 未脱敏的数据样本

如果必须描述这些对象，只允许写：

- 脱敏示例
- 占位字段
- 抽象结构图

### 企业版是否要复杂版本控制

当前结论：`不需要`。

先用这 4 层控制复杂度：

1. `Git` 负责历史追踪。
2. `front matter` 里的 `status` 和 `last_updated` 负责当前状态。
3. `decision-log.md` 负责记录关键取舍。
4. `99_archive/` 负责冻结历史阶段稿。

不要做的事：

- 不在文件名里叠加 `v2-final-enterprise-new`
- 不为企业版复制整套 `BP / PRD / backlog`
- 不同时维护多个“当前真相”文件

只有在以下条件同时出现时，才考虑拆出企业版独立文档线：

- 企业版已有独立路线图或独立交付节奏
- 与主线相比，功能或安全要求差异已明显超过一小半
- 需要面向外部客户交付一套单独可读的材料

## 执行周新增的两类文件

从开始真正动手的那一周起，允许多看两类文件，但职责必须分清：

### `05_todo/`

用途：

- 只记录 `这周实际要完成什么`
- 只记录 `怎么验收`
- 只记录 `当前状态`

它不是：

- 新的 backlog
- 新的 spec
- 新的长文说明

### `06_learning/`

用途：

- 只记录通过当前项目顺手学到的关键知识
- 帮用户建立指挥 AI、理解复杂系统所需的最小知识骨架

它不是：

- 传统学习笔记仓库
- 教科书摘抄区
- 大而全的知识百科

### 学习记录最小化单元

所有学习记录统一压缩成一个 `Learning Atom`，至少回答 5 个问题：

- 这是在什么场景下碰到的？
- 这是什么概念？
- 它为什么对 `ClawNet` 当前推进重要？
- 怎样算“我已经学会了”？
- 它应该回链到哪个功能、文件或下一步动作？

### `05_todo/`

用途：

- 管本周执行，而不是长期范围
- 按 `P0 / P1 / P2` 排布当前工作
- 把“做完了没有”与“怎么证明做完了”拆开管理

最小文件组合：

- `todo-list.md`
- `todo-verification-log.md`

使用规则：

- `ID` 只做稳定引用，不代表执行顺序。
- 实际推进顺序按 `P0 -> P1 -> P2`。
- 一个任务只有在 `todo-list.md` 的完成定义成立，且 `todo-verification-log.md` 留下通过记录后，才算真正完成。

## 命名规则

- 当前文件名统一使用英文 kebab-case。
- 当前正式目录统一使用固定编号目录，不使用带空格目录名。
- 用目录表达类别，不靠文件名后缀表达状态。
- 功能规格统一使用 `F###-slug.md`。
- 历史版本进入 `99_archive/`，不要在当前目录继续叠加 `v3`、`最终版`、`新版`。

推荐命名：

- `business-plan.md`
- `product-direction.md`
- `ui-decomposition.md`
- `F001-home-feed.md`

不推荐命名：

- `BP-final-v3.md`
- `需求沉淀最终版.md`
- `new_new_plan.md`
- `TO DO/`
- `TO learn/`

## 状态规则

活动文档统一在 front matter 里写状态，限定为：

- `draft`
- `active`
- `reviewing`
- `archived`

## Front Matter 标准

所有新的受管文档都使用以下头部：

```yaml
---
title: 文档标题
status: draft
owner: founder
last_updated: YYYY-MM-DD
---
```

## 归属规则

- 战略与商业问题归 `01_strategy/`。
- 产品定义、交互、信息架构归 `02_product/`。
- 移动端视觉稿需求也归 `02_product/`，但必须单独放在 design brief 文件里，不和 `ui-decomposition` 混写。
- 如果该文档要交给外部设计 agent，内容必须自洽，不能只写索引和引用。
- 规划、任务、规格、决策归 `03_execution/`。
- 模板归 `04_templates/`。
- 历史材料归 `99_archive/`。

## 编辑规则

- 不在 `99_archive/` 中继续做当前版本编辑，除非只是修复格式或补充说明。
- 如果已有“当前真相”文件，就不要再新建另一个平行文件。
- 不允许只依据聊天记录直接进入代码实现。
- 功能进入实现前，必须先创建或更新对应的 `feature spec`。
- 文档默认写中文，除非是代码标识、命令、路径、协议名。
- 用户投喂的外部参考资料，不能只停留在聊天里；必须先进入 `01_strategy/reference-ingestion-log.md`。
- 如果外部资料形成了稳定判断，还要同步更新 `reference-baselines.md`、`decision-log.md` 或 `open-questions.md`。

## 复盘节奏

- 每周至少看一次 `backlog.md` 和 `open-questions.md`。
- 每次开始一个新功能前，先看 `product-direction.md`、`ui-decomposition.md`、对应 `feature spec`。
- 每次功能完成后，更新 `decision-log.md`、`backlog.md`，必要时更新 `roadmap.md`。

## 参考资料管理规则

新增外部材料时，按以下顺序管理：

1. `reference-ingestion-log.md`
   - 记录来源、类型、提炼判断、落点文件。
2. `reference-baselines.md`
   - 只保留已经稳定的参考价值判断。
3. `agent-network-research.md` 或其他专题研究
   - 放需要长期保留的深入研究内容。
4. `decision-log.md` / `open-questions.md`
   - 分别承接已决结论和未决问题。
