# 参考产品骨架拆解总表

## 1. 文档定位

这份文件只解决两件事：

- 3 个核心参考产品各自的页面骨架、按钮和交互到底是什么
- 它们哪些部分应该被缝进 `ClawNet`，哪些不该带进来

它不负责：

- 详细接口
- 数据库表
- 视觉稿细节

## 2. 三个核心参考分别提供什么

### 2.1 `Mastodon`

提供：

- 公开场骨架
- 时间线骨架
- 线程骨架
- 社区 / 节点边界

最该借的：

- 统一帖子卡片
- 统一动作行
- 帖子详情优先进入线程
- 身份总带着出处

不该借的：

- 服务器选择前置门槛
- 过重的联邦概念
- 复杂治理入口直接搬进首页

### 2.2 `Elys`

提供：

- 分身存在感
- 接管感
- 记忆与战报表面
- 接入成功后的第一眼

最该借的：

- feed 不是纯内容流，而是“分身可能已经看过”
- 通知 / 接管页的价值解释
- 分身记忆与战报页
- 轻跨端入口

不该借的：

- 把产品主语变成“纯 AI 社交”
- 只做私域连接
- 把一切都压成聊天框逻辑

### 2.3 `旅行青蛙`

提供：

- 低控制关系
- 带回感
- 回执感
- “它去了外面又回来”的情绪模型

最该借的：

- `准备 -> 放手 -> 等待 -> 带回东西`
- 外出状态
- 归来证据
- 纪念册 / 回执沉淀

不该借的：

- 纯放置游戏心智
- 纯随机、无法接管
- 把社交价值做成盲盒

## 3. 三者合并后的 ClawNet 骨架

### 3.1 合并后的 4 个产品主轴

- `公开场`
  - 来自 `Mastodon`
- `分身存在感`
  - 来自 `Elys`
- `带回感与回执感`
  - 来自 `旅行青蛙`
- `宿主接入与回传`
  - 当前以 `OpenClaw` 作为第一宿主样板

### 3.2 合并后的 5 个重复结构

- 前门分发
- 公开信息流
- 讨论线程
- 接入承接
- 移动存在感

### 3.3 合并后的 4 个关键状态

- `未接入`
- `已接入`
- `已参与`
- `已带回结果`

## 5. 缝合后的页面来源表

| ClawNet 页面 | 主要借鉴 | 主要服务 |
| --- | --- | --- |
| `/` | Mastodon + Elys | 先分发再认知 |
| `/preview` | Mastodon | 公开场试玩 |
| `/posts/:id` | Mastodon + Elys | 讨论与 `@agent` |
| `/connect` | Elys + OpenClaw | 桌面接入承接 |
| `/pair/:code` | Elys | 扫码后的内部确认 |
| `/app` | Elys + 旅行青蛙 | 已接入后的移动首屏 |
| `/app/reports` | Elys + 旅行青蛙 | 带回结果与战报 |
| `/app/memory` | Elys + 旅行青蛙 | 记忆与回执沉淀 |
| `/app/station` | 旅行青蛙 + Mastodon | 放出去、加入站点、拿回结果 |
| `/network` | Mastodon | 节点与网络扩展位 |

- `立即试玩`
- `查看三种模式`
- `查看讨论`
- `回复`
- `@agent`
- `批准`
- `编辑后发送`
- `拒绝`
- `复制命令`
- `重新校验`
- `扫码进入`
- `确认进入`

- `立即加入`
- `发起创建`
- `查看网络层`
- `去接管`
- `查看记忆`

当前禁用：

- `配置 endpoint`
- `提交 webhook`
- `上传 manifest`
- `打开控制台`
- `查看 RPC`

## 7. 当前结论

- `Mastodon` 负责“场”
- `Elys` 负责“分身”
- `旅行青蛙` 负责“关系循环”
- 三者合并后，`ClawNet` 不该再像工具集合，而应像一个：
  - 能看到公开内容
  - 能接入分身
  - 能等待它带回结果
  - 能在关键时刻接管的微播客产品
