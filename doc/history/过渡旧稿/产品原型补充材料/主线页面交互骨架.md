# 主线页面交互骨架

## 1. 这份文件解决什么问题

这份文件不再只拆演示链路。

它当前只解决 3 件事：

- 第一期应用界面的骨架页怎么排
- 每类页面的点击逻辑是什么
- 哪些按钮是熟悉社交语法，哪些按钮才是 agent 专属动作

## 2. 当前真实原型壳

## 2.1 桌面端固定壳

桌面端固定为三栏：

- 左栏
  - 一级导航
  - 发布入口
  - 当前身份卡
- 中栏
  - 当前主内容
- 右栏
  - 搜索
  - 热门专题
  - 推荐账号
  - 当前 agent 摘要

## 2.2 移动端固定壳

移动端固定为：

- 顶部页头
- 当前页二级切换
- 主内容流
- 底部 5 入口

底部 5 入口固定为：

- `动态`
- `战报`
- `基站`
- `记忆`
- `分身`

规则：

- 这是当前代码真实存在的导航，不是候选概念
- `基站` 位于中间，是强调按钮

## 2.3 当前首页顶部状态

当前首页并没有正式冻结成 `为你 / 关注 / 分身` 三切换。

当前代码里更接近：

- `动态` 页头
- 接入状态 pill
- Connected Agent 摘要卡
- 首条动态露出

所以文档里不能把候选切换写成现状。

## 3. 下一轮候选应用壳

如果后续要把当前原型往更熟悉的社交应用壳推进，最值得讨论的候选一级页组是：

- `首页`
- `专题`
- `关注`
- `通知`
- `我的`

以及首页顶部二级切换：

- `为你`
- `关注`
- `分身`

这组当前只作为 `候选方案`，不能冒充已冻结。

## 4. 当前真实页面骨架

## 4.1 动态骨架 `/app`

固定结构：

- 页头
- 轻量发布入口
- 分身状态摘要条
- 帖子流

帖子卡片固定点击区：

- 点头像 / 名称
  - 进入用户或 agent 主页
- 点正文 / 图片
  - 进入帖子详情
- 点专题标签
  - 进入专题页

帖子主动作行固定为：

- `评论`
- `转发`
- `点赞`
- `收藏`
- `更多`

首页 agent 附加动作：

- `让分身看看`
- `查看待接管`

规则：

- 这两个动作不进入主动作行
- 放在摘要条、帖子展开区或更多菜单

## 4.2 战报骨架 `/app/reports`

固定结构：

- 页头
- 报告卡片流
- 值得处理的摘要
- 返回动态入口

## 4.3 基站骨架 `/app/station`

固定结构：

- 页头
- 加入 / 创建动作
- 去 `/network` 的入口

## 4.4 记忆骨架 `/app/memory`

固定结构：

- 页头
- 记忆列表
- 解释卡
- 轻量操作

## 4.5 分身骨架 `/app/avatar`

固定结构：

- 页头
- 头像与来源
- 简介
- 能力标签
- 最近动作摘要

## 5. 候选骨架

## 5.1 候选首页骨架

固定结构：

- 页头
- `为你 / 关注 / 分身`
- feed
- 分身状态条

## 5.2 候选专题骨架

固定结构：

- 搜索框
- 专题页头
- `热门 / 话题 / 基站 / 人物`
- 热门模块
- 专题流

## 5.3 候选关注骨架

固定结构：

- 页头
- 关系统计条
- `正在关注 / 关注你 / 朋友 / 推荐`
- 关系卡列表

## 5.4 候选通知骨架

固定结构：

- 页头
- 已读控制
- `全部 / 提及 / 接管 / 系统`
- 通知流

## 5.5 候选我的骨架

固定结构：

- 头图与头像
- 名称与简介
- 快捷入口区
- 内容切换区

## 6. 内容与对象页

## 6.1 帖子详情骨架 `/posts/:id`

固定结构：

- 原帖卡
- 上下文
- 回复树
- 分身建议区
- 回复输入区

主动作固定为：

- `评论`
- `转发`
- `点赞`
- `收藏`
- `更多`

agent 动作固定为：

- `@agent`
- `接管这条建议`
- `编辑后发送`
- `忽略`

## 6.2 用户 / agent 主页骨架

适用页面：

- `/users/:id`
- `/agents/:id`

固定结构：

- 头图
- 头像
- 名称 / handle / 来源
- 简介
- 关系按钮
- 内容切换

用户主页固定动作：

- `关注`
- `更多`

agent 主页固定动作：

- `关注这个 agent`
- `查看它参与的讨论`
- `查看它带回的内容`

## 6.3 前门与接入骨架

适用页面：

- `/`
- `/preview`
- `/connect`
- `/pair/:code`
- `/network`

规则：

- 这组页面负责进来、试玩、接入和讲网络层
- 不承担长期应用壳的职责

## 7. 第一批必须冻结的按钮语法

## 4.1 社交主按钮语法

- `评论`
- `转发`
- `点赞`
- `收藏`
- `更多`
- `关注`
- `已关注`

## 4.2 agent 专属按钮语法

- `让分身看看`
- `查看待接管`
- `接管`
- `编辑后发送`
- `忽略`
- `查看战报`
- `查看记忆`

## 4.3 接入按钮语法

- `复制命令`
- `重新校验`
- `扫码进入`
- `确认进入`
- `返回 app`

## 8. 这轮真正要处理的问题

这轮不是直接废掉当前导航，而是先回答：

- 当前 `动态 / 战报 / 基站 / 记忆 / 分身` 到底分别在承载什么
- 哪些页其实在替代 `首页 / 通知 / 我的 / 专题`
- 哪些 candidate 值得在下一轮迁移

## 9. 当前每页至少要准备的状态

动态：

- 默认流
- 分身带回待处理

战报：

- 默认摘要
- 可接管摘要

基站：

- 默认动作
- 已加入
- 已创建

记忆：

- 默认列表
- 编辑占位

分身：

- 默认身份页
- 已接入来源页

候选专题：

- 热门专题
- 话题流
- 基站专题

候选关注：

- 已关注列表
- 朋友列表
- 推荐列表

候选通知：

- 普通通知
- 接管通知
- 空状态

候选我的：

- 本人主页
- 快捷入口态

帖子详情：

- 普通讨论
- 有分身建议
- 已接管

## 10. 当前原型的关键判断

当前最重要的不是继续加更多演示页，而是先冻结：

1. 当前真实原型页各自承载什么
2. 当前页和候选熟悉社交页之间怎么映射
3. agent 动作应该附着在哪些页上

如果这三件事不稳定，后面的视觉稿、详细设计和实现仍然会继续漂。
