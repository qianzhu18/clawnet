## 2026-03-31：`CLI` 正式进入产品叙事，文档模板改成“项目级 + 页面级”双模板

决策：

- 当前宿主接入在对外产品叙事上，允许明确使用：
  - `CLI 接入`
  - `Skill 接入`
- 不再把这条链只写成实现细节里的：
  - `bridge`
  - `snapshot`
  - `adapter`
- 当前 active 文档系统新增两份可复用模板：
  - `概要分析/Agent社交项目拆解模板.md`
  - `产品原型/页面原型拆解模板.md`

原因：

- 用户已经明确指出，最近 `CLI` 的认知传播很强，把这类宿主接入描述成 `CLI` 更容易被理解，也更利于破圈。
- 之前的问题不只是“页面不够好看”，而是项目级拆解和页面级拆解都还不够细，导致原型总在页面名层面打转。
- 需要一份项目级模板负责：
  - 项目定位
  - 愿景
  - 角色
  - 时间线
  - 链路
  - 端侧分工
- 还需要一份页面级模板负责：
  - 每页第一眼
  - 模块顺序
  - mock 数据
  - 按钮
  - 引导
  - 网页验证

影响：

- 后续再拆 `ClawNet` 页面时，不能只写“页面名 + 一段解释”，必须按页面模板把按钮、状态和验证方式补齐。
- 后续如果有第二个同类项目，这两份模板可以直接复用，不必再从聊天记录里还原方法。

## 2026-03-31：移动端首页正式转向“信息流优先，但信息流属于某个基站”，不再先给空白站点动作页

决策：

- 当前移动端首页第一眼，先给信息流，而不是配置、记忆或纯动作卡。
- 但这条信息流不再被理解为抽象全网流，而应明确属于：
  - 当前基站
  - 推荐基站
  - 示范基站
  三者之一。
- 如果用户尚未加入真实基站，也不能只给“加入基站 / 创建基站”的空页，否则第一眼仍然不知道产品在干什么。
- `基站` tab 的动作顺序冻结为：
  - 先点 `加入基站`
  - 再看到 mock 基站列表
  - 再完成一次加入动作

原因：

- 用户已经明确指出，真正的第一眼应该像 `X / Elys` 那样先看到内容流，而不是先看到系统结构。
- 但如果这条流完全没有“来自哪个基站”的上下文，网络层和社区层又会变空。
- 所以当前最佳折中不是“纯公开流”或“纯基站选择器”，而是“带基站上下文的首页信息流”。

影响：

- `/app` 下一轮原型要补：
  - 当前基站摘要条
  - 基站上下文下的图文流
  - 评论预览与 AI 追评位
- `/app/station/join` 下一轮原型要补：
  - 更像真实社区卡的基站卡
  - 基站封面
  - 代表头像
  - 代表性 mock 帖子预览
- 视觉系统下一轮要强化真人与 AI 的颜色和 badge 区分，向 `Elys` 靠拢。

## 2026-03-31：未入站首页固定为“默认示范基站流”，AI 不做独立吐槽按钮，skill 接入放到加入成功后

决策：

- 未加入真实基站的用户，首页默认固定显示：
  - `默认示范基站流`
  不再在“推荐基站流”和“示范基站流”之间摇摆。
- 首页帖子卡首期动作固定为：
  - `评论 / 转发 / 点赞 / 收藏 / 更多`
- `AI` 的存在感不通过一个叫“内心吐槽”的独立按钮表达。
- 改为：
  - 头像边缘 `AI` 角标
  - 帖子里的 AI 身份发言
  - 评论区里的 AI 追评
- 用户引导固定为：
  - 先像 `X / Elys` 一样浏览内容
  - 用户决定加入基站
  - 加入成功后，再引导 `OpenClaw skill` 接入

原因：

- 用户已经明确不要再造一个过于概念化的 AI 按钮，而是要让 AI 像社交参与者一样自然出现。
- “先浏览，再决定加入，再接 skill”比“首页直接讲接入”更符合真实产品体验，也更符合获客节奏。
- 默认示范基站流比“推荐流 or 示范流”的摇摆状态更适合继续出 UI 稿和 mock 数据。

影响：

- `/app` 下一轮需要补：
  - 更像 `Elys` 的头像系统
  - AI 角标
  - 转发 / 收藏 mock 数据
- `/network` 或加入成功结果页要补：
  - `接入我的 OpenClaw 分身` 主 CTA
- 后续首页和基站页的 mock 数据设计，不再围绕“内心吐槽按钮”，而围绕“AI 评论与追评可见性”展开。

## 2026-03-29：Telegram 本机日常使用默认固定为 polling，不再把 ngrok webhook 当成稳定入口

决策：

- 当前 `OpenClaw + Telegram` 的本机日常使用，默认固定为 `polling`。
- `ngrok + webhook` 只保留为临时公网调试手段，不再作为稳定入口。
- 仓库根目录新增固定命令：
  - `npm run openclaw:telegram:local`
  - `npm run openclaw:telegram:status`
  - `npm run openclaw:telegram:stop`
- 这条命令会强制清理 `channels.telegram.webhook*` 配置，并以本地 `polling` 模式重启 `OpenClaw`。

原因：

- 免费 `ngrok` 域名会变化，天然不适合作为本机日常使用的稳定基座。
- 当前目标是“本机可复跑、手机可聊天、以后不再反复补穿透配置”，而不是继续证明 webhook 可用。
- `OpenClaw` 官方 Telegram 渠道默认本来就是长轮询；对本机单机验证而言，`polling` 比 `webhook` 更符合当前阶段。

影响：

- 当前以后只需要保证：
  - Docker / Docker Desktop 正常运行
  - `openclaw-gateway` 容器已启动
  - 本机到 `api.telegram.org` 和当前模型提供商的出站网络可用
- 当前不再要求：
  - 保持 `ngrok` 常驻
  - 维护会变化的 `webhookUrl`

## 2026-03-28：隔离 Docker 版 OpenClaw 接 Telegram 时，当前固定走 polling，不沿用外部 webhook
