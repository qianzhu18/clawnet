# 标准项目目录模板

## 1. 这份文件的用途

这份文件不是给当前 `ClawNet` 继续折腾目录用的。  
它是给“下一个项目”建仓时直接复用的标准目录模板。

目标只有两个：

- 新项目一开始就按软件工程顺序组织文档
- 把参考、命令、设计、修复、历史分开，避免越做越乱

## 2. 对你这次提议的判断

你给的方向是对的，尤其是这 5 个点很值得保留：

- `参考/`
- `commands/`
- `design/`
- `fixes/`
- `history/`

它比“所有文档都塞进一个 `doc/` 目录再慢慢分”更适合长期项目。

但我不建议原样照抄，有两个原因：

- 太重
  - 对还在 `1 / 2 / 3` 步的项目会提前引入过多空目录
- 设计层太深
  - `标准文档 / 概要细节 / 详细设计 / json / tech` 一开始就全开，容易回到“目录很完整，但内容没冻结”

## 4. 这套模板的设计原则

### 4.1 参考和设计必须分开

- `01参考/` 放外部世界
- `03设计/` 放当前项目判断

这样不会把“别人怎么做”和“我们现在怎么做”混在一起。

### 4.2 标准文档和专题文档必须分开

- `标准文档/` 只放阶段主文档
- `概要专题/` 和 `详细专题/` 才放细化件

这样主线清楚，专题也有位置。

### 4.3 commands 必须独立

如果团队要长期和 AI 协作，`commands/` 很值钱。

因为它解决的是：

- 反复提示词复用
- 团队口径统一
- AI 操作可沉淀

### 4.4 fixes 和 history 不能混

- `06修复/` 记录问题和修复逻辑
- `07历史/` 记录时间线和旧文件

这样不会把“问题复盘”和“历史归档”堆在一起。

### 4.5 sql 只在真有数据库时才建

如果项目还在前期，不要一上来就建 `sql/` 空目录。

标准模板里保留它，是为了中后期项目；不是要求所有项目 Day 1 就有。

## 5. 命名建议

### 5.1 目录命名

- 用编号 + 中文
- 不用下划线
- 不用空格

例如：

- `00治理`
- `01参考`
- `03设计`

### 5.2 文件命名

- 主文档用 `数字-阶段名.md`
- 专题文档用 `主题名.md`
- 结构化文件用英文 kebab-case 或 snake_case 都可以，但团队要统一

例如：

- `1-需求分析.md`
- `2-概要设计.md`
- `页面设计.md`
- `route-map.json`

### 5.3 历史文件命名

- 可以保留原始文件名
- 但建议额外补一层目录解释，而不是让它直接混在 active 区

## 6. 什么时候用这套标准模板

### 6.1 适合

- 从 0 开始的新项目
- 准备长期协作的项目
- 多人共写文档的项目
- 会持续使用 AI 命令库的项目

### 6.2 不适合

- 已经推进到中途、且当前目录刚稳定下来的项目
- 只有单人、短期、一次性交付的项目
- 还在找方向，内容远少于目录的项目

## 7. 对当前 ClawNet 的建议

当前 `ClawNet` 不要再立刻切到这套大模板。

原因：

- 当前已经刚完成一轮 active 区收口
- 再重构一次会继续打断阅读和推进
- 当前最缺的不是目录，而是：
  - 页面骨架冻结
  - 原型输出
  - 宿主接入在产品中的表达

所以当前做法应是：

- 保持现有 `doc/` 结构稳定
- 把这份模板当成下一个项目的建仓标准

## 8. 当前结论

你的方向是对的，但最好的处理不是“马上把 ClawNet 再改一遍”，而是：

1. 把它沉成标准模板
2. 当前项目继续用简化版 active 结构
3. 下一个项目从 Day 1 直接按这套模板建
