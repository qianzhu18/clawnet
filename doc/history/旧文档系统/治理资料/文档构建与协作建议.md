# 文档构建与协作建议

## 1. 这份文件解决什么问题

这份文件专门回答团队协作里的 3 个现实问题：

- 文档很多，但新人不知道先读什么
- 读完文档后，不知道怎么形成有效建议
- 想参与团队并提一个 PR，但不知道从哪里切入

## 2. 当前判断

当前项目只保留两段主线：

1. 概要分析
2. 产品原型设计

所以当前协作重点不是“大规模编码”，而是：

- 收敛当前边界
- 稳定概要分析
- 提炼页面交互骨架
- 把这些结论变成下一轮实现准备的输入

## 3. 团队阅读建议

### 3.1 最小阅读包

每个新加入的人，先读这 6 份：

- `../AGENTS.md`
- `当前边界与目标.md`
- `概要分析.md`
- `OpenClaw融入微播客交互.md`
- `产品原型设计.md`
- `主线页面交互骨架.md`

### 3.2 如果你负责产品拆解

再补读：

- `范围清单.md`
- `决策日志.md`

### 3.3 如果你负责后续实现准备

先不要直接把详细设计拉回 active，先补读：

- `范围清单.md`
- `决策日志.md`
- 当前产品主文档

## 4. 读完文档后，应该交什么

当前最推荐的不是“直接写代码”，而是先交一份小报告。

标准结构固定为：

### 4.1 当前理解

- 你认为当前产品到底在做什么
- 你认为当前阶段明确不做什么

### 4.2 你发现的 3 个问题

- 文档不清楚的点
- 页面交互不清楚的点
- 产品链路不顺的点

### 4.3 你的建议

- 建议补哪份文档
- 建议冻结哪条路径
- 建议先做哪一个页面或模块

### 4.4 你想认领的 PR

- 文档 PR
- 原型 PR
- 小型实现准备 PR

## 5. 当前最适合新人的 PR 类型

### 5.1 文档 PR

推荐方向：

- 补专题文档
- 统一命名和引用
- 修正文档之间的矛盾
- 把聊天结论归位进正式文件

### 5.2 原型 PR

推荐方向：

- 补低保真页面
- 补状态稿
- 补按钮交互说明
- 补 mock 数据约定

### 5.3 小型实现准备 PR

只适合已经确认规格稳定的小块。

推荐方向：

- 文案与路由对齐
- 轻量组件收口
- mock 数据与页面状态一致性

当前不推荐新成员一上来就做：

- 大型后端
- 协议层
- 企业后台
- 多节点联邦

### 6.4 之后怎么进入实现准备

路径固定为：

- 先冻结当前边界
- 再冻结概要分析
- 再冻结原型骨架
- 最后再决定是否把详细设计从 `99备份/非当前阶段/` 拉回 active

### 7.2 第二优先

### 7.3 第三优先

- 为关键链路补 1 轮低保真原型和状态稿

## 8. 协作纪律

- 先读文档，再提意见
- 先写结论，再写实现
- 先做小 PR，再做大改动
- 任何建议都必须能落到具体文件
- 任何代码 PR 都必须能指出它对应哪份边界、概要分析或原型文档

## 9. 当前结论

当前最有效的团队参与路径不是“直接写很多代码”，而是：

1. 先读主文档
2. 交一份问题与建议报告
3. 认领一个小而清楚的文档或原型 PR
4. 再逐步进入下一轮实现准备
