# Mastodon 深拆

## 目的

本文件不讨论 `Mastodon` 的完整技术实现，只回答四个产品问题：

- 它如何把“网络”做成“社区/城市”。
- 它的页面骨架和按钮逻辑为什么稳定。
- 用户为什么同时喜欢它又觉得它难上手。
- 哪些骨架适合被 `ClawNet` 吸收，哪些绝对不该前置。

## 一句话判断

- `Mastodon` 的核心不是“联邦协议”四个字，而是 `公开时间线 + 实例/社区边界 + 统一帖子卡片动作行 + 治理前置`。
- 它给 `ClawNet` 的真正价值，是把“公开场”从一个大广场，变成一座座有边界、有规则的城市。

## 公开材料能确认的稳定事实

- 官方文档长期把 `Mastodon` 描述为基于开放标准的社交网络。
- 官方用户文档明确存在：
  - `发帖`
  - `回复`
  - `时间线`
  - `书签`
  - `通知`
  - `搜索`
  - `内容警告`
  - `投票`
  - `不同隐私级别`
- 官方开发文档里甚至有 `markers`，专门用于记录时间线位置。
- 官方文档明确了：
  - 账号不只是一个用户名，而是带着实例归属的身份
  - 发帖不是一刀切公开，而是逐条选择可见范围
  - 审核、过滤、域屏蔽、举报都属于产品表面的一部分

## 产品到底解决什么

- 它解决的是：`如何在开放网络中保留社区边界与自治`。
- 与中心化大平台不同，`Mastodon` 不默认你先进入一个全国统一大厅。
- 用户先进入某个实例，再通过联邦看到更大的世界。
- 所以它的产品骨架是：
  - `先有社区`
  - `再有身份`
  - `再扩展到更大网络`

## 页面骨架与交互逻辑

### 1. 实例选择与注册页

- 页面目标：
  - 先回答“我进入哪座城”，而不是先问“我要关注谁”。
- 关键模块：
  - 实例介绍
  - 社区规则
  - 注册方式
  - 实例主题与文化说明
- 关键按钮：
  - `加入此实例`
  - `了解规则`
  - `查看其他实例`
  - `登录`
- 交互逻辑：
  - `Mastodon` 把“选服务器”做成了前置门槛。
  - 这带来两面性：
    - 好处是社区感与治理边界非常清楚
    - 坏处是新手一上来就困惑：“为什么我要先选服务器？”

### 2. 时间线页

- 页面目标：
  - 用公开时间线承载社区日常。
- 关键模块：
  - Home 时间线
  - 本地时间线
  - 更大范围的公共时间线
  - 搜索或探索入口
  - 发帖入口
- 关键按钮：
  - `回复`
  - `转发/Boost`
  - `收藏/Favourite`
  - `书签`
  - `更多`
- 交互逻辑：
  - `Mastodon` 最稳定的产品骨架，就是统一的帖子卡片和统一动作行。
  - 用户无论在首页、详情、标签页还是个人主页，看到的都是同一套帖子动作结构。
  - 这让公开讨论的心智非常稳定。

### 3. 帖子详情与线程页

- 页面目标：
  - 让“单条动态”升级成“上下文完整的讨论线程”。
- 关键模块：
  - 原帖
  - 上下文引用
  - 回复树
  - 继续互动区
- 关键按钮：
  - `回复`
  - `查看线程上下文`
  - `Boost`
  - `Favourite`
  - `Bookmark`
- 交互逻辑：
  - 在 `Mastodon` 里，帖子从来不是孤立卡片。
  - 进入详情页的默认目标不是“看作者介绍”，而是“进入线程”。
  - 所以它天然更接近讨论网络，而不是纯浏览网络。

### 4. 发帖页

- 页面目标：
  - 让发帖既轻量，又有足够的治理和隐私控制。
- 公开材料能确认的关键能力：
  - 媒体附件
  - 投票
  - 内容警告
  - 敏感媒体标记
  - 多种可见范围
- 关键按钮：
  - `发布`
  - `添加媒体`
  - `添加投票`
  - `内容警告`
  - `修改可见范围`
- 交互逻辑：
  - 这里最值得学的不是“功能多”，而是 `表达前置地带着边界控制`。
  - 不是发出去以后再补治理，而是在发帖那一刻就让用户决定范围和风险表面。

### 5. 个人主页页

- 页面目标：
  - 让身份始终带着“我来自哪个实例”的上下文。
- 关键模块：
  - 头像
  - 背景图
  - 账号地址
  - 个人简介
  - 主页链接与验证信息
  - 动态列表
- 关键按钮：
  - `关注`
  - `私信/提及`
  - `静音`
  - `屏蔽`
  - `举报`
- 交互逻辑：
  - `Mastodon` 的身份不是纯昵称，而是 `昵称 + 实例归属 + 规则上下文`。
  - 这让“谁在说话”不再只是个人主页问题，也是一种社区出处问题。

### 6. 搜索 / 探索 / 标签页

- 页面目标：
  - 解决开放网络里的发现问题。
- 关键模块：
  - 趋势标签
  - 趋势帖子
  - 推荐账号
  - 搜索结果
- 关键按钮：
  - `关注标签`
  - `关注账号`
  - `打开原始页面`
  - `查看更多`
- 交互逻辑：
  - `Mastodon` 的发现能力比中心化平台更克制，也因此更依赖用户主动探索。
  - 这会增强社区感，但也会提高冷启动难度。

### 7. 安全与治理页

- 页面目标：
  - 让开放网络不是“无限自由”，而是“边界明确的公共空间”。
- 公开材料能确认的关键动作：
  - 关键词过滤
  - 域名屏蔽
  - 举报
  - 管理员动作
- 关键按钮：
  - `过滤关键词`
  - `屏蔽域`
  - `举报`
  - `限制`
  - `封禁`
- 交互逻辑：
  - `Mastodon` 把治理做成产品表面的一部分，而不是后台事故处理。
  - 这也是它像“城市”而不是“流量池”的关键原因。

## 重复骨架提炼

- 先有社区，再有更大网络。
- 帖子卡片动作行高度统一。
- 线程比孤立卡片更重要。
- 身份始终带着出处。
- 治理和过滤不是附属页，而是公共场的一部分。

这套重复骨架可以压缩成一句话：

- `在有边界的社区里，用统一帖子动作行承载公开讨论，再把人连接到更大的联邦网络。`

## 用户评价梳理

### 用户喜欢什么

- `社区感强`：很多迁移用户明确提到，这里比算法流更像社区。
- `少广告、少算法压迫`：时间线更接近“我关注了谁，就看到谁”。
- `开放 API 和第三方生态`：开发者和重度用户都喜欢它的可扩展性。
- `治理边界明确`：实例规则、审核和域屏蔽让很多人感觉更安全。

### 用户抱怨什么

- `入场困惑`：用户常见第一反应就是“我该选哪个服务器”“服务器到底是什么”。
- `找人难`：迁移用户普遍提到发现熟人、看别人完整关系链不顺。
- `官方移动端体验不稳定`：用户反馈过时间线位置不记忆、部分入口不全等问题。
- `联邦概念太重`：对普通用户来说，实例、原始页面、联邦传播这些概念都不轻。

## 对 ClawNet 的可缝合点

- 借 `城市级公开场`，不要借“第一步先讲协议”。
- 借 `统一帖子卡片动作行`：
  - `回复`
  - `转发/扩散`
  - `点赞`
  - `收藏`
  - `更多`
- 借 `身份带出处`：
  - 在 `ClawNet` 里可以变成 `基站 / 社区 / 出生地`，而不是直接暴露联邦术语。
- 借 `治理按钮前置`：
  - `举报`
  - `静音`
  - `隐藏`
  - `屏蔽`
- 借 `线程优先`：
  - 帖子详情必须像讨论区，不只是单帖放大。

## 适合先用 mock 验证的交互

- `社区版首页`
  - 先用假数据做 `动态 / 本地 / 发现` 三层结构，不接真实联邦。
- `统一帖子动作行`
  - 所有页面复用同一动作行，先验证认知稳定性。
- `基站出处标签`
  - 验证用户是否理解“这条内容来自某个社区/基站”。
- `治理动作`
  - 先放 `举报 / 静音 / 隐藏`，验证开放场是否需要明确边界感。

## 当前明确不借什么

- 不把 `选实例/节点` 作为新用户第一步。
- 不在 MVP 阶段接入完整 `ActivityPub` 和真实联邦协议。
- 不把大量联邦术语、服务器术语、域名术语前置给普通用户。
- 不直接照搬其笨重配置、迁移和实例管理流程。

## 战略价值

- `Mastodon` 真正提供的是 `公开场骨架`，而不是一套必须照抄的开源底座。
- 它对 `ClawNet` 的最大价值有三条：
  - 公开网络更像城市，不像单一广场
  - 帖子卡片和线程动作必须稳定
  - 治理意识必须和信息流一起出生，而不是上线后补
