# 学习原子

## 用途

这份文件不是传统学习笔记。

它只记录通过当前项目顺手学到、以后还能复用的 `知识最小单元`。

## 学习原子格式

每个 `Learning Atom` 固定包含：

- `场景`
- `概念`
- `为什么对当前项目重要`
- `怎么判断自己学会了`
- `关联文件或下一步`

## 当前学习原子

### LA-001：公开内容网络不等于 agent 协议

- `场景`：讨论 `ClawNet` 是否该先做节点、A2A、MCP、联邦时
- `概念`：公开内容网络、agent 协作协议、工具协议、目录发现层是 4 个不同层级
- `为什么对当前项目重要`：如果不分层，MVP 会被协议复杂度拖死
- `怎么判断自己学会了`：能用自己的话解释为什么 `/network` 页面现在只做方向证明，而不做真实联邦
- `关联文件或下一步`：`概要设计.md`、`F005-network-layer演示.md`

### LA-002：Mock 先于集成

- `场景`：准备做 `connect + QR + app` 演示链路时
- `概念`：先验证产品表面和演示闭环，再决定是否接真实 API
- `为什么对当前项目重要`：当前最缺的是可讲清的 MVP，不是底层联调
- `怎么判断自己学会了`：能清楚区分“产品化接入演示”和“真实协议接入”
- `关联文件或下一步`：`决策日志.md`、`F010-agent接入.md`

### LA-003：扫码进入移动 Web 表面，不等于原生移动端优先

- `场景`：MVP 要参考 `Elys` 的 `扫一扫` 和移动表面
- `概念`：同一个产品可以先用二维码把用户带入移动 Web 表面，而不承诺做完整原生端
- `为什么对当前项目重要`：这能显著提升演示说服力，又不把范围拉爆
- `怎么判断自己学会了`：能解释为什么当前只做 `/app`，而不立刻做 iOS / Android
- `关联文件或下一步`：`产品原型设计.md`、`主线页面交互骨架.md`

### LA-004：规格先于开发

- `场景`：准备开始写页面前
- `概念`：`范围清单` 定范围，`功能规格` 定验收，`本周待办` 定本周执行
- `为什么对当前项目重要`：否则会一边写代码一边改目标
- `怎么判断自己学会了`：能在动手前明确说出“我现在是按哪个 spec 在做”
- `关联文件或下一步`：`范围清单.md`、`功能规格/`

### LA-005：产品拆解先拆演示闭环，再拆系统实现

- `场景`：决定这周先从 `T001 /connect` 开始，而不是先补协议和 API
- `概念`：先拆“用户能走通的一条演示链路”，再拆“系统底层怎么真正跑起来”
- `为什么对当前项目重要`：当前最容易失控的地方，就是一上来把去中心化连接当成底层工程题
- `怎么判断自己学会了`：能把 `connect -> QR -> app -> network` 当作产品闭环讲清，而不先掉进协议细节
- `关联文件或下一步`：`本周待办.md`、`F010-agent接入.md`

### LA-006：连接感来自连续身份，不来自真实通信先打通

- `场景`：把本周 MVP 提高到“命令行接入后，扫码进入网页 app，并完成加入基站”
- `概念`：对用户来说，强连接感首先来自身份连续、状态连续、动作连续，而不是先来自底层通信已经连通
- `为什么对当前项目重要`：这样可以在 mock 数据阶段先把 `npm 命令 -> 二维码 -> 移动信息流 -> 加入基站` 做得像真实产品
- `怎么判断自己学会了`：能解释为什么这周先要求“接入身份可见、加入状态可变”，而不是先做协议联调
- `关联文件或下一步`：`概要设计.md`、`F005-network-layer演示.md`

### LA-007：先验证本地可执行命令，再做公网分发

- `场景`：讨论 `connect` 是否现在就要做公网 npm 包和后端接口
- `概念`：分发形态和演示闭环是两个问题，先验证“任意本地 Node.js 目录里能跑命令并弹出二维码”，再决定是否进入公网发布
- `为什么对当前项目重要`：这样能把本周焦点锁在 `CLI -> QR -> 手机 Web App`，避免过早陷入包发布和后端规划
- `怎么判断自己学会了`：能解释为什么本周先做 local CLI demo mode，而不是先做 npm 公网包
- `关联文件或下一步`：`F010-agent接入.md`、`本周待办.md`

### LA-008：任务编号不是优先级，检测通过才算完成

- `场景`：发现本周 TO DO 仍然按 `T001 / T002 / T003` 排，容易误导执行顺序
- `概念`：任务 `ID` 只负责稳定引用；真正推进顺序要按 `P0 / P1 / P2` 管理，且完成必须有检测记录
- `为什么对当前项目重要`：这样才能把精力持续压在当前关键路径，而不是被历史流水号牵着走
- `怎么判断自己学会了`：能说清当前先做哪些 P0 项，以及为什么某任务虽然“看起来做了”但还不能算 `done`
- `关联文件或下一步`：`本周待办.md`、`验证日志.md`

### LA-009：Demo 构建不能依赖外网资源

- `场景`：本机 `npm run build` 因为在线 Google 字体拉取失败而中断
- `概念`：演示版产品的构建链应优先选择本地可复跑的资源，不把外网请求当成默认前提
- `为什么对当前项目重要`：否则看起来只是“字体问题”，实际会直接阻断接入链路演示和交付节奏
- `怎么判断自己学会了`：能解释为什么 Phase 0 先用本地 fallback 字体栈，而不是继续追求在线字体一致性
- `关联文件或下一步`：`src/app/layout.tsx`、`src/app/globals.css`、`验证日志.md`

### LA-010：MVP 验证先跑最短用户闭环，不先补整套系统

- `场景`：决定继续从 `connect demo` 往前推，直到用户能亲手试玩 MVP
- `概念`：真正可验证的 MVP，不是“页面数量变多”，而是用户能按固定顺序走完一条价值链
- `为什么对当前项目重要`：这样才能判断公开信息流、帖子详情、agent 创建/接入是否真的连成产品，而不是各自成立
- `怎么判断自己学会了`：能用一句话说清当前试玩顺序是 `/ -> /posts/:id -> /agents/new 或 /connect`
- `关联文件或下一步`：`核心需求定义.md`、`本周待办.md`、`决策日志.md`

### LA-011：真实手机扫码不等于 `localhost` 可用

- `场景`：准备把本地 CLI 输出的二维码交给真实手机扫码试玩
- `概念`：CLI 里写 `http://localhost:3000` 只适合当前电脑本机预览；真实手机需要 `0.0.0.0` 监听加局域网地址
- `为什么对当前项目重要`：如果不区分这两种运行方式，MVP 看起来已经完成，但用户一拿手机就会误判为“二维码不可用”
- `怎么判断自己学会了`：能解释为什么要用 `npm run start:lan`，并把 `CLAWNET_HOST` 改成 `http://<局域网IP>:3000`
- `关联文件或下一步`：`package.json`、`本地CLI演示执行.md`、`MVP验证手册.md`

### LA-012：OpenClaw workspace skill 不是“把文件放进去”就算成立

- `场景`：为 `T030` 把 `clawnet-connect-bridge` 放进 `OpenClaw` 的隔离 workspace
- `概念`：workspace skill 至少同时满足 3 个条件才会被宿主识别：位于 `<workspace>/skills/<name>/`、`SKILL.md` 带 `name / description` frontmatter、运行脚本能在被复制后仍定位真实仓库
- `为什么对当前项目重要`：否则会出现“目录里有文件，但 `skills list` 看不到”或“skill 被识别了，但 bridge 找不到 `ClawNet` repo”这两种假完成
- `怎么判断自己学会了`：能用 `skills info clawnet_connect_bridge --json` 看见 `source = openclaw-workspace`，并从 workspace 内 `bridge.sh` 真实拿到 `connect_url / pair_url / host_mode / scan_ready`
- `关联文件或下一步`：`examples/openclaw-skill/clawnet-connect-bridge/`、`OpenClaw本地测试手册.md`、`T031`

### LA-013：真实宿主接入要拆成“bridge 可验收”和“宿主内动作可触发”两层

- `场景`：为 `T031` 继续推进 `OpenClaw -> bridge -> /connect -> /pair -> /app` 时
- `概念`：同一 API key 在宿主外直连模型接口成功，不代表宿主内嵌 `agent --local` 或 `/skill` 触发路径也一定成功；两者必须分层验证
- `为什么对当前项目重要`：这样才能在上游运行时卡住时，先固定“手机扫码进入微博客表面”的验收链，不把 UI 与产品验证一起拖住
- `怎么判断自己学会了`：能解释为什么当前 `demo:openclaw:bridge` 已可作为本地验收基线，而 `T031` 仍需保持 `blocked`
- `关联文件或下一步`：`OpenClaw宿主设计.md`、`OpenClaw本地测试手册.md`、`验证日志.md`

### LA-014：Docker 版 OpenClaw 的入口是宿主机 CLI 包装，不是进容器手敲

- `场景`：在 `T030 / T033` 的隔离 `OpenClaw` 环境里，发现一进容器手动 `bash` 很难复跑与教学
- `概念`：Docker 版日常使用应统一理解为 `docker compose run --rm openclaw-cli <command>`；宿主机终端是入口，`openclaw-gateway` 容器只是常驻服务
- `为什么对当前项目重要`：否则 `dashboard / devices / models / config / onboard` 都会退化成一次性容器黑魔法，无法稳定交接
- `怎么判断自己学会了`：能用 `alias oc='docker compose run --rm openclaw-cli'` 连续执行 `oc dashboard --no-open`、`oc devices list`、`oc models list`、`oc config get agents.defaults.model.primary`
- `关联文件或下一步`：`openclaw/docker-compose.yml`、`openclaw/docs/zh-CN/install/docker.md`、`T033`

### LA-015：OpenClaw 的模型下拉框不是 provider 目录，而是 allowlist

- `场景`：智谱 provider 已接通，但 Web UI 模型选择器里仍只看到 `glm-4.7 + google`
- `概念`：`agents.defaults.models` 同时负责模型允许列表和 `/model` / Web 选择器展示；provider 目录里有模型，不代表 UI 一定会显示
- `为什么对当前项目重要`：否则会误以为“模型没接上”，实际只是没有把目标模型写进允许列表
- `怎么判断自己学会了`：能解释为什么把 `zai/glm-5` 加进 `agents.defaults.models` 后，下拉框才会出现 `GLM-5`
- `关联文件或下一步`：`OpenClaw本地测试手册.md`、`~/.openclaw-t030/openclaw.json`

### LA-016：Z.AI 的“套餐未开放模型权限”在 OpenClaw 里可能表现成 rate limit

- `场景`：把 `zai/glm-5` 设成主模型后，Web UI 报 `API rate limit reached`
- `概念`：智谱有些权限不足错误会以 `429` 返回；OpenClaw 会把这类 `429` 统一收敛成 rate limit 提示，但网关日志里的 `rawError` 会保留上游原文
- `为什么对当前项目重要`：否则会误判为“请求太频繁”，实际问题是套餐没开对应模型
- `怎么判断自己学会了`：能在日志里区分 `429 当前订阅套餐暂未开放GLM-5权限` 和真正的高并发限流
- `关联文件或下一步`：`OpenClaw本地测试手册.md`、`docker logs openclaw-openclaw-gateway-1`

### LA-017：OpenClaw 一旦设置 `plugins.allow`，内建 Telegram channel 也要手动放行

- `场景`：隔离 `OpenClaw` 宿主已经写入了 `channels.telegram` 和 bot token，但 `channels status` 仍然是空白
- `概念`：`plugins.allow` 不只约束第三方扩展，也会约束内建 channel 插件；如果没有显式包含 `telegram`，provider 根本不会启动
- `为什么对当前项目重要`：否则会误把问题归咎于 token、webhook 或 pairing，而真正的阻塞其实是 plugin allowlist
- `怎么判断自己学会了`：能在 `plugins list --json` 里看出 `telegram` 是 `not in allowlist`，并在补上 allowlist 后让 `channels status --json --probe` 出现 `configured=true / running=true / mode=polling`
- `关联文件或下一步`：`OpenClaw本地测试手册.md`、`决策日志.md`

## 使用规则

- 只有能复用的知识才记录。
- 只有能回链到当前项目的知识才记录。
- 一条原子尽量控制在 5 行内，不写成长文。
