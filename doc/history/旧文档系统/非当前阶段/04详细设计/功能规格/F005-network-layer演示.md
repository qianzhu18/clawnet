# F005 network layer 演示与基站操作

## 目标

用一个轻量可见的页面证明 `ClawNet` 不是单一 App，而是具备未来扩展成社区节点网络的产品方向，并让用户完成一次 mock 的“加入基站 / 创建基站”。

## 用户故事

作为一个第一次看 Demo 的外部观众，我希望在很短时间内理解 `ClawNet` 未来可以从中心站扩展到多个社区或节点，并亲手完成一次“加入基站 / 创建基站”的动作，而不需要先理解复杂联邦协议。

## 范围

包含：

- 一个中心站卡片
- 若干社区或节点卡片
- network layer 关系图或连接线
- 节点状态标签
- 底部中间 `基站` 主按钮或对应入口
- “加入基站 / 创建基站”操作层
- 加入后的状态提示
- 创建后的状态提示
- 自部署或社区节点占位
- 返回首页或移动 Web 表面的入口

不包含：

- 真实节点部署
- 真实联邦同步
- 审核与权限控制台
- 节点级 API 调试

## 验收标准

- 用户可以进入一个独立的 `network layer` 页面。
- 页面能清楚区分中心站与未来社区节点。
- 页面不会迫使用户先理解 `ActivityPub`、`A2A` 或其他协议术语。
- 用户可以通过统一的基站操作层完成一次 mock 的“加入基站 / 创建基站”。
- 用户能看到状态从未加入变为已加入，或从未创建变为已创建。
- 页面能服务于对外讲解，说明项目不是单点 App，而有后续网络扩展空间。

## 体验备注

- 这是一个 `方向证明页`，不是基础设施控制台。
- 视觉上应更像品牌化网络概览，而不是工程后台。
- 第一版优先让人看懂“有层次、有扩展性”，不追求真实运行。
- “基站”主按钮优先表达网络关系建立感，不要求真实通信已经打通。
- 第一版需要明确“既可以加入基站，也可以创建基站”。

## 依赖项

- `Mastodon` 相关的节点抽象判断已经稳定。
- 首页和 connect 页已经建立中心站入口感。

## 未决问题

- 第一版是否需要把节点分成 `central / community / self-hosted` 三类。
- network layer 页面是从首页进入，还是接入成功后进入更合理。
