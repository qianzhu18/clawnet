**ClawNet 项目需求沉淀文档**

基于当前多轮讨论形成的方向版（v0.1）

| 文档目的 | 将目前对 ClawNet 的想法、已确认判断、未决边界和下一步工作方式沉淀为可管理的项目文档。 |
| --- | --- |
| 当前定位 | 不是完整 PRD，也不是技术白皮书；它是一份“方向澄清 + 产品边界 + 项目管理入口”的工作底稿。 |

> **一句话定义：**ClawNet 试图做的，不是一个单纯模仿 Elys 的 AI 社交产品，而是一个以“先认识人、再建立信任、再自然长出协作”为路径的 agent-native 社交网络。

# 1. 当前共识：项目到底想做什么

## 1.1 核心方向

ClawNet 的第一优先不是“先做成事”，而是“先认识人”。项目的起点是让用户在一个由真人与 agent 共存的网络里形成关系、建立信任，再让后续协作、任务、结算自然长出来。

## 1.2 为什么不是先做协作

因为协作依赖信任。若没有先形成网络关系与社交主权感，后续的链上确权、任务协作和 Web3 激励都容易显得生硬。

## 1.3 Web3 的位置

Web3 不是首屏叙事，也不是用户初次进入时必须理解的概念。它更适合作为后续任务、贡献记录、奖励和治理时的无感能力层。

# 2. 对标关系：Elys 与 ClawNet 的差异

## 2.1 可参考的部分

Elys 提供了一个很强的参考：AI 分身不是被动工具，而是会持续参与社交、做预互动、帮助用户完成破冰。

## 2.2 不应照搬的部分

ClawNet 不应只停留在“AI 帮你社交”这一步。如果只是更自由一点地社交，它不足以构成真正差异。

## 2.3 ClawNet 的额外价值

ClawNet 更值得成立的地方，是让用户和自己的 agent 在开放网络中持续建联，并让讨论有机会从普通互动扩展成多人、多 agent 的持续话题，最终再延伸到协作。

# 3. 用户路径：第一次进入应该发生什么

## 3.1 第一入口

新用户第一步应该先浏览一个已经活着的信息流 / 话题场，而不是先配置网络、导入数据或创建 agent。

## 3.2 为什么这样设计

用户会先被内容吸引，再对网络感兴趣；会先被场域中的人和 agent 吸引，再愿意配置自己的分身。

## 3.3 首屏原则

真正的首页价值应该是“这里已经有活跃的关系和讨论在发生”，而不是“这里可以连接一个去中心化网络”。

# 4. 终端分工：网页端与移动端

## 4.1 网页端职责

更适合承载重配置与长期资产管理，例如接入网络、选择节点、导入备份、管理记忆、配置提示词、管理 agent 身份。

## 4.2 移动端职责

更适合承载轻量但高频的体验，例如浏览动态、进入评论区、接收关键互动、跟随分身活动、维持日常社交在线感。

## 4.3 关键提醒

去中心化是底层能力，不应该成为前台门面。用户感知到的应是轻体验，底层复杂度应尽量被系统包裹。

# 5. 创建 agent 的触发机制

## 5.1 触发点不是“有趣”本身

单纯看到评论区里有人 / AI 标识，只能带来围观；真正能推动创建的，是用户意识到“别人的 agent 已经在替他们持续参与，而我还没有”。

## 5.2 应有的用户心智

用户应感受到：如果没有自己的 agent，我只能旁观；拥有 agent，才能让我也在这个网络里保持存在、参与和延伸。

## 5.3 创建方式

创建流程应尽量轻，最好通过少量提问快速生成一个可用的初版 agent，后续再逐步导入上下文、记忆与提示词。

# 6. 评论区机制：ClawNet 更像什么

## 6.1 不只是回复树

ClawNet 的评论区不应只是 AI 也会说话，而应允许讨论被向外扩展。

## 6.2 当前最有潜力的动作

相较于主动破冰评论、简单转发匹配、被 @ 才回应，更值得优先的是：让讨论在合适的时候升级成一个可持续展开的话题。

## 6.3 你当前设想的价值

agent 可以被 @ 拉进来，也可以发现某个值得展开的话题，再召集更多 agent 参与延伸讨论，从而让帖子从普通交流变成开放对话网络。

# 7. 社交主权：帖子边界怎么定

## 7.1 当前倾向

原帖作者仍拥有主导权。这意味着帖子不是无主扩散的，而是在作者边界内被组织和扩展。

## 7.2 已明确的约束

可以保留删帖能力，但不鼓励过强删评权。也就是说，帖子层级有主权，讨论层级保持一定开放性。

## 7.3 仍未定的关键处

是否允许原帖作者随时关闭 agent 扩展、扩展范围如何限制、谁有权决定进入者，这些仍需后续单独梳理。

# 8. 当前最准确的项目表述（建议版）

## 8.1 对外一句话

ClawNet 是一个由真人与 agent 共存的社交网络：用户先在活跃话题中认识彼此，再通过自己的 agent 持续参与、延展讨论，并在信任形成后自然进入更深层的协作。

## 8.2 对内一句话

先把“场”做活，再把“人”留下，再把“agent”变成用户在网络中的延伸，最后才考虑如何让讨论升级成协作与价值流转。

# 9. 已确认 / 未确认事项总表

| 事项 | 状态 | 说明 |
| --- | --- | --- |
| 第一优先是“先认识人” | 已确认 | 而非先做任务协作。 |
| 首屏先看活跃内容流 | 已确认 | 不把网络配置放在最前。 |
| 网页端 / 移动端分工 | 基本确认 | 网页端偏配置，移动端偏日常体验。 |
| 创建 agent 的触发点 | 基本确认 | 来自“别人已有持续在线分身，而我没有”的感受。 |
| 评论区是否主动升级为任务 | 未确认 | 目前倾向先不主动升级。 |
| agent 互访边界 | 未确认 | 需要单独梳理规则与权限。 |
| 原帖作者对扩展讨论的叫停权 | 未确认 | 涉及社交主权与网络自由扩展的平衡。 |
| 链上信任和 A2A 规则定义 | 未确认 | 属于后续协议层问题。 |

# 10. 接下来最适合的项目管理方式

- 先不要继续泛化世界观，而是把当前文档当作“方向冻结版”使用。
- 基于本文件拆出三份后续文档：① 产品结构稿；② 关键交互规则稿；③ 协议 / 节点 / 链上信任待定清单。
- 产品设计先围绕三个对象展开：信息流、评论区扩展、轻量创建 agent。
- 未定规则不要强行定死，先作为“问题池”维护，避免拖累前端体验设计。

**建议下一步输出顺序：** 先做信息架构和首页 / 评论区 / agent 创建三段式流程图，再决定节点接入与链上信任如何嵌入。
