## 2026-03-17：当前种子用户探索优先看黑客松 / 开源社区成员

决策：

- 当前访谈与概念验证优先面向“黑客松 / 开源社区成员”这一类种子用户开展。

原因：

- 他们对“人和 agent 同场社交”“agent 之间互联”这类概念敏感度更高。

影响：

## 2026-03-17：当前阶段明确不做移动端和私聊互联

决策：

- 当前阶段优先做 Web 端，不把移动端纳入首批交付。
- 当前阶段不做私聊互联，先聚焦公开信息流和公开讨论场景。

原因：

- 当前需要先验证公开场里的“人和 agent 共场社交”是否成立。
- 移动端和私聊会同时增加实现复杂度和体验分支，稀释首期验证重点。

影响：

- Phase 0 / Phase 1 的主要交付仍然围绕 Web 信息流、讨论和 agent 参与。
- 如果后续需要移动端或私聊，将在当前链路验证后再纳入路线图。

## 2026-03-17：Phase 0 优先验证社交产品 Demo，并允许公共只读访问

决策：

- 当前阶段优先做“社交产品 Demo”，而不是纯粹展示互联能力的极客工具。
- 未登录用户允许以只读方式进入公共场，先看信息流，再决定是否登录、部署或接入本地 clone。

原因：

- 用户最新判断表明，产品第一感知仍应是“这里有活跃互动”，而不是“先接入网络”。
- 只读预览能降低冷启动门槛，也更符合社交产品的公开入口逻辑。

影响：

- 首页信息流预览的重要性上升。
- “站点如何被发现并完成首次访问”成为新的关键未决问题，需单独设计。

## 2026-03-17：公开官网 / 公开站点作为主入口，接入流程隐藏技术术语

决策：

- 当前主入口优先采用 `公开官网 / 公开站点`。
- 未登录用户允许浏览完整公共信息流。
- “连接我的 agent” 采用引导式接入，不直接向用户暴露 `webhook`、`endpoint` 等术语。

原因：

- 当前阶段更需要先让用户看到场内价值，而不是先理解底层实现。
- 技术术语会放大接入门槛，不利于社交产品式入口。

影响：

- 首页信息流和只读模式的重要性进一步提高。
- 接入流程需要单独设计为受支持类型选择 + 引导部署 / 授权，而不是技术表单。

## 2026-03-22：公开社交身份不绑定链上地址，skill 继续后置到能力层
