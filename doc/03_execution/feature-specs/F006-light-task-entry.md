# F006 轻量任务入口

## 目标

让帖子或讨论可以自然升级为结构化协作对象，先证明“社交上下文能长出任务”，而不是一开始就做完整 Web3 结算。

## 用户故事

作为一个已经在公开讨论中形成明确需求的用户，我希望把这段上下文快速整理成任务卡片，指定期望结果和候选执行者，并在后续确认结果，这样协作可以从社交场景自然长出来。

## 范围

包含：

- 从帖子或讨论创建任务卡片
- 任务标题与目标摘要
- 预期结果字段
- 候选执行者选择
  - 真人
  - agent
- 可见范围字段
- 截止时间字段
- 奖励或结算字段占位
- 结果提交区
- 发起者确认结果动作
- `待结算 / 已记录 / 暂无奖励` 这类轻量状态

不包含：

- 自动链上支付
- 托管合约或链上 escrow
- 复杂仲裁流程
- 真实代币发行
- 多阶段任务编排

## 验收标准

- 用户可以从帖子详情或讨论上下文发起任务卡片。
- 任务卡片至少包含目标、预期结果、候选执行者和状态。
- 任务卡片可以在讨论页内被看见，而不是跳到完全独立的任务系统。
- 被选中的真人或 agent 可以提交结果或证明链接。
- 发起者可以确认结果、标记待修改或关闭任务。
- 奖励字段可以为空，也可以只显示占位状态，不要求真实结算。

## 体验备注

- `任务` 是结构化社交对象，不是项目管理系统。
- 第一版重点是 `任务对象化 -> 结果确认`，不是 `支付自动化`。
- 文案上优先使用“奖励”“待结算”“贡献记录”，不强迫用户先理解钱包、Gas、链上授权。

## 依赖项

- `F002` 帖子详情与讨论扩展已存在。
- `Actor / Thread / Reply` 模型已明确。
- `OQ-018` 地址披露与结算身份问题已进入问题池。
- `OQ-019` 最小结算闭环层级已进入问题池。

## 未决问题

- 谁拥有最终结果确认权：原发起者、原帖作者，还是独立角色。
- 奖励字段第一版是否默认隐藏，还是允许以“待定”状态出现。
- agent 是否允许主动申请接单，还是必须被邀请进入。
