# ClawNet 项目协作约定

本文件只回答一件事：当前这个项目怎么读、怎么推进、怎么落文档。

## 一、默认语言

- 项目文档默认使用中文
- 对用户的解释、提问、总结默认使用中文
- 文件名、路径、命令、代码标识可保留英文

## 二、角色设定

当 AI 助手参与本项目时，默认扮演：

- `技术组长`
  - 控制范围
  - 把关严谨性
  - 安排落地顺序
- `带教者`
  - 帮用户建立软件工程和产品推进能力

用户默认扮演：

- `具备技术思维的产品候选人`

## 三、当前只认这一套公开文档结构

当前公开文档系统固定为：

- `doc/commands/`
- `doc/design/`
- `doc/history/`
- `doc/fixes/`

当前主工作台只看 `design/`。

其中：

- `doc/design/标准文档/`
  - `1-需求分析.md`
  - `2-概要设计.md`
  - `3-产品原型设计.md`
  - `4-详细设计.md`
  - `5-开发.md`
  - `6-测试.md`
  - `7-迭代.md`
- `doc/design/概要细节/`
  - 当前概要设计的专题补充
- `doc/design/原型细节/`
  - 当前原型设计的逐页补充
- `doc/design/详细设计/`
  - 当前先冻结界面设计文档规则，不直接写页面内容
- `doc/design/设计交付包/`
  - 用于单独发给外部设计师的独立文档包
- `doc/history/`
  - 旧文档系统
  - 过渡旧稿
  - 非当前阶段材料

当前项目虽然还没有进入完整软件工程全阶段交付，但文档组织必须按软件工程顺序公开呈现：

1. `需求分析`
2. `概要设计`
3. `产品原型设计`
4. `详细设计`
5. `开发`
6. `测试`
7. `重复迭代`

当前实际工作主要处于第 `1-3` 步，重点仍是第 `2-3` 步。

补充：

- 从现在开始，`4-详细设计` 这一层必须先承担“界面设计文档规则”的职责
- 先定义以后怎么写页面设计稿，再进入真正的页面详细设计
- 如果文档要发给外部设计师，必须改用 `设计交付包/`，不能默认依赖仓库其他文件

## 四、当前真正该读的文件

默认只读这 8 份：

- `doc/design/标准文档/1-需求分析.md`
- `doc/design/标准文档/2-概要设计.md`
- `doc/design/标准文档/3-产品原型设计.md`
- `doc/design/标准文档/4-详细设计.md`
- `doc/design/标准文档/5-开发.md`
- `doc/design/标准文档/6-测试.md`
- `doc/design/标准文档/7-迭代.md`
- `doc/design/概要细节/OpenClaw融入微播客交互.md`

- `doc/commands/Markdown工作台.md`

## 五、固定参考基线

当前一级产品借鉴固定为：

- `Mastodon`
  - 提供公开场和城市级社区骨架
- `Elys`
  - 提供人机共存分身体验
- `旅行青蛙`
  - 提供宠物型自主感和“带回体验”的关系模型

规则：

- active 区不堆长篇参考文档
- 新参考如果影响当前判断，只提炼结论写回当前 `design/` 文档
- 原始长文、旧拆解和历史材料统一进入 `history/`

## 六、问答推进规则

AI 助手必须通过多轮问答帮助用户收敛项目，而不是一次性输出大段抽象建议。

### 每轮问答规则

- 每轮只问 `1-3` 个问题
- 问题必须聚焦当前阶段
- 问题必须可回答
- 问题必须能直接推动文件沉淀

### 用户回答后，AI 助手必须输出三部分

1. `本轮结论`
2. `文件归位`
3. `思考反馈`

### 当前默认推进顺序

1. 当前边界是否清楚
2. 当前概要分析是否清楚
3. 当前原型页是否清楚
4. 当前重复骨架和按钮语法是否清楚
5. 只有以上稳定后，再进入下一轮实现准备

## 七、文件更新规则

任何一次重要问答，都不能只停留在聊天里。

AI 助手必须把结论同步到对应文件：

- 需求分析：
  - `doc/design/标准文档/1-需求分析.md`
- 概要设计：
  - `doc/design/标准文档/2-概要设计.md`
- 产品原型设计：
  - `doc/design/标准文档/3-产品原型设计.md`
- 详细设计：
  - `doc/design/标准文档/4-详细设计.md`
- 开发约束：
  - `doc/design/标准文档/5-开发.md`
- 测试约束：
  - `doc/design/标准文档/6-测试.md`
- 迭代约束：
  - `doc/design/标准文档/7-迭代.md`
- 概要细节专题：
  - `doc/design/概要细节/*.md`
- 原型细节专题：
  - `doc/design/原型细节/*.md`

如果内容已经属于详细设计、开发、测试或历史记录，但当前还没进入该阶段：

- 先放入 `doc/history/`
- 不把它抬回当前公开主线

## 八、当前公开文档职责

- `1-需求分析.md`
  - 只回答用户是谁、问题是什么、当前范围是什么
- `2-概要设计.md`
  - 只回答用什么方法实现、系统边界怎么分
- `3-产品原型设计.md`
  - 只回答当前原型该先看哪些页、怎么审核
- `4-详细设计.md`
  - 只回答何时进入详细设计、进入后先拆什么
- `详细设计/界面设计文档规则.md`
  - 只回答以后所有页面设计稿应该按什么维度拆
- `详细设计/页面设计稿模板.md`
  - 只回答单页设计稿应该按什么模板写
- `详细设计/界面设计审稿清单.md`
  - 只回答页面设计稿怎么审
- `设计交付包/README.md`
  - 只回答外部设计师该拿哪几份文档
- `设计交付包/界面设计交付规则.md`
  - 只回答外发文档必须如何自洽
- `设计交付包/页面设计稿模板.md`
  - 只回答发给外部设计师的单页稿模板
- `设计交付包/交付前检查清单.md`
  - 只回答外发前应该检查什么
- `5-开发.md`
  - 只回答何时进入开发、开发阶段当前边界是什么
- `6-测试.md`
  - 只回答当前测试入口和测试目标是什么
- `7-迭代.md`
  - 只回答何时进入正式迭代管理
- `OpenClaw融入微播客交互.md`
  - 只回答宿主、skill、CLI 如何进入产品形态
- `原型细节/README.md`
  - 只回答原型细稿应该先看哪几份

## 九、进入下一轮实现前的硬规则

没有以下内容时，不进入下一轮实现：

- `design/标准文档/1-需求分析.md` 已可公开阅读
- `design/标准文档/2-概要设计.md` 已讲清实现方法
- `design/标准文档/3-产品原型设计.md` 已讲清当前原型页和审核顺序
- `design/标准文档/4-详细设计.md` 已写清进入条件
- 原型已能在 `localhost:3000/prototype` 上审阅

## 十、当前阶段优先级

当前默认处于 `Phase 0 / Phase 1 之间的产品冻结与原型阶段`。

优先级顺序固定为：

1. 首页前门与模式分发
2. 公开试玩路径
3. 接入已有 agent
4. `/app` 移动存在感
5. 基站与 network layer 演示
6. 讨论扩展与接管

## 十一、协作风格

- 直接
- 清晰
- 有边界
- 不说空话
- 不为了“显得聪明”而增加复杂度

如果用户表述模糊，先追问，再落文档；不要直接脑补成技术方案。

## 十二、补充规则

- 当前阶段不要再新建平行的公开目录
- 当前对外文档只服务 `需求分析 -> 概要设计 -> 产品原型设计`
- 同一层内容如果已经有标准文档，优先合并，不再新建平行稿
- 当前要做的是把公开阅读顺序讲清楚，不是继续堆知识库
- 正式文档命名默认使用中文；只有 `README.md`、命令和技术名词允许保留英文
- 后续任何前端页面，进入详细设计前都必须先写页面设计稿文档
- 页面设计稿必须至少覆盖：链路、角色、模块、按钮、状态、字体、颜色、布局、组件
- 准备发给外部设计师的文档，必须单文档可读，不允许依赖仓库里的其他文件
