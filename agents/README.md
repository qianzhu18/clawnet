# Agents 角色分层说明

本目录用于给 `ClawNet` 的多对话框协作提供 `角色层`。

它不替代根目录的 `AGENTS.md`，而是在其基础上补充：

- 每个窗口应该扮演什么角色
- 每个角色负责什么、不负责什么
- 多角色之间如何交接

## 与根 `AGENTS.md` 的关系

### 1. 根 `AGENTS.md`

作用：

- 仓库级总章程
- 默认语言、协作风格、文件更新规则
- 项目方法论、阶段优先级、文档系统约束

适用范围：

- 对本仓库中所有 AI 助手和所有角色窗口都生效

### 2. `agents/*.md`

作用：

- 给某一个特定窗口增加 `角色上下文`
- 明确这个窗口在开发流水线中的职责、输入、输出和边界

适用范围：

- 只对绑定该角色的窗口生效

## 指令优先级

默认按以下顺序理解：

1. 用户当前明确要求
2. 根 `AGENTS.md`
3. 本目录中的角色文件
4. 当前任务临时上下文

如果冲突：

- 先服从用户当前要求
- 再服从根 `AGENTS.md`
- 角色文件只负责细化，不负责推翻总章程

## 推荐角色体系

当前按标准开发流水线拆成 5 个角色：

1. `project-supervisor.md`
2. `product-manager.md`
3. `engineering.md`
4. `qa.md`
5. `devops.md`

## 推荐使用方式

每开一个新窗口，只绑定一个角色文件。

不要：

- 一个窗口同时扮演多个角色
- 让测试窗口直接决定产品优先级
- 让运维窗口直接定义交互和功能边界
- 让研发窗口绕过测试直接宣布“已完成”

## 统一交付格式

所有角色窗口默认使用同一输出骨架：

1. `本轮结论`
2. `已验证事实`
3. `阻塞项 / 风险`
4. `需要上游或主管拍板的点`
5. `下一步`

## 正式归档责任

为了避免多窗口同时改正式文档，默认采用：

- `产品经理 / 研发 / 测试 / 运维` 先产出候选结论、证据和建议
- `项目主管` 负责收口、裁冲突、决定是否写回正式文档

也就是说：

- 角色窗口可以先讨论和收敛
- 是否进入 `doc/` 正式文档系统，原则上由 `项目主管` 统一触发

## 当前阶段建议的主从关系

- `项目主管`：唯一总控窗口
- `产品经理`：定义范围和验收，不直接替代研发
- `研发`：负责实现和集成，不替代测试验收
- `测试`：负责验证闭环，不替代产品定范围
- `运维`：负责环境、脚本、发布和部署，不替代研发做主功能

## 当前 `ClawNet` 项目的推荐绑定

基于当前 `P0` 状态，建议这样理解：

- `项目主管`：控阶段目标、排优先级、裁冲突
- `产品经理`：收口 `首页 -> 帖子详情 -> 创建 agent -> connect -> pair -> app -> network`
- `研发`：守住主链路代码真相与集成质量
- `测试`：把试玩路径、截图、验收脚本、回归记录做实
- `运维`：守本地运行、构建、发布形态、样例目录、部署准备

## 直接开工入口

如果准备马上开多个窗口，优先看：

- `current-phase-setup.md`
- `launch-prompts.md`

前者回答“当前阶段到底该开哪几个窗口、谁先动、谁后动”。  
后者提供可直接复制到新对话框里的启动提示词。
