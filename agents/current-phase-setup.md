# 当前阶段多窗口执行方案

## 当前判断

`T026` 已经把“真机扫码接入 agent”这条链路打通。

当前最值得投入的，不再是继续补接入链路，而是把“已接入 agent 的微播客存在感”收成一版更像产品的 UI。

当前阶段最值得投入的是：

1. `产品经理`
   - 冻结 `T027`：接入成功后的微播客首屏目标、层级、即时动作和不做项
   - 明确哪些元素要强化“我的 agent 已接入”，哪些 demo 感元素要降级
2. `研发`
   - 承接 `T028`：按产品冻结结果实现首轮 UI 收口
   - 保持 `T026` 路径不回退，不破坏真机接入链路
3. `测试`
   - 在桌面、模拟移动与真机接入模式下复核 UI 收口结果
   - 补“真实手机摄像头扫屏”外部证据，并给出是否允许通过
4. `项目主管`
   - 控制这一轮只做接入成功后的微播客 UI
   - 决定哪些 UI 改动进入实现，哪些延后
   - 决定测试通过后是否归档到正式文档

`运维` 当前不是主角，只在本地运行、LAN host、构建或发布方式再次卡住时介入。

## 当前推荐激活的窗口

### 必开

1. `项目主管`
2. `产品经理`
3. `研发`
4. `测试`

### 按需再开

5. `运维`

## 当前阶段的主从关系

- `项目主管`：唯一总控
- `产品经理`：先冻结 UI 目标与不做项
- `研发`：按冻结结果实现
- `测试`：对实现结果做验证、回归和证据沉淀
- `运维`：只在环境和交付形态问题上介入

## 当前阶段建议的推进顺序

1. `项目主管` 下发本轮唯一目标
2. `产品经理` 先冻结 UI 目标、保留项和不做项
3. `研发` 再做首轮实现
4. `测试` 基于研发产出做验证
5. `项目主管` 决定是否通过、是否归档、下一轮做什么

如果中途出现：

- 本地跑不起来
- 构建不稳定
- 演示脚本无法复跑
- 发布方式需要决策

再叫 `运维` 介入。

## 当前最适合主管下发的任务类型

### 发给研发

- `按冻结结果收口首屏 UI`
- `削弱 demo 感元素`
- `强化已接入 agent 的存在感`
- `保证 T026 接入链路不回退`

### 发给测试

- `验证桌面 / 模拟移动 / 真机接入三种视角`
- `补真实扫码证据`
- `输出 before / after 对比`
- `记录缺陷和复现步骤`

### 发给产品经理

- `明确首屏目标`
- `删除多余信息`
- `定义接入成功后的视觉主语`
- `列出不做项`

### 暂时不要发给运维

- “现在就上完整部署”
- “现在就做正式线上运维体系”
- “现在就做公网级发布承诺”

## 主管窗口的最小工作流

每一轮都只做 4 步：

1. 读 `todo-list.md` 和 `todo-verification-log.md`
2. 确定唯一当前目标
3. 把任务发给研发或测试
4. 回收结果并决定：
   - 通过
   - 返工
   - 升级给产品 / 运维

## 当前阶段的默认唯一目标写法

推荐主管窗口始终用这种句式发任务：

> 当前唯一目标是：让接入成功后的微播客表面更像真实产品，同时不破坏已经跑通的真机接入链路。

这样能避免窗口们重新发散回“大而全产品规划”或重新改坏接入链路。
