# 当前阶段多窗口执行方案

## 当前判断

项目已经完成大部分 `P0` 主链路，当前最值得投入的，是：

1. `研发`
   - 完成 `T004 /network`
   - 为 `T005` 的完整 Demo 剧本补齐页面与状态承接
2. `测试`
   - 把完整试玩路径跑实
   - 固化截图、问题清单、复现步骤和最终演示证据
3. `项目主管`
   - 控制范围
   - 决定先做 `T004` 还是先做 `T005`
   - 决定哪些结果正式写回 `doc/`

产品经理和运维当前不是主角，但仍然保留：

- `产品经理`：只在范围不清、页面目的摇摆、Demo 剧本需要删减时介入
- `运维`：只在本地脚本、构建、发布方式、样例目录或运行环境出问题时介入

## 当前推荐激活的窗口

### 必开

1. `项目主管`
2. `研发`
3. `测试`

### 按需再开

4. `产品经理`
5. `运维`

## 当前阶段的主从关系

- `项目主管`：唯一总控
- `研发`：先产出可运行结果
- `测试`：对研发产出做验证、回归和证据沉淀
- `产品经理`：只在范围不清时裁产品边界
- `运维`：只在环境和交付形态问题上介入

## 当前阶段建议的推进顺序

1. `项目主管` 下发本轮唯一目标
2. `研发` 先做实现或补齐链路
3. `测试` 基于研发产出做验证
4. `项目主管` 决定是否通过、是否归档、下一轮做什么

如果中途出现：

- 页面定义摇摆
- 动作太多
- 叙事影响主链路

再叫 `产品经理` 介入。

如果中途出现：

- 本地跑不起来
- 构建不稳定
- 演示脚本无法复跑
- 发布方式需要决策

再叫 `运维` 介入。

## 当前最适合主管下发的任务类型

### 发给研发

- `补一个页面`
- `打通一条链路`
- `统一一处命令 / payload / 页面状态`
- `修复一个阻断试玩的问题`

### 发给测试

- `验证一条路径`
- `做一轮回归`
- `输出截图证据`
- `记录缺陷和复现步骤`

### 暂时不要发给产品经理

- “再想想还有什么大方向”
- “把长期企业版也一起梳理”
- “顺便把世界观再写完整”

### 暂时不要发给运维

- “现在就上完整部署”
- “现在就做正式线上运维体系”
- “现在就做公网级发布承诺”

## 主管窗口的最小工作流

每一轮都只做 4 步：

1. 读 `todo-list.md` 和 `todo-verification-log.md`
2. 确定唯一当前目标
3. 把任务发给研发或测试
4. 回收结果并决定：
   - 通过
   - 返工
   - 升级给产品 / 运维

## 当前阶段的默认唯一目标写法

推荐主管窗口始终用这种句式发任务：

> 当前唯一目标是：让某条试玩 / 演示 / 验证链路更完整、更可复现，不新增无关功能。

这样能避免窗口们重新发散回“大而全产品规划”。
